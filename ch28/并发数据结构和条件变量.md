# 1. 并发计数器

## 1.1 普通计数器的问题

普通计数器中，多个线程同时执行：

```c
value++;
```

看起来是一条语句，但实际上通常包含三步：

```text
1. 读取 value
2. value + 1
3. 写回 value
```

如果两个线程同时读取旧值，就可能发生 **丢失更新**。

例如：

```text
初始 value = 5

线程 A 读到 5
线程 B 读到 5

线程 A 写回 6
线程 B 写回 6
```

本来应该加两次变成 `7`，结果只变成了 `6`。

---

## 1.2 精确计数器

一种简单做法是给计数器加锁。

```c
typedef struct counter_t {
    int value;
    pthread_mutex_t lock;
} counter_t;
```

每次更新计数器时：

```c
pthread_mutex_lock(&c->lock);
c->value++;
pthread_mutex_unlock(&c->lock);
```

这样可以保证同一时刻只有一个线程修改 `value`。

### 问题

这种方法虽然正确，但扩展性差。

原因是：

```text
所有线程都竞争同一把锁。
```

线程越多，抢锁越激烈，程序越慢。

---

## 1.3 懒惰计数器

懒惰计数器的思想是：

```text
每个 CPU 有自己的局部计数器。
全局计数器只偶尔更新。
```

例如有 4 个 CPU：

```text
CPU0 -> L0
CPU1 -> L1
CPU2 -> L2
CPU3 -> L3
```

每个 CPU 上运行的线程优先更新对应的局部计数器。

当局部计数器达到阈值 `S` 时，再合并到全局计数器。

```text
L0 = S

G = G + L0
L0 = 0
```

---

## 1.4 L0 的锁会被谁使用？

`L0` 不是某个线程私有的，而是 CPU0 对应的局部计数器。

如果多个线程先后被调度到 CPU0 上运行，它们都会访问 `L0`。

```text
线程 A 在 CPU0 上运行 -> 更新 L0
线程 B 在 CPU0 上运行 -> 更新 L0
```

因此 `L0` 仍然需要自己的锁。

但是正常情况下，多个线程会被调度到不同 CPU 上：

```text
线程 A -> CPU0 -> L0
线程 B -> CPU1 -> L1
线程 C -> CPU2 -> L2
线程 D -> CPU3 -> L3
```

这样大多数时候不同线程不会竞争同一把锁。

---

## 1.5 懒惰计数器的权衡

```text
S 越小：
    全局计数更精确
    但全局锁访问更频繁

S 越大：
    局部更新更多
    全局锁访问更少
    但全局计数暂时不精确
```

核心思想：

```text
牺牲一点实时精确性，换取更好的并发性能。
```

# 2. 并发链表

## 2.1 节点结构

```c
typedef struct node_t {
    int key;
    struct node_t *next;
} node_t;
```

逐行解释：

```c
typedef struct node_t {
```

定义一个结构体，结构体标签名是 `struct node_t`。

```c
int key;
```

节点保存的数据。

```c
struct node_t *next;
```

指向下一个节点的指针。

这里必须写 `struct node_t *next`，因为在结构体内部，`node_t` 这个 typedef 别名还没有完全生效。

```c
} node_t;
```

给 `struct node_t` 起一个别名 `node_t`。

以后可以写：

```c
node_t *p;
```

而不用写：

```c
struct node_t *p;
```

---

## 2.2 链表结构

```c
typedef struct list_t {
    node_t *head;
    pthread_mutex_t lock;
} list_t;
```

逐行解释：

```c
node_t *head;
```

指向链表第一个节点。

```c
pthread_mutex_t lock;
```

保护整条链表的一把锁。

这种链表是 **粗粒度锁链表**：

```text
整条链表只有一把锁。
```

---

## 2.3 初始化函数

```c
void List_Init(list_t *L) {
    L->head = NULL;
    pthread_mutex_init(&L->lock, NULL);
}
```

逐行解释：

```c
L->head = NULL;
```

一开始链表为空。

```c
pthread_mutex_init(&L->lock, NULL);
```

初始化链表锁。

---

## 2.4 插入函数

```c
void List_Insert(list_t *L, int key) {
    node_t *new = malloc(sizeof(node_t));
    if (new == NULL) {
        perror("malloc");
        return;
    }

    new->key = key;

    pthread_mutex_lock(&L->lock);
    new->next = L->head;
    L->head = new;
    pthread_mutex_unlock(&L->lock);
}
```

逐行解释：

```c
node_t *new = malloc(sizeof(node_t));
```

在堆区申请一个新节点。

```c
if (new == NULL) {
    perror("malloc");
    return;
}
```

如果内存申请失败，打印错误并返回。

注意：此时还没有加锁，所以不需要解锁。

```c
new->key = key;
```

给新节点赋值。

此时新节点还没有接入共享链表，只有当前线程能看到它，所以不需要锁。

```c
pthread_mutex_lock(&L->lock);
```

开始进入真正的临界区。

```c
new->next = L->head;
```

让新节点指向原来的头节点。

```c
L->head = new;
```

让链表头指针指向新节点。

```c
pthread_mutex_unlock(&L->lock);
```

释放锁。

---

## 2.5 为什么 malloc 放在锁外？

因为 `malloc()` 得到的新节点还没有接入共享链表。

此时：

```text
new 只是当前线程自己的局部对象。
其他线程还看不到它。
```

真正需要保护的是：

```c
new->next = L->head;
L->head = new;
```

这两句会修改共享链表结构，所以必须加锁。

---

## 2.6 查找函数

```c
int List_Lookup(list_t *L, int key) {
    int rv = -1;

    pthread_mutex_lock(&L->lock);

    node_t *curr = L->head;
    while (curr) {
        if (curr->key == key) {
            rv = 0;
            break;
        }
        curr = curr->next;
    }

    pthread_mutex_unlock(&L->lock);
    return rv;
}
```

逐行解释：

```c
int rv = -1;
```

默认查找失败。

```c
pthread_mutex_lock(&L->lock);
```

加锁，防止查找过程中链表被其他线程修改。

```c
node_t *curr = L->head;
```

从头节点开始遍历。

```c
while (curr) {
```

只要当前节点不为空，就继续查找。

```c
if (curr->key == key) {
    rv = 0;
    break;
}
```

如果找到目标值，就把返回值设为成功，并跳出循环。

```c
curr = curr->next;
```

继续访问下一个节点。

```c
pthread_mutex_unlock(&L->lock);
return rv;
```

统一解锁，统一返回。

---

## 2.7 为什么要减少多个返回路径？

如果代码写成：

```c
pthread_mutex_lock(&L->lock);

if (某种情况) {
    return -1;
}

pthread_mutex_unlock(&L->lock);
```

就可能忘记释放锁，导致其他线程永远卡住。

更好的写法是：

```c
pthread_mutex_lock(&L->lock);

/* 修改返回值，而不是直接 return */

pthread_mutex_unlock(&L->lock);
return rv;
```

# 3. 过手锁链表

## 3.1 什么是过手锁？

过手锁也叫锁耦合。

思想是：

```text
每个节点都有一把锁。
遍历链表时，先锁下一个节点，再释放当前节点。
```

例如链表：

```text
A -> B -> C -> D
```

遍历过程：

```text
锁 A
锁 B
释放 A

锁 C
释放 B

锁 D
释放 C
```

---

## 3.2 为什么先锁下一个，再释放当前？

假设当前在节点 A，要移动到节点 B。

如果先释放 A，再去锁 B，中间可能有其他线程删除 B。

安全做法是：

```text
持有 A 的锁
先获得 B 的锁
再释放 A 的锁
```

这样可以保证移动过程中链表结构不会被破坏。

---

## 3.3 过手锁的问题

单锁链表查找一次：

```text
加锁 1 次
解锁 1 次
```

过手锁查找 100 个节点：

```text
加锁 100 次
解锁 100 次
```

虽然并发性更强，但锁操作开销很大。

所以：

```text
更多并发不一定更快。
```

# 4. 并发队列

## 4.1 节点结构

```c
typedef struct node_t {
    int value;
    struct node_t *next;
} node_t;
```

逐行解释：

```c
int value;
```

节点保存的数据。

```c
struct node_t *next;
```

指向下一个节点。

---

## 4.2 队列结构

```c
typedef struct queue_t {
    node_t *head;
    node_t *tail;
    pthread_mutex_t headLock;
    pthread_mutex_t tailLock;
} queue_t;
```

逐行解释：

```c
node_t *head;
```

指向队头。

```c
node_t *tail;
```

指向队尾。

```c
pthread_mutex_t headLock;
```

保护出队操作。

```c
pthread_mutex_t tailLock;
```

保护入队操作。

这个队列使用两把锁：

```text
入队使用 tailLock
出队使用 headLock
```

这样入队和出队可以在一定程度上并发。

---

## 4.3 初始化函数

```c
void Queue_Init(queue_t *q) {
    node_t *tmp = malloc(sizeof(node_t));
    tmp->next = NULL;
    q->head = q->tail = tmp;
    pthread_mutex_init(&q->headLock, NULL);
    pthread_mutex_init(&q->tailLock, NULL);
}
```

逐行解释：

```c
node_t *tmp = malloc(sizeof(node_t));
```

创建一个节点。

```c
tmp->next = NULL;
```

该节点后面没有其他节点。

```c
q->head = q->tail = tmp;
```

`head` 和 `tail` 都指向这个节点。

这个节点是 **dummy node**，也叫哑节点。

初始化后：

```text
head
 ↓
[dummy] -> NULL
 ↑
tail
```

```c
pthread_mutex_init(&q->headLock, NULL);
pthread_mutex_init(&q->tailLock, NULL);
```

初始化队头锁和队尾锁。

---

## 4.4 入队函数

```c
void Queue_Enqueue(queue_t *q, int value) {
    node_t *tmp = malloc(sizeof(node_t));
    assert(tmp != NULL);
    tmp->value = value;
    tmp->next = NULL;

    pthread_mutex_lock(&q->tailLock);
    q->tail->next = tmp;
    q->tail = tmp;
    pthread_mutex_unlock(&q->tailLock);
}
```

逐行解释：

```c
node_t *tmp = malloc(sizeof(node_t));
```

创建新节点。

```c
assert(tmp != NULL);
```

确认内存分配成功。

```c
tmp->value = value;
```

保存入队数据。

```c
tmp->next = NULL;
```

新节点将成为队尾，因此它的 `next` 为 `NULL`。

```c
pthread_mutex_lock(&q->tailLock);
```

锁住队尾。

```c
q->tail->next = tmp;
```

把新节点接到当前队尾后面。

```c
q->tail = tmp;
```

更新队尾指针。

```c
pthread_mutex_unlock(&q->tailLock);
```

释放队尾锁。

---

## 4.5 出队函数

```c
int Queue_Dequeue(queue_t *q, int *value) {
    pthread_mutex_lock(&q->headLock);

    node_t *tmp = q->head;
    node_t *newHead = tmp->next;

    if (newHead == NULL) {
        pthread_mutex_unlock(&q->headLock);
        return -1;
    }

    *value = newHead->value;
    q->head = newHead;

    pthread_mutex_unlock(&q->headLock);
    free(tmp);

    return 0;
}
```

逐行解释：

```c
int Queue_Dequeue(queue_t *q, int *value)
```

从队列中取出一个值。

`value` 是输出参数。

例如：

```c
int x;
int ret = Queue_Dequeue(&q, &x);
```

如果出队成功，函数内部：

```c
*value = newHead->value;
```

等价于：

```c
x = newHead->value;
```

```c
pthread_mutex_lock(&q->headLock);
```

锁住队头。

```c
node_t *tmp = q->head;
```

`tmp` 指向旧 dummy 节点。

```c
node_t *newHead = tmp->next;
```

`newHead` 指向第一个有效节点。

```c
if (newHead == NULL)
```

如果 dummy 后面没有有效节点，说明队列为空。

```c
pthread_mutex_unlock(&q->headLock);
return -1;
```

释放锁并返回失败。

```c
*value = newHead->value;
```

把第一个有效节点的值写入调用者传进来的变量。

```c
q->head = newHead;
```

让第一个有效节点变成新的 dummy。

```c
pthread_mutex_unlock(&q->headLock);
```

释放队头锁。

```c
free(tmp);
```

释放旧 dummy 节点。

```c
return 0;
```

返回成功。

---

## 4.6 为什么释放 tmp，而不是 newHead？

因为这个队列中：

```text
head 永远指向 dummy 节点。
真正的第一个有效元素是 head->next。
```

出队时：

```text
旧 dummy -> 第一个有效节点
```

出队操作是：

```text
取出第一个有效节点的 value
让第一个有效节点变成新的 dummy
释放旧 dummy
```

所以释放的是：

```c
free(tmp);
```

不能释放 `newHead`，否则新的 `head` 就指向已释放内存。

---

## 4.7 单一返回路径版本

原出队函数有两个返回点，可以改成：

```c
int Queue_Dequeue(queue_t *q, int *value) {
    int rv = -1;
    node_t *oldHead = NULL;

    pthread_mutex_lock(&q->headLock);

    oldHead = q->head;
    node_t *newHead = oldHead->next;

    if (newHead != NULL) {
        *value = newHead->value;
        q->head = newHead;
        rv = 0;
    }

    pthread_mutex_unlock(&q->headLock);

    if (rv == 0) {
        free(oldHead);
    }

    return rv;
}
```

注意：

```text
不能无条件 free(oldHead)。
```

如果队列为空，`oldHead` 是唯一的 dummy 节点，释放它会破坏队列。

# 5. 并发散列表

## 5.1 散列表结构

```c
#define BUCKETS (101)

typedef struct hash_t {
    list_t lists[BUCKETS];
} hash_t;
```

逐行解释：

```c
#define BUCKETS (101)
```

定义桶数量为 101。

```c
list_t lists[BUCKETS];
```

散列表由 101 个桶组成，每个桶是一个链表。

---

## 5.2 初始化函数

```c
void Hash_Init(hash_t *H) {
    int i;
    for (i = 0; i < BUCKETS; i++) {
        List_Init(&H->lists[i]);
    }
}
```

逐行解释：

```c
for (i = 0; i < BUCKETS; i++)
```

遍历所有桶。

```c
List_Init(&H->lists[i]);
```

初始化每个桶对应的链表。

---

## 5.3 插入函数

```c
int Hash_Insert(hash_t *H, int key) {
    int bucket = key % BUCKETS;
    return List_Insert(&H->lists[bucket], key);
}
```

逐行解释：

```c
int bucket = key % BUCKETS;
```

根据 key 计算桶号。

```c
return List_Insert(&H->lists[bucket], key);
```

把 key 插入对应桶的链表。

---

## 5.4 查找函数

```c
int Hash_Lookup(hash_t *H, int key) {
    int bucket = key % BUCKETS;
    return List_Lookup(&H->lists[bucket], key);
}
```

逐行解释：

```c
int bucket = key % BUCKETS;
```

计算 key 所在的桶。

```c
return List_Lookup(&H->lists[bucket], key);
```

只在对应桶中查找。

---

## 5.5 为什么散列表并发性能好？

每个桶是一个独立链表，每个链表有自己的锁。

如果不同线程访问不同桶：

```text
线程 A -> bucket 5
线程 B -> bucket 77
线程 C -> bucket 1
```

它们就可以并发执行。

这种设计比整个散列表一把锁更好，也比每个节点一把锁简单。

---

# 6. 条件变量

## 6.1 条件变量的作用

条件变量用于：

```text
当某个条件不满足时，让线程睡眠。
当条件可能满足时，唤醒等待线程。
```

常见接口：

```c
pthread_cond_wait(pthread_cond_t *c, pthread_mutex_t *m);
pthread_cond_signal(pthread_cond_t *c);
```

---

## 6.2 父线程等待子线程示例

```c
int done = 0;
pthread_mutex_t m = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t c = PTHREAD_COND_INITIALIZER;
```

逐行解释：

```c
int done = 0;
```

状态变量，表示子线程是否结束。

```c
pthread_mutex_t m = PTHREAD_MUTEX_INITIALIZER;
```

互斥锁，保护 `done`。

```c
pthread_cond_t c = PTHREAD_COND_INITIALIZER;
```

条件变量，用来睡眠和唤醒。

---

## 6.3 子线程退出函数

```c
void thr_exit() {
    Pthread_mutex_lock(&m);
    done = 1;
    Pthread_cond_signal(&c);
    Pthread_mutex_unlock(&m);
}
```

逐行解释：

```c
Pthread_mutex_lock(&m);
```

加锁，准备修改共享变量。

```c
done = 1;
```

表示子线程已经结束。

```c
Pthread_cond_signal(&c);
```

唤醒等待在条件变量 `c` 上的线程。

```c
Pthread_mutex_unlock(&m);
```

释放锁。

---

## 6.4 父线程等待函数

```c
void thr_join() {
    Pthread_mutex_lock(&m);
    while (done == 0)
        Pthread_cond_wait(&c, &m);
    Pthread_mutex_unlock(&m);
}
```

逐行解释：

```c
Pthread_mutex_lock(&m);
```

加锁，准备检查 `done`。

```c
while (done == 0)
```

如果子线程还没有结束，就继续等待。

```c
Pthread_cond_wait(&c, &m);
```

`wait()` 会原子地完成：

```text
1. 释放锁 m
2. 让当前线程睡眠
3. 被唤醒后重新获得锁 m
```

```c
Pthread_mutex_unlock(&m);
```

条件满足后释放锁。

---

## 6.5 为什么 wait 必须和锁配合？

如果没有锁，可能出现丢失唤醒。

错误顺序：

```text
父线程检查 done == 0
父线程准备 wait，但还没有真正睡眠

子线程运行
子线程 done = 1
子线程 signal

signal 没有等待者，于是丢失

父线程开始睡眠
再也没有线程唤醒它
```

正确写法必须是：

```c
pthread_mutex_lock(&m);
while (done == 0)
    pthread_cond_wait(&c, &m);
pthread_mutex_unlock(&m);
```

---

## 6.6 为什么用 while，而不是 if？

线程从 `wait()` 返回后，不能保证条件一定成立。

原因可能是：

```text
1. 虚假唤醒
2. 被唤醒后条件又被其他线程改变
3. signal 唤醒了不该醒的线程
```

所以标准写法是：

```c
while (条件不满足)
    pthread_cond_wait(...);
```

而不是：

```c
if (条件不满足)
    pthread_cond_wait(...);
```

---

## 6.7 条件变量的核心模式

等待方：

```c
pthread_mutex_lock(&m);

while (条件不满足) {
    pthread_cond_wait(&c, &m);
}

pthread_mutex_unlock(&m);
```

通知方：

```c
pthread_mutex_lock(&m);

修改共享状态;
pthread_cond_signal(&c);

pthread_mutex_unlock(&m);
```

# 7. 生产者消费者问题

## 7.1 单槽缓冲区

```c
int buffer;
int count = 0;
```

逐行解释：

```c
int buffer;
```

缓冲区，只能存一个整数。

```c
int count = 0;
```

表示缓冲区状态：

```text
count = 0：缓冲区为空
count = 1：缓冲区已满
```

---

## 7.2 put 函数

```c
void put(int value) {
    assert(count == 0);
    count = 1;
    buffer = value;
}
```

逐行解释：

```c
assert(count == 0);
```

只有缓冲区为空时才能放入数据。

```c
count = 1;
```

标记缓冲区已满。

```c
buffer = value;
```

把数据写入缓冲区。

---

## 7.3 get 函数

```c
int get() {
    assert(count == 1);
    count = 0;
    return buffer;
}
```

逐行解释：

```c
assert(count == 1);
```

只有缓冲区已满时才能取数据。

```c
count = 0;
```

标记缓冲区为空。

```c
return buffer;
```

返回缓冲区中的数据。

---

## 7.4 生产者线程

```c
void *producer(void *arg) {
    int i;
    int loops = (int) arg;
    for (i = 0; i < loops; i++) {
        put(i);
    }
}
```

逐行解释：

```c
int loops = (int) arg;
```

把传入参数当作循环次数。

```c
for (i = 0; i < loops; i++)
```

循环生产多个数据。

```c
put(i);
```

把 `i` 放入缓冲区。

---

## 7.5 消费者线程

```c
void *consumer(void *arg) {
    int i;
    while (1) {
        int tmp = get();
        printf("%d\n", tmp);
    }
}
```

逐行解释：

```c
while (1)
```

无限循环。

```c
int tmp = get();
```

从缓冲区取出数据。

```c
printf("%d\n", tmp);
```

打印数据。

---

## 7.6 为什么直接 put/get 不安全？

如果生产者连续 `put()` 两次：

```text
第一次 put 后 count = 1
第二次 put 时缓冲区已经满
触发 assert(count == 0)
```

如果消费者连续 `get()` 两次：

```text
第一次 get 后 count = 0
第二次 get 时缓冲区为空
触发 assert(count == 1)
```

所以必须使用锁和条件变量。

# 8. 最终生产者消费者方案

## 8.1 多槽环形缓冲区

```c
int buffer[MAX];
int fill = 0;
int use = 0;
int count = 0;
```

逐行解释：

```c
int buffer[MAX];
```

缓冲区数组，可以保存 `MAX` 个数据。

```c
int fill = 0;
```

下一个生产者写入的位置。

```c
int use = 0;
```

下一个消费者读取的位置。

```c
int count = 0;
```

当前缓冲区中的有效数据数量。

---

## 8.2 put 函数

```c
void put(int value) {
    buffer[fill] = value;
    fill = (fill + 1) % MAX;
    count++;
}
```

逐行解释：

```c
buffer[fill] = value;
```

把数据写入 `fill` 指向的位置。

```c
fill = (fill + 1) % MAX;
```

更新下一个写入位置。

`% MAX` 用于实现环形缓冲区：

```text
0 -> 1 -> 2 -> ... -> MAX-1 -> 0
```

```c
count++;
```

有效数据数量加 1。

---

## 8.3 get 函数

```c
int get() {
    int tmp = buffer[use];
    use = (use + 1) % MAX;
    count--;
    return tmp;
}
```

逐行解释：

```c
int tmp = buffer[use];
```

从 `use` 指向的位置取出数据。

```c
use = (use + 1) % MAX;
```

更新下一个读取位置。

```c
count--;
```

有效数据数量减 1。

```c
return tmp;
```

返回取出的数据。

---

## 8.4 条件变量和锁

```c
cond_t empty, fill;
mutex_t mutex;
```

含义：

```text
empty：表示有空位，生产者等待它。
fill ：表示有数据，消费者等待它。
mutex：保护 buffer、fill、use、count。
```

---

## 8.5 生产者最终代码

```c
void *producer(void *arg) {
    int i;

    for (i = 0; i < loops; i++) {
        Pthread_mutex_lock(&mutex);

        while (count == MAX)
            Pthread_cond_wait(&empty, &mutex);

        put(i);

        Pthread_cond_signal(&fill);
        Pthread_mutex_unlock(&mutex);
    }
}
```

逐行解释：

```c
Pthread_mutex_lock(&mutex);
```

加锁，准备访问共享缓冲区。

```c
while (count == MAX)
```

如果缓冲区已满，生产者不能继续生产。

```c
Pthread_cond_wait(&empty, &mutex);
```

等待“有空位”。

```c
put(i);
```

缓冲区没满，放入数据。

```c
Pthread_cond_signal(&fill);
```

通知消费者：现在有数据了。

```c
Pthread_mutex_unlock(&mutex);
```

释放锁。

---

## 8.6 消费者最终代码

```c
void *consumer(void *arg) {
    int i;

    for (i = 0; i < loops; i++) {
        Pthread_mutex_lock(&mutex);

        while (count == 0)
            Pthread_cond_wait(&fill, &mutex);

        int tmp = get();

        Pthread_cond_signal(&empty);
        Pthread_mutex_unlock(&mutex);

        printf("%d\n", tmp);
    }
}
```

逐行解释：

```c
Pthread_mutex_lock(&mutex);
```

加锁，准备访问共享缓冲区。

```c
while (count == 0)
```

如果缓冲区为空，消费者不能消费。

```c
Pthread_cond_wait(&fill, &mutex);
```

等待“有数据”。

```c
int tmp = get();
```

从缓冲区取出数据。

```c
Pthread_cond_signal(&empty);
```

通知生产者：现在有空位了。

```c
Pthread_mutex_unlock(&mutex);
```

释放锁。

```c
printf("%d\n", tmp);
```

打印取出的数据。

---

## 8.7 为什么用两个条件变量？

如果只有一个条件变量，可能唤醒错线程。

两个条件变量语义更清楚：

```text
生产者等待 empty，被消费者唤醒。
消费者等待 fill，被生产者唤醒。
```

# 9. 覆盖条件与广播唤醒

## 9.1 内存分配例子

```c
int bytesLeft = MAX_HEAP_SIZE;

cond_t c;
mutex_t m;
```

含义：

```text
bytesLeft：当前空闲内存大小。
c：条件变量。
m：保护 bytesLeft 的锁。
```

---

## 9.2 allocate 函数

```c
void *allocate(int size) {
    Pthread_mutex_lock(&m);

    while (bytesLeft < size)
        Pthread_cond_wait(&c, &m);

    void *ptr = ...;
    bytesLeft -= size;

    Pthread_mutex_unlock(&m);
    return ptr;
}
```

逐行解释：

```c
Pthread_mutex_lock(&m);
```

加锁，准备检查空闲内存。

```c
while (bytesLeft < size)
```

如果空闲内存不够，就等待。

```c
Pthread_cond_wait(&c, &m);
```

等待其他线程释放内存。

```c
void *ptr = ...;
```

从堆中取出一块内存。

```c
bytesLeft -= size;
```

减少空闲内存数量。

```c
Pthread_mutex_unlock(&m);
```

释放锁。

```c
return ptr;
```

返回分配到的内存。

---

## 9.3 free 函数

```c
void free(void *ptr, int size) {
    Pthread_mutex_lock(&m);

    bytesLeft += size;

    Pthread_cond_signal(&c);

    Pthread_mutex_unlock(&m);
}
```

逐行解释：

```c
Pthread_mutex_lock(&m);
```

加锁。

```c
bytesLeft += size;
```

释放内存后，空闲空间增加。

```c
Pthread_cond_signal(&c);
```

唤醒一个等待线程。

```c
Pthread_mutex_unlock(&m);
```

释放锁。

---

## 9.4 signal 的问题

假设：

```text
bytesLeft = 0

Ta 等待 allocate(100)
Tb 等待 allocate(10)
```

现在：

```text
Tc free(50)
bytesLeft = 50
```

此时：

```text
Ta 仍然不能运行，因为 50 < 100
Tb 可以运行，因为 50 >= 10
```

但是 `signal` 只能唤醒一个线程，而且不能指定唤醒谁。

如果唤醒了 `Ta`：

```text
Ta 醒来后发现内存仍然不够，又睡回去。
Tb 明明可以运行，却没有被唤醒。
```

程序可能卡住。

---

## 9.5 broadcast 的解决方法

使用：

```c
pthread_cond_broadcast(&c);
```

代替：

```c
pthread_cond_signal(&c);
```

`broadcast` 会唤醒所有等待线程。

所有线程醒来后重新检查条件：

```text
Ta 检查 50 >= 100，不满足，继续睡。
Tb 检查 50 >= 10，满足，可以继续运行。
```

---

## 9.6 覆盖条件

这种条件变量叫 **覆盖条件**。

含义是：

```text
一个条件变量覆盖多种具体等待条件。
```

例如：

```text
bytesLeft >= 10
bytesLeft >= 50
bytesLeft >= 100
bytesLeft >= 1000
```

优点：

```text
正确、简单。
```

缺点：

```text
可能唤醒太多线程，影响性能。
```

# 10. UNIX 管道与生产者消费者模型

## 10.1 管道例子

```bash
grep foo file.txt | wc -l
```

对应关系：

```text
grep foo file.txt：生产者
wc -l：消费者
pipe：内核中的有界缓冲区
```

---

## 10.2 对应表

| 生产者消费者代码 | UNIX 管道例子 |
|---|---|
| `producer()` | `grep` 进程 |
| `consumer()` | `wc -l` 进程 |
| `buffer[MAX]` | 内核 pipe buffer |
| `put()` | `write()` 写管道 |
| `get()` | `read()` 读管道 |
| `count == MAX` | 管道满 |
| `count == 0` | 管道空 |
| `empty` | 等待管道有空位 |
| `fill` | 等待管道有数据 |

---

## 10.3 管道满时

如果 `grep` 写得太快，`wc` 读得太慢，管道会满。

此时：

```text
grep 调用 write() 时阻塞。
```

类似生产者：

```c
while (count == MAX)
    pthread_cond_wait(&empty, &mutex);
```

---

## 10.4 管道空时

如果 `wc` 读得太快，`grep` 还没有写入数据，管道会空。

此时：

```text
wc 调用 read() 时阻塞。
```

类似消费者：

```c
while (count == 0)
    pthread_cond_wait(&fill, &mutex);
```

---

## 10.5 用户态代码和内核管道的区别

生产者消费者代码：

```text
多个线程共享 buffer、count、mutex、cond。
```

UNIX 管道：

```text
grep 和 wc 是两个进程。
它们不共享用户态内存。
它们通过内核中的 pipe buffer 交换数据。
同步由内核完成。
```

# 11. 总结

## 11.1 并发数据结构的核心思想

```text
正确性靠锁保护共享数据。
性能靠减少不必要的锁竞争。
```

不同结构的策略：

```text
计数器：
    简单锁正确但慢。
    懒惰计数器用局部计数减少全局锁竞争。

链表：
    整条链表一把锁简单但扩展性差。
    过手锁并发更高但开销大。

队列：
    dummy node 简化空队列处理。
    headLock 和 tailLock 分别保护出队和入队。

散列表：
    每个桶一把锁，是并发性和复杂度之间的折中。
```

---

## 11.2 条件变量的核心思想

```text
锁保护状态变量。
条件变量负责睡眠和唤醒。
```

状态变量才是真正的条件：

```text
done
count
bytesLeft
```

条件变量本身不保存状态。

---

# 12. 易错点

## 12.1 锁相关

```text
1. 锁保护的是共享状态，不是函数名。
2. 不要把非共享操作也放进临界区。
3. 拿锁后要避免多个提前 return。
4. 多个返回路径容易忘记 unlock。
5. malloc 可以放在锁外，前提是新对象还没有共享。
```

---

## 12.2 条件变量相关

```text
1. wait 必须持锁调用。
2. signal 最好也持锁调用。
3. wait 返回后必须重新检查条件。
4. 所以要用 while，不要用 if。
5. 条件变量不保存状态，状态变量才表示条件是否成立。
```

---

## 12.3 生产者消费者相关

```text
1. 生产者不能在缓冲区满时 put。
2. 消费者不能在缓冲区空时 get。
3. empty 表示有空位，是生产者等待的条件。
4. fill 表示有数据，是消费者等待的条件。
5. signal 要在改变状态之后执行。
```

---

## 12.4 数据结构相关

```text
1. 单锁简单但扩展性差。
2. 过手锁并发高但开销大。
3. 队列释放的是旧 dummy，不是 newHead。
4. 散列表每个桶一把锁，通常是实用折中。
```

---

# 13. 简短记忆版

```text
计数器：
    一把锁正确但慢。
    懒惰计数器用局部计数减少全局锁竞争。

链表：
    malloc 和初始化节点可以放锁外。
    真正要锁的是修改 head 和 next。

过手锁：
    先锁下一个，再释放当前。
    并发高，但加锁解锁开销大。

队列：
    用 dummy node。
    入队锁 tail，出队锁 head。
    出队释放旧 dummy。

散列表：
    多个桶，每个桶一把锁。
    不同桶可以并发访问。

条件变量：
    锁保护状态变量。
    条件变量负责睡眠和唤醒。
    wait 用 while，不用 if。

生产者消费者：
    生产者等 empty。
    消费者等 fill。
    count 判断满和空。

broadcast：
    不知道该唤醒谁时，唤醒所有人。
    正确但可能浪费性能。

管道：
    grep 是生产者。
    wc 是消费者。
    pipe buffer 是内核中的有界缓冲区。
```
