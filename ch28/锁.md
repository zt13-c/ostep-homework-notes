# OSTEP 并发锁机制笔记：Ticket Lock、Yield Lock、Futex 与 Pthread

## 1. Ticket Lock：票据锁

### 1.1 代码

```asm
.var ticket
.var turn
.var count

.main
.top    

.acquire
mov $1, %ax
fetchadd %ax, ticket  # grab a ticket 

.tryagain
mov turn, %cx         # check if it's your turn 
test %cx, %ax
jne .tryagain

# critical section
mov  count, %ax       # get the value at the address
add  $1, %ax          # increment it
mov  %ax, count       # store it back

# release lock
mov $1, %ax
fetchadd %ax, turn

# see if we're still looping
sub  $1, %bx
test $0, %bx
jgt .top    

halt
```

---

### 1.2 核心思想

Ticket lock 的思想类似银行取号排队：

```text
ticket：下一个要发出去的票号
turn：当前轮到哪个票号进入临界区
count：共享计数器
```

初始状态通常是：

```text
ticket = 0
turn   = 0
count  = 0
```

每个线程要进入临界区之前，先从 `ticket` 中取一个号码，然后等待 `turn` 等于自己的号码。

---

### 1.3 获取票号

```asm
mov $1, %ax
fetchadd %ax, ticket
```

`fetchadd %ax, ticket` 是原子操作，可以理解为：

```c
old = ticket;
ticket = ticket + ax;
ax = old;
```

由于前面有：

```asm
mov $1, %ax
```

所以它实际做的是：

```c
old = ticket;
ticket = ticket + 1;
ax = old;
```

也就是说，线程把自己的票号保存在 `%ax` 中。

例如：

```text
初始 ticket = 0

线程 0 执行 fetchadd：
    %ax = 0
    ticket = 1

线程 1 执行 fetchadd：
    %ax = 1
    ticket = 2

线程 2 执行 fetchadd：
    %ax = 2
    ticket = 3
```

---

### 1.4 等待轮到自己

```asm
.tryagain
mov turn, %cx
test %cx, %ax
jne .tryagain
```

这段代码等价于：

```c
while (turn != my_ticket)
    ;
```

其中：

```text
%ax：当前线程自己的票号
%cx：当前 turn 的值
```

如果：

```text
turn == my_ticket
```

说明轮到当前线程进入临界区。

如果：

```text
turn != my_ticket
```

说明还没轮到自己，需要继续等待。

---

### 1.5 临界区

```asm
mov  count, %ax
add  $1, %ax
mov  %ax, count
```

等价于：

```c
count = count + 1;
```

由于 ticket lock 保证同一时刻只有一个线程满足：

```text
turn == my_ticket
```

所以同一时刻只有一个线程可以执行临界区。

---

### 1.6 释放锁

```asm
mov $1, %ax
fetchadd %ax, turn
```

这段代码等价于：

```c
turn = turn + 1;
```

意思是：

```text
当前票号执行完了，叫下一个票号进入。
```

例如：

```text
turn = 0 时，票号 0 的线程进入
票号 0 执行完后，turn 加 1，变成 1
票号 1 的线程进入
```

---

### 1.7 Ticket Lock 的 C 伪代码

```c
int ticket = 0;
int turn = 0;
int count = 0;

void lock() {
    int my_ticket = fetchadd(&ticket, 1);

    while (turn != my_ticket)
        ;
}

void unlock() {
    fetchadd(&turn, 1);
}

void thread() {
    do {
        lock();

        count = count + 1;

        unlock();

        bx--;
    } while (bx > 0);
}
```

---

### 1.8 Ticket Lock 的特点

优点：

```text
1. 保证互斥。
2. 按取票顺序进入临界区。
3. 公平性较好。
4. 不容易出现某个线程长期抢不到锁的情况。
```

缺点：

```text
1. 等待线程仍然在自旋。
2. 自旋会浪费 CPU。
3. 多核环境下，多个线程反复读取 turn，也会带来缓存一致性开销。
```

---

## 2. 运行 Ticket Lock：`bx=1000,bx=1000`

### 2.1 运行命令

```bash
python3 ./x86.py -p ticket.s -a bx=1000,bx=1000 -M ticket,turn,count -R ax,bx,cx -c -S
```

其中：

```text
-a bx=1000,bx=1000
```

表示两个线程各循环 1000 次。

---

### 2.2 正确结果

两个线程各执行 1000 次：

```text
线程 0：count++ 1000 次
线程 1：count++ 1000 次
```

所以最终正确结果应该是：

```text
count = 2000
```

---

### 2.3 是否会花很多时间自旋？

会。

等待锁的代码是：

```asm
.tryagain
mov turn, %cx
test %cx, %ax
jne .tryagain
```

也就是：

```c
while (turn != my_ticket)
    ;
```

如果还没有轮到当前线程，它会一直反复执行这几条指令。

所以 ticket lock 虽然公平，但仍然会浪费 CPU 时间。

---

### 2.4 如何观察自旋开销？

可以看：

```text
STATS:: Instructions
```

例如：

```bash
python3 ./x86.py -p ticket.s -a bx=1000,bx=1000 -M count -i 10 -c -S
```

如果最终：

```text
count = 2000
```

说明锁保证了正确性。

如果：

```text
STATS:: Instructions
```

数值很大，说明执行了很多额外指令，其中大量指令来自自旋等待。

## 3. Test-and-Set + Yield 锁

### 3.1 代码

```asm
.var mutex
.var count

.main
.top    

.acquire
mov  $1, %ax        
xchg %ax, mutex     # atomic swap of 1 and mutex
test $0, %ax        # if we get 0 back: lock is free!
je .acquire_done    
yield               # if not, yield and try again
j .acquire

.acquire_done

# critical section
mov  count, %ax
add  $1, %ax
mov  %ax, count

# release lock
mov  $0, mutex

# see if we're still looping
sub  $1, %bx
test $0, %bx
jgt .top    

halt
```

---

### 3.2 全局变量含义

```asm
.var mutex
.var count
```

含义：

```text
mutex = 0：锁空闲
mutex = 1：锁被占用

count：共享计数器
```

---

### 3.3 获取锁

```asm
.acquire
mov  $1, %ax        
xchg %ax, mutex
test $0, %ax
je .acquire_done    
yield
j .acquire
```

其中：

```asm
mov $1, %ax
```

表示准备把锁设置为 1。

```asm
xchg %ax, mutex
```

是原子交换。

执行前：

```text
%ax = 1
mutex = 原来的锁状态
```

执行后：

```text
mutex = 1
%ax = mutex 原来的值
```

如果 `%ax == 0`，说明锁原来是空闲的，当前线程成功拿到锁。

如果 `%ax == 1`，说明锁原来已经被别人持有，当前线程获取失败。

---

### 3.4 `yield` 的作用

```asm
yield
j .acquire
```

如果拿不到锁，当前线程不会一直自旋，而是执行 `yield`，主动让出 CPU。

它的意思是：

```text
我现在拿不到锁，先让别的线程运行。
```

然后再跳回 `.acquire` 重新尝试。

---

### 3.5 C 伪代码

```c
while (1) {
    int old = xchg(&mutex, 1);

    if (old == 0)
        break;

    yield();
}
```

---

### 3.6 临界区

```asm
mov  count, %ax
add  $1, %ax
mov  %ax, count
```

等价于：

```c
count = count + 1;
```

---

### 3.7 释放锁

```asm
mov  $0, mutex
```

等价于：

```c
mutex = 0;
```

表示锁被释放。

---

### 3.8 Yield 锁的优缺点

优点：

```text
1. 使用 xchg 保证互斥。
2. 拿不到锁时主动 yield，减少单 CPU 上的无效自旋。
3. 持锁线程更容易被调度回来释放锁。
```

缺点：

```text
1. yield 不保证公平。
2. 醒来后仍然要重新竞争锁。
3. 竞争激烈时可能频繁切换。
4. 仍然可能出现线程长期拿不到锁的情况。
```

---

## 4. Test-and-Set 与 Yield 的比较

### 4.1 Test-and-Set 的问题

普通 test-and-set 锁通常是：

```asm
.acquire
mov  $1, %ax        
xchg %ax, mutex
test $0, %ax
jne  .acquire
```

如果锁已经被持有，那么等待线程会不断执行：

```asm
mov  $1, %ax
xchg %ax, mutex
test $0, %ax
jne  .acquire
```

这会浪费大量 CPU 指令。

---

### 4.2 Yield 锁的改进

Yield 锁在失败后执行：

```asm
yield
j .acquire
```

也就是：

```text
拿不到锁时，不继续空转，而是主动让出 CPU。
```

---

### 4.3 什么时候 Yield 能节省指令？

典型场景：

```text
1. 单 CPU。
2. 线程 A 已经拿到锁。
3. 线程 A 还没释放锁就被切走。
4. 线程 B 开始运行并尝试获取锁。
```

在普通 test-and-set 中：

```text
线程 B 会一直自旋，浪费指令。
```

在 yield 锁中：

```text
线程 B 发现锁被占用后，会 yield，让线程 A 更可能运行并释放锁。
```

---

### 4.4 如何计算节省了多少指令？

分别运行：

```bash
python3 ./x86.py -p test-and-set.s -a bx=100,bx=100 -M mutex,count -i 5 -c -S
```

```bash
python3 ./x86.py -p yield.s -a bx=100,bx=100 -M mutex,count -i 5 -c -S
```

重点比较：

```text
STATS:: Instructions
```

节省的指令数为：

```text
节省指令数 = test-and-set.s 的 Instructions - yield.s 的 Instructions
```

例如：

```text
test-and-set.s: Instructions = 5000
yield.s:        Instructions = 3200
```

则：

```text
节省指令数 = 5000 - 3200 = 1800
```

这些节省主要来自减少了等待线程在 `.acquire` 中的空转。

## 5. Test-and-Test-and-Set 锁

### 5.1 代码

```asm
.var mutex
.var count

.main
.top    

.acquire
mov  mutex, %ax
test $0, %ax
jne .acquire

mov  $1, %ax        
xchg %ax, mutex     # atomic swap of 1 and mutex
test $0, %ax        # if we get 0 back: lock is free!
jne .acquire        # if not, try again

# critical section
mov  count, %ax
add  $1, %ax
mov  %ax, count

# release lock
mov  $0, mutex

# see if we're still looping
sub  $1, %bx
test $0, %bx
jgt .top    

halt
```

---

### 5.2 核心思想

这段代码是 **test-and-test-and-set lock**。

它比普通 test-and-set 多了一步普通检查：

```asm
mov  mutex, %ax
test $0, %ax
jne .acquire
```

意思是：

```text
先普通读取 mutex。
如果 mutex != 0，说明锁明显被占用，就继续普通等待。
只有 mutex 看起来为 0 时，才执行昂贵的 xchg。
```

---

### 5.3 为什么要先普通读取？

普通 test-and-set 在锁被占用时，会反复执行：

```asm
xchg %ax, mutex
```

`xchg` 是原子操作，通常比普通读更昂贵，会带来更大的缓存一致性和总线开销。

test-and-test-and-set 的改进是：

```text
锁明显被占用时，只普通读取 mutex。
锁看起来空闲时，才用 xchg 原子尝试获取锁。
```

---

### 5.4 获取锁过程

第一阶段：普通检查。

```asm
mov  mutex, %ax
test $0, %ax
jne .acquire
```

如果：

```text
mutex != 0
```

说明锁被占用，继续等待。

第二阶段：原子尝试获取。

```asm
mov  $1, %ax        
xchg %ax, mutex
test $0, %ax
jne .acquire
```

如果 `xchg` 返回旧值 0，说明获取锁成功。

如果返回旧值 1，说明刚才看到锁空闲后，有其他线程先抢到了锁，因此继续等待。

---

### 5.5 C 伪代码

```c
while (1) {
    while (mutex != 0)
        ;

    old = xchg(&mutex, 1);

    if (old == 0)
        break;
}
```

---

### 5.6 优缺点

优点：

```text
1. 仍然使用 xchg 保证互斥。
2. 锁被占用时避免反复执行昂贵的原子交换。
3. 在多核竞争下，可以减少总线和缓存一致性开销。
```

缺点：

```text
1. 线程仍然在自旋等待。
2. 单 CPU 上，如果持锁线程被切走，等待线程仍然会浪费 CPU。
3. 多个线程同时看到 mutex=0 后，仍然会一起竞争 xchg。
4. 不保证严格公平。
```

---

## 6. Linux Futex 锁

### 6.1 代码

```c
void mutex_lock (int *mutex) {
    int v;

    /* Bit 31 was clear, we got the mutex (this is the fastpath) */
    if (atomic_bit_test_set(mutex, 31) == 0)
        return;

    atomic_increment(mutex);

    while (1) {
        if (atomic_bit_test_set(mutex, 31) == 0) {
            atomic_decrement(mutex);
            return;
        }

        /* We have to wait now. First make sure the futex value
           we are monitoring is truly negative (i.e. locked). */
        v = *mutex;

        if (v >= 0)
            continue;

        futex_wait(mutex, v);
    }
}

void mutex_unlock (int *mutex) {
    /* Adding 0x80000000 to the counter results in 0 if and only if
       there are not other interested threads */
    if (atomic_add_zero(mutex, 0x80000000))
        return;

    /* There are other threads waiting for this mutex,
       wake one of them up. */
    futex_wake(mutex);
}
```

---

### 6.2 `mutex` 是什么？

函数参数是：

```c
void mutex_lock(int *mutex)
```

所以：

```text
mutex 的类型是 int *
```

它是一个指针，指向某个整数。

这个整数 `*mutex` 被用来保存锁状态。

因此：

```c
atomic_increment(mutex);
```

不是让指针 `mutex` 自己加 1，而是对它指向的整数做原子加 1。

逻辑上等价于：

```c
(*mutex)++;
```

但区别是：

```text
atomic_increment 是原子的。
```

---

### 6.3 锁状态如何存储？

这个整数 `*mutex` 同时记录两类信息：

```text
最高位 bit 31：表示锁是否被持有
其余位：表示等待者数量或感兴趣线程数量
```

可以粗略理解为：

```text
bit 31 = 0：锁空闲
bit 31 = 1：锁被占用
```

在有符号整数中，如果最高位是 1，这个数通常表现为负数。

所以代码里有：

```c
v = *mutex;

if (v >= 0)
    continue;

futex_wait(mutex, v);
```

含义是：

```text
如果 v >= 0，说明锁状态可能已经变化，不应该睡眠，继续尝试。
如果 v < 0，说明最高位为 1，锁仍被占用，可以睡眠等待。
```

---

### 6.4 快速路径 fast path

```c
if (atomic_bit_test_set(mutex, 31) == 0)
    return;
```

`atomic_bit_test_set(mutex, 31)` 的意思是：

```text
原子地测试 mutex 指向整数的第 31 位，并把第 31 位设置为 1。
```

如果返回 0，说明第 31 位原来是 0：

```text
锁原来空闲。
```

当前线程已经原子地把第 31 位设置为 1：

```text
当前线程成功拿到锁。
```

于是直接返回。

这就是 fast path。

无竞争情况下，加锁只需要一次用户态原子操作，不需要进入内核。

---

### 6.5 竞争路径 slow path

如果 fast path 失败，说明锁已经被别人持有。

```c
atomic_increment(mutex);
```

意思是：

```text
当前线程把自己记入等待者或感兴趣线程数量。
```

然后进入循环：

```c
while (1) {
    if (atomic_bit_test_set(mutex, 31) == 0) {
        atomic_decrement(mutex);
        return;
    }

    v = *mutex;

    if (v >= 0)
        continue;

    futex_wait(mutex, v);
}
```

---

### 6.6 为什么成功后要 `atomic_decrement(mutex)`？

```c
if (atomic_bit_test_set(mutex, 31) == 0) {
    atomic_decrement(mutex);
    return;
}
```

因为之前失败时执行过：

```c
atomic_increment(mutex);
```

这表示：

```text
我正在等待这把锁。
```

现在当前线程已经成功拿到锁了，所以它不再是等待者，需要把等待者计数减掉。

---

### 6.7 `futex_wait(mutex, v)` 的作用

`futex_wait(mutex, v)` 的语义可以理解为：

```text
如果 *mutex 仍然等于 v，就让当前线程睡眠；
如果 *mutex 已经不等于 v，就不要睡眠，直接返回。
```

它的重要作用是避免丢失唤醒。

例如：

```text
线程 A 准备睡眠
线程 B 正好释放锁并唤醒等待者
```

如果没有原子检查，线程 A 可能错过这次唤醒，然后睡下去，再也醒不过来。

而 `futex_wait(mutex, v)` 会在内核中原子完成：

```text
检查 *mutex 是否仍然等于 v
如果等于，才睡眠
如果不等于，直接返回
```

---

### 6.8 解锁过程

```c
void mutex_unlock (int *mutex) {
    if (atomic_add_zero(mutex, 0x80000000))
        return;

    futex_wake(mutex);
}
```

这里：

```c
atomic_add_zero(mutex, 0x80000000)
```

可以理解为：

```text
清掉最高位，也就是释放锁。
```

因为 `0x80000000` 对应第 31 位。

如果加完之后结果为 0，说明：

```text
锁已经释放，并且没有等待者。
```

于是直接返回。

如果结果不是 0，说明：

```text
还有等待线程。
```

于是调用：

```c
futex_wake(mutex);
```

唤醒一个等待线程。

---

### 6.9 Futex 锁的优点

这段 futex 锁综合了前面几种锁的优点：

```text
1. 像 test-and-set 一样，用原子操作保证互斥。
2. 无竞争时走 fast path，不进入内核，开销很小。
3. 有竞争时可以 futex_wait 睡眠，避免长时间自旋。
4. 解锁时只有发现有等待者，才 futex_wake。
5. futex_wait(address, expected) 可以避免丢失唤醒。
```

它本质上是一个混合式锁：

```text
第一阶段：用户态原子操作快速尝试拿锁。
第二阶段：竞争失败时进入内核 futex 睡眠。
```

---

### 6.10 Futex 锁是否保证公平？

不严格保证。

Ticket lock 是严格排队：

```text
线程 A 拿到票号 0
线程 B 拿到票号 1
线程 C 拿到票号 2

进入顺序必须是 0 -> 1 -> 2
```

而 futex 锁不是严格排队。

释放锁时：

```c
futex_wake(mutex);
```

只是唤醒一个等待线程。

但是被唤醒的线程：

```text
不一定立刻运行；
醒来后也不是自动获得锁；
仍然要重新竞争锁。
```

可能出现：

```text
线程 A 释放锁并唤醒 B
B 还没来得及运行
线程 C 先运行并抢到锁
```

这说明新来的线程可能插队。

所以 futex 锁主要保证：

```text
互斥和效率
```

而不是严格保证：

```text
先等待的线程一定先拿到锁
```

## 7. Futex、Pthread 和 POSIX 的关系

### 7.1 总体层次

可以按下面的层次理解：

```text
POSIX 标准
   ↓
pthread 线程库 API
   ↓
glibc / musl 等 C 库实现
   ↓
Linux 内核 futex 机制
   ↓
硬件原子指令，如 xchg / cmpxchg / fetchadd
```

---

### 7.2 POSIX 是什么？

POSIX 是一套操作系统接口标准。

它规定类 Unix 系统应该提供哪些接口，例如：

```c
pthread_create()
pthread_join()
pthread_mutex_lock()
pthread_mutex_unlock()
pthread_cond_wait()
pthread_cond_signal()
open()
read()
write()
fork()
exec()
```

POSIX 不是一个具体库，也不是一个具体操作系统，而是一套标准规范。

它规定：

```text
应该有什么函数；
函数参数是什么；
函数语义是什么。
```

但它不规定底层必须怎么实现。

---

### 7.3 pthread 是什么？

pthread 是 POSIX threads，也就是 POSIX 线程库。

它提供用户程序直接调用的线程接口：

```c
#include <pthread.h>

pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;

pthread_mutex_lock(&lock);
count++;
pthread_mutex_unlock(&lock);
```

其中：

```c
pthread_mutex_lock()
pthread_mutex_unlock()
```

就是 pthread 提供的高级锁接口。

---

### 7.4 futex 是什么？

futex 是 Linux 内核提供的底层同步机制。

futex 可以理解为：

```text
fast userspace mutex
```

意思是快速用户态互斥锁机制。

它的核心思想是：

```text
无竞争时：尽量在用户态解决，不进入内核。
有竞争时：需要睡眠或唤醒，再进入内核。
```

futex 不是 POSIX 标准的一部分，而是 Linux 特有的底层机制。

---

### 7.5 pthread 和 futex 的关系

用户程序调用：

```c
pthread_mutex_lock(&lock);
```

表面上调用的是 pthread API。

但在 Linux 上，pthread 库底层通常会用 futex 实现阻塞和唤醒。

大致过程是：

```text
pthread_mutex_lock()
    先在用户态用原子指令尝试拿锁
    如果成功，直接返回
    如果失败，调用 futex_wait() 让线程睡眠

pthread_mutex_unlock()
    在用户态释放锁
    如果发现有人在等，调用 futex_wake() 唤醒等待线程
```

所以：

```text
pthread_mutex 是程序员使用的高级接口；
futex 是 Linux 内核提供给 pthread 库实现高效锁的底层机制。
```

## 8. 几种锁的对比

| 锁机制                   | 核心思想             |  是否自旋 |  是否公平 | 是否进入内核 | 主要问题          |
| --------------------- | ---------------- | ----: | ----: | -----: | ------------- |
| test-and-set          | 用 `xchg` 反复抢锁    |     是 |     否 |      否 | 浪费 CPU，可能不公平  |
| yield lock            | 抢不到锁就 `yield`    |    部分 |     否 | 可能依赖调度 | 不保证公平，可能频繁切换  |
| ticket lock           | 取票排队，按号进入        |     是 |     是 |      否 | 公平但仍然自旋       |
| test-and-test-and-set | 先普通读，空闲时再 `xchg` |     是 |     否 |      否 | 减少总线开销，但仍自旋   |
| futex lock            | 用户态快路径，竞争时睡眠     |    少量 |   不严格 |  竞争时进入 | 实现复杂，不严格 FIFO |
| pthread mutex         | 程序员使用的高级互斥锁接口    | 取决于实现 | 取决于实现 |  取决于实现 | 底层实现复杂        |

## 9. 易错点总结

### 9.1 `mutex` 是指针，不是锁变量本身

在 futex 代码中：

```c
void mutex_lock(int *mutex)
```

`mutex` 是指针。

所以：

```c
atomic_increment(mutex);
```

修改的是：

```c
*mutex
```

不是：

```c
mutex
```

也就是说，它不是让指针地址加 1，而是让锁状态整数原子加 1。

---

### 9.2 `futex_wake()` 不是直接把锁交给等待线程

`futex_wake(mutex)` 只是唤醒一个等待线程。

被唤醒线程：

```text
只是从睡眠状态变成可运行状态；
不一定马上运行；
醒来后仍然需要重新抢锁。
```

因此 futex 锁不严格公平。

---

### 9.3 Ticket lock 公平，但仍然浪费 CPU

Ticket lock 按票号顺序进入临界区，公平性好。

但是等待线程执行：

```c
while (turn != my_ticket)
    ;
```

仍然是自旋。

所以它解决了公平性问题，但没有解决 CPU 空转问题。

---

### 9.4 Yield 不是睡眠

`yield` 的意思是：

```text
主动让出 CPU。
```

但线程仍然处于可运行状态，之后还会被调度回来重新竞争锁。

它不是像 `futex_wait()` 那样进入内核睡眠等待唤醒。

---

### 9.5 Test-and-test-and-set 仍然不是公平锁

它只是减少了反复 `xchg` 带来的总线和缓存一致性开销。

它没有排队机制，所以仍然可能出现某些线程长期拿不到锁。

---

## 10. 简短记忆版

```text
test-and-set：
    一直用原子 xchg 抢锁，正确但浪费 CPU。

yield lock：
    抢不到锁就 yield，减少单 CPU 上的无效自旋，但不公平。

ticket lock：
    像银行取号，按票号进入，公平但仍然自旋。

test-and-test-and-set：
    先普通读 mutex，锁看起来空闲时才 xchg，减少原子操作开销。

futex：
    无竞争时用户态快速拿锁；
    有竞争时进入内核睡眠；
    高效但不严格公平。

pthread：
    程序员直接用的 POSIX 线程接口；
    Linux 上底层通常借助 futex 实现高效 mutex。

POSIX：
    操作系统接口标准，规定 pthread 等 API 的语义。
```
