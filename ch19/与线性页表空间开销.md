# OSTEP 笔记：TLB、Cache 与线性页表空间开销

> 来源：Operating Systems: Three Easy Pieces, OSTEP  
> 主题：TLB 地址转换、Cache 与 TLB 的关系、TLB 测量实验、线性页表空间开销

---

## 1. TLB 的基本作用

TLB, Translation Lookaside Buffer, 是地址转换旁路缓冲。

CPU 执行程序时，程序使用的是虚拟地址。虚拟地址需要通过页表转换成物理地址。如果每次访存都去内存中查页表，速度会很慢。因此 CPU 使用 TLB 缓存最近使用过的虚拟页到物理页的映射。

基本形式：

```text
VPN -> PFN
```

其中：

```text
VPN = Virtual Page Number，虚拟页号
PFN = Physical Frame Number，物理页框号
```

虚拟地址可以拆成：

```text
虚拟地址 = VPN + offset
```

物理地址可以拆成：

```text
物理地址 = PFN + offset
```

TLB 只负责把 VPN 翻译成 PFN。offset 不需要保存在 TLB 中，因为页内偏移在地址转换前后保持不变。

---

## 2. 为什么 TLB 不需要保存 offset

假设页大小是 4KB：

```text
4KB = 2^12
```

所以页内偏移占低 12 位。

虚拟地址：

```text
| VPN | 12-bit offset |
```

物理地址：

```text
| PFN | 12-bit offset |
```

TLB 中只保存：

```text
VPN -> PFN
```

CPU 查到 PFN 后，把原虚拟地址中的 offset 原封不动拼到 PFN 后面，就得到最终物理地址。

例如：

```text
虚拟地址 = 0x12345ABC

VPN    = 0x12345
offset = 0xABC
```

如果 TLB 中有：

```text
VPN 0x12345 -> PFN 0xABCD
```

则物理地址为：

```text
物理地址 = 0xABCDABC
```

所以：

```text
TLB 只保存 VPN 和 PFN
offset 直接来自原来的虚拟地址
```

---

## 3. TLB 项中保存什么

一条典型 TLB 项可以表示为：

```text
VPN | PFN | 其他位
```

其中：

```text
VPN：用于匹配当前访问的虚拟页
PFN：匹配成功后得到的物理页框号
其他位：valid、dirty、protection、ASID、global 等控制信息
```

核心仍然是：

```text
VPN -> PFN
```

---

## 4. 什么是全相联 TLB

典型 TLB 有 32 项、64 项或 128 项，并且通常是全相联的。

全相联 fully associative 的意思是：

> 一条地址映射可以放在 TLB 中的任意位置。

例如一条映射：

```text
VPN 7 -> PFN 100
```

可以放在 TLB 第 0 项、第 1 项、第 2 项，或者任意一项。

所以硬件查找时，不能只查某个固定位置，而是并行比较所有 TLB 项：

```text
目标 VPN == 第 0 项 VPN？
目标 VPN == 第 1 项 VPN？
目标 VPN == 第 2 项 VPN？
...
```

如果某一项 VPN 匹配，就取出对应 PFN。

---

## 5. 全相联与直接映射的区别

### 5.1 直接映射

如果是直接映射，某个 VPN 只能放到固定位置。

例如：

```text
位置 = VPN % TLB项数
```

如果 TLB 有 4 项：

```text
VPN 7  -> 第 3 项
VPN 11 -> 第 3 项
```

那么 VPN 7 和 VPN 11 会冲突。

---

### 5.2 全相联

全相联中，任意 VPN 可以放在任意 TLB 项中。

```text
VPN 7  可以放第 0 项
VPN 11 可以放第 1 项
```

优点是冲突少，命中率高。

缺点是硬件复杂，因为需要并行比较所有项。但 TLB 本身通常很小，所以这个代价可以接受。

---

## 6. TLB miss 处理中的无限递归问题

TLB miss 发生时，CPU 会进入操作系统的 TLB miss handler。

但是 handler 本身也是一段代码，也需要 CPU 取指执行。如果 handler 的代码地址也需要经过 TLB 翻译，而 TLB 中又没有对应项，就会出现：

```text
用户程序访问地址 A
        ↓
TLB miss
        ↓
进入 TLB miss handler
        ↓
handler 代码地址也 TLB miss
        ↓
再次进入 TLB miss handler
        ↓
无限递归
```

这就是 TLB miss 处理中的无限递归问题。

---

## 7. 如何避免 TLB miss 递归

常见方法有两种。

### 7.1 将 TLB miss handler 放在 unmapped 物理内存中

unmapped 不是没有内存，而是：

> 这段代码不通过普通虚拟地址映射机制访问。

CPU 可以直接执行这部分代码，不需要 TLB 地址转换。

---

### 7.2 使用 wired TLB entries

wired entries 是固定在 TLB 中的项。

这些项不会被普通替换算法赶出去。

操作系统可以把以下内容的映射固定在 TLB 中：

```text
TLB miss handler 代码
内核栈
页表访问所需的关键数据结构
```

这样处理 TLB miss 时，这些关键地址总能命中 TLB。

---

## 8. TLB 替换策略：LRU 与 Random

当 TLB 满了，需要替换旧项。常见策略包括：

```text
LRU
Random
```

---

## 9. LRU 策略

LRU, Least Recently Used，替换最近最少使用的项。

它的假设是：

> 最近很久没有用过的项，未来可能也不会很快使用。

这种策略利用了程序访问的局部性。

---

## 10. LRU 的极端失败情况

假设：

```text
TLB 容量 = n
程序循环访问 n + 1 个页
```

例如 TLB 只能放 3 个页，但程序循环访问 4 个页：

```text
A, B, C, D, A, B, C, D, ...
```

过程如下：

```text
访问 A：miss，放入 A
访问 B：miss，放入 B
访问 C：miss，放入 C
访问 D：miss，换出 A
访问 A：miss，换出 B
访问 B：miss，换出 C
访问 C：miss，换出 D
访问 D：miss，换出 A
...
```

此时每次要访问的页，刚好都是刚被 LRU 换出去的页。

所以 LRU 命中率可能接近 0%。

---

## 11. 为什么 Random 可能更好

Random 策略随机选择一项换出。

在上面的极端访问模式中，Random 不一定每次都换出下一次马上要访问的页。

因此 Random 虽然看起来简单，但不会稳定地犯同一个错误。

所以在某些极端情况下，Random 反而比 LRU 表现更好。

---

## 12. MIPS R4000 TLB 项

MIPS R4000 的 TLB 项包含更多字段，例如：

```text
VPN | PFN | G | C | D | V | ASID
```

核心仍然是：

```text
VPN -> PFN
```

但同时还包含权限、有效性、进程区分、缓存控制等信息。

---

## 13. VPN 与 PFN 位数

假设：

```text
虚拟地址 = 32 位
页大小 = 4KB = 2^12
```

则理论上：

```text
VPN 位数 = 32 - 12 = 20 位
```

MIPS R4000 中，用户地址空间只占 32 位地址空间的一部分，因此 VPN 可能只需要 19 位。

PFN 最多可以有 24 位。

因此物理地址最大可以是：

```text
PFN 24 位 + offset 12 位 = 36 位
```

对应物理地址空间：

```text
2^36 = 64GB
```

这说明：

> 虚拟地址空间大小和物理地址空间大小不一定相同。

---

## 14. 虚拟地址不是一对多映射

一个进程中的一个虚拟页，在某一时刻通常只能映射到一个物理页框。

不能出现：

```text
同一个进程中：
VPN X -> PFN A
VPN X -> PFN B
```

否则 CPU 不知道应该访问哪个物理地址。

但是，不同进程可以有相同 VPN，映射到不同 PFN：

```text
进程 A：
VPN 0x1000 -> PFN 0xAAAA

进程 B：
VPN 0x1000 -> PFN 0xBBBB
```

所以需要 ASID 区分不同进程的地址空间。

---

## 15. ASID 的作用

ASID, Address Space Identifier，是地址空间标识符。

它用于区分不同进程的 TLB 项。

TLB 查找时，不仅比较 VPN，还比较 ASID：

```text
当前进程 ASID == TLB 项 ASID？
目标 VPN == TLB 项 VPN？
```

如果没有 ASID，进程切换时通常需要清空 TLB，否则新进程可能误用旧进程的地址映射。

有了 ASID，不同进程的 TLB 项可以共存。

---

## 16. G 位：Global

G 是 Global 位。

```text
G = 1：全局映射，忽略 ASID
G = 0：进程私有映射，需要检查 ASID
```

内核代码、内核固定映射等所有进程共享的地址可以使用全局映射。

---

## 17. V 位：Valid

V 是有效位。

```text
V = 1：TLB 项有效
V = 0：TLB 项无效
```

如果 V 位为 0，即使 VPN 匹配，也不能认为 TLB 命中。

---

## 18. D 位：Dirty

D 是脏位。

它表示页面是否被修改过，或是否允许写入。

脏位可以帮助操作系统判断页面换出时是否需要写回磁盘。

---

## 19. C 位：Cache 控制位

C 位用于控制 cache 行为，例如：

```text
页面是否可缓存
使用哪种缓存策略
是否需要特殊一致性处理
```

这与 cache 一致性、硬件缓存策略相关。

---

## 20. Page Mask

Page mask 用于支持不同大小的页面。

例如：

```text
4KB
16KB
64KB
1MB
```

页面越大，offset 位数越多，VPN 位数越少。

大页的好处是：

```text
一条 TLB 项可以覆盖更大的地址范围
减少 TLB miss
```

---

## 21. MIPS 软件管理 TLB

MIPS R4000 采用软件管理 TLB。

发生 TLB miss 后：

```text
硬件触发异常
        ↓
操作系统查页表
        ↓
操作系统将映射写入 TLB
        ↓
返回用户程序继续执行
```

这与 x86 中硬件自动页表遍历的方式不同。

---

## 22. MIPS TLB 管理指令

MIPS 提供特权指令管理 TLB。

### 22.1 TLBP

Probe，查找 TLB 中是否有某个 VPN/ASID 的映射。

```text
TLBP
```

---

### 22.2 TLBR

Read，读取某个 TLB 项到寄存器。

```text
TLBR
```

---

### 22.3 TLBWI

Write Indexed，写入指定位置。

```text
TLBWI
```

---

### 22.4 TLBWR

Write Random，随机写入一个可替换 TLB 项。

```text
TLBWR
```

这些指令是特权指令，普通用户程序不能执行。

否则用户程序可以任意修改地址映射，破坏内存保护。

---

## 23. RAM 不总是 RAM

程序员看起来都是访问内存，但不同内存访问的成本差别很大。

影响因素包括：

```text
TLB hit / miss
cache hit / miss
是否跨页
是否发生 page fault
是否访问特殊映射区域
```

TLB miss 可能需要：

```text
陷入内核
查页表
更新 TLB
返回用户程序
```

如果页面不在物理内存中，还可能触发缺页，从磁盘读取，代价更大。

---

# Cache 与 TLB 的关系

## 24. Cache 不是实际内存

Cache 是 CPU 和主存 RAM 之间的高速缓存。

CPU 访问内存时一般流程是：

```text
CPU 产生地址
        ↓
查 cache
        ↓
cache hit：直接返回数据
cache miss：访问 RAM，将数据加载到 cache
```

Cache 中保存的是主存中一部分数据的副本。

---

## 25. Cache 的基本单位是 cache line

Cache 不是按页存储，而是按 cache line 存储。

常见大小：

```text
页 page = 4KB
cache line = 64B
```

因此一页包含：

```text
4096 / 64 = 64 条 cache line
```

所以不能理解成：

```text
一个 cache 存一页
一个 way 存一页
```

正确理解是：

```text
一个页会被切成很多条 cache line
cache 中存的是这些 cache line
```

---

## 26. Cache 地址拆分

Cache 查找时，地址通常拆成：

```text
| tag | index | block offset |
```

含义：

```text
block offset：cache line 内的字节偏移
index：选择 cache 的哪一组
tag：确认该组中的数据是否就是目标地址
```

---

## 27. block offset 为什么常为 6 位

如果 cache line 大小为 64B：

```text
64 = 2^6
```

则需要 6 位表示一条 cache line 内的 64 个字节。

所以：

```text
block offset = 6 位
```

---

## 28. index 为什么可以是 6 位

如果 cache 有 64 组：

```text
64 = 2^6
```

则需要 6 位表示组号。

所以：

```text
index = 6 位
```

---

## 29. 12 位页内偏移如何拆分

假设：

```text
页大小 = 4KB = 2^12
cache line = 64B = 2^6
cache 组数 = 64 = 2^6
```

则：

```text
页内 offset = 12 位
block offset = 6 位
index = 6 位
```

刚好：

```text
12 位页内 offset = 6 位 index + 6 位 block offset
```

地址形式：

```text
虚拟地址：
| VPN | index | block offset |
      |<--- 12-bit page offset --->|
```

---

## 30. 为什么 index 对应 cache 的某一组

这是硬件设计规定的。

如果 cache 有 64 组，index 是 6 位：

```text
000000 -> 第 0 组
000001 -> 第 1 组
...
111111 -> 第 63 组
```

可以理解为：

```c
cache[index]
```

index 只是缩小查找范围，不能唯一确定数据。

最终还要比较 tag。

---

## 31. 组相联 cache 与 way

如果一个 cache 是 4 路组相联，表示每个 set 有 4 个 way。

例如第 17 组：

```text
set 17:
way 0: tag = 0x123, data = 一条 cache line
way 1: tag = 0x456, data = 一条 cache line
way 2: tag = 0x789, data = 一条 cache line
way 3: tag = 0xABC, data = 一条 cache line
```

访问某个地址时：

```text
index = 17
tag = 0x456
```

CPU 会到第 17 组中比较所有 way 的 tag。

如果 way 1 的 tag 匹配，则 cache hit。

---

## 32. 多路不是多个 cache

多路组相联不是多个 cache，而是：

> 一个 set 中有多个槽位，可以容纳多个映射到同一组的 cache line。

直接映射 cache：

```text
每组 1 个 way
```

4 路组相联 cache：

```text
每组 4 个 way
```

多路可以减少冲突 miss。

---

## 33. 物理地址索引 cache

Physically-indexed cache 使用物理地址选择 cache 组。

流程：

```text
虚拟地址 VA
        ↓
TLB 翻译
        ↓
物理地址 PA
        ↓
用 PA 访问 cache
```

缺点是：

```text
必须先完成地址转换，才能访问 cache
```

这会让 TLB 成为 CPU 流水线瓶颈。

---

## 34. 虚拟地址索引 cache

Virtually-indexed cache 使用虚拟地址的一部分先访问 cache。

这样可以让 cache 查找与 TLB 翻译并行。

流程：

```text
VA 的 index 访问 cache
VA 的 VPN 查 TLB
```

优点：

```text
减少等待 TLB 翻译的时间
```

缺点：

```text
可能出现别名问题
不同进程可能有相同虚拟地址
硬件设计更复杂
```

---

## 35. VIPT Cache

VIPT 是：

```text
Virtually Indexed, Physically Tagged
虚拟地址索引，物理地址标记
```

流程：

```text
CPU 得到虚拟地址 VA
        ↓
用 VA 的 index 先选 cache 组
        ↓
同时用 VPN 查 TLB 得到 PFN
        ↓
用 PFN 形成物理 tag
        ↓
与 cache 组中各 way 的 tag 比较
        ↓
匹配则 cache hit
不匹配则 cache miss
```

关键点：

```text
index 用虚拟地址
tag 用物理地址
```

---

## 36. VIPT 为什么要求 index 来自页内 offset

页内 offset 在 VA 和 PA 中相同。

```text
VA = VPN | offset
PA = PFN | offset
```

如果 cache 的：

```text
index + block offset
```

全部来自页内 offset，那么：

```text
VA 选出的 cache 组 == PA 选出的 cache 组
```

这样可以安全地提前用虚拟地址索引 cache。

---

## 37. VIPT 的约束

为了让 index 和 block offset 全部落在页内 offset 中，需要满足：

```text
index 位数 + block offset 位数 <= 页内 offset 位数
```

等价地：

```text
cache 组数 × cache line 大小 <= 页大小
```

例如：

```text
页大小 = 4KB
cache line = 64B
cache 组数 = 64
```

则：

```text
64 × 64B = 4096B = 4KB
```

满足要求。

---

## 38. Cache 一般有多大

常见 CPU cache 大小：

```text
L1 cache：几十 KB
L2 cache：几百 KB 到几 MB
L3 cache：几 MB 到几十 MB，甚至更大
```

典型配置：

```text
L1 I-cache：32KB 或 64KB
L1 D-cache：32KB 或 64KB
L2 cache：256KB、512KB、1MB、2MB
L3 cache：8MB、16MB、32MB、64MB
```

例如：

```text
L1 D-cache = 32KB
cache line = 64B
8-way set associative
```

则：

```text
cache line 总数 = 32KB / 64B = 512
set 数 = 512 / 8 = 64
index 位数 = log2(64) = 6
block offset 位数 = log2(64B) = 6
```

因此：

```text
index + block offset = 12 位
```

刚好等于 4KB 页的页内 offset 位数。

# TLB 测量实验

## 39. 实验目标

通过程序测量访问不同数量页时的平均访问时间，从而估计 TLB 容量和 TLB miss 开销。

核心思想：

```text
访问页数 <= TLB 容量：大多 TLB hit，访问快
访问页数 > TLB 容量：TLB miss 增多，访问变慢
```

---

## 40. 为什么每次跨一页访问

假设：

```text
页大小 = 4096B
sizeof(int) = 4B
```

则一页中有：

```text
4096 / 4 = 1024 个 int
```

所以：

```c
int jump = PAGESIZE / sizeof(int);
```

如果每次访问：

```c
a[i]
i += jump
```

则访问序列为：

```text
a[0]       -> 第 0 页
a[1024]    -> 第 1 页
a[2048]    -> 第 2 页
a[3072]    -> 第 3 页
...
```

这样每次访问都落在不同页上，能制造 TLB 压力。

---

## 41. tlb.c 示例程序

```c
#include <stdio.h>
#include <stdlib.h>
#include <sys/time.h>
#include <unistd.h>

double get_time_in_seconds() {
    struct timeval tv;
    gettimeofday(&tv, NULL);
    return tv.tv_sec + tv.tv_usec / 1000000.0;
}

int main(int argc, char *argv[]) {
    if (argc != 3) {
        fprintf(stderr, "Usage: %s <num_pages> <num_trials>\n", argv[0]);
        exit(1);
    }

    int num_pages = atoi(argv[1]);
    int num_trials = atoi(argv[2]);

    if (num_pages <= 0 || num_trials <= 0) {
        fprintf(stderr, "num_pages and num_trials must be positive.\n");
        exit(1);
    }

    int page_size = getpagesize();
    int jump = page_size / sizeof(int);
    int total_ints = num_pages * jump;

    int *a = (int *)malloc(total_ints * sizeof(int));
    if (a == NULL) {
        perror("malloc");
        exit(1);
    }

    // 初始化数组，避免正式计时中包含首次缺页和清零开销
    for (int i = 0; i < total_ints; i++) {
        a[i] = 0;
    }

    // 预热，按正式访问模式先访问一遍
    for (int i = 0; i < total_ints; i += jump) {
        a[i] += 1;
    }

    double start = get_time_in_seconds();

    for (int t = 0; t < num_trials; t++) {
        for (int i = 0; i < total_ints; i += jump) {
            a[i] += 1;
        }
    }

    double end = get_time_in_seconds();

    double elapsed = end - start;
    long long total_accesses = (long long)num_pages * num_trials;
    double time_per_access_ns = elapsed * 1e9 / total_accesses;

    printf("Number of pages: %d\n", num_pages);
    printf("Number of trials: %d\n", num_trials);
    printf("Page size: %d bytes\n", page_size);
    printf("Total time: %.6f seconds\n", elapsed);
    printf("Average time per access: %.2f ns\n", time_per_access_ns);

    free(a);
    return 0;
}
```

```bash

```

参数为：

```text
argv[0] = "./tlb"
argv[1] = "16"
argv[2] = "1000000"
argc = 3
```

所以代码：

```c
if (argc != 3) {
    fprintf(stderr, "Usage: %s <num_pages> <num_trials>\n", argv[0]);
    exit(1);
}
```

意思是：

```text
如果用户没有输入两个参数，就打印正确用法并退出
```

## 43. 为什么需要初始化数组

`malloc()` 通常只分配虚拟地址空间，不一定立即分配物理页。

第一次访问某个页面时，可能发生：

```text
访问虚拟地址
        ↓
页面还没有物理页
        ↓
触发 page fault
        ↓
操作系统分配物理页
        ↓
将物理页清零
        ↓
建立页表映射
        ↓
返回程序继续执行
```

这个过程远比普通内存访问慢。

因此正式计时前需要初始化：

```c
for (int i = 0; i < total_ints; i++) {
    a[i] = 0;
}
```

这样可以提前消除首次缺页和清零开销。

---

## 44. 为什么需要预热

初始化后再按正式访问模式访问一遍：

```c
for (int i = 0; i < total_ints; i += jump) {
    a[i] += 1;
}
```

作用是减少以下干扰：

```text
第一次 TLB 填充
第一次 cache miss
页表项缓存状态不稳定
CPU 流水线和分支预测未稳定
```

正式计时应该从预热之后开始。

---

## 45. 为什么乘 1e9

代码：

```c
double time_per_access_ns = elapsed * 1e9 / total_accesses;
```

其中：

```text
elapsed 单位是秒
total_accesses 是访问次数
```

因为：

```text
1s = 10^9 ns
```

所以：

```text
elapsed * 1e9
```

把总时间从秒转换成纳秒。

最后除以访问次数，得到：

```text
平均每次访问时间，单位 ns
```

## 47. gettimeofday() 精度问题

`gettimeofday()` 返回秒和微秒，表面分辨率为微秒级。

一次内存访问可能只有几 ns 到几十 ns，远小于计时器分辨率。

所以不能只测一次访问。

正确做法是：

```text
重复访问很多次
测总时间
总时间 / 总访问次数 = 平均每次访问时间
```

例如如果一次访问约 20ns，希望总时间达到 100ms：

```text
100ms = 100,000,000ns
```

需要访问次数约：

```text
100,000,000 / 20 = 5,000,000
```

即约 500 万次访问。

## 48. 多 CPU 对 TLB 测量的影响

现代系统有多个 CPU 核心，每个核心通常有自己的 TLB。

如果程序一开始在 CPU 0 上运行，后来被调度到 CPU 1：

```text
CPU 0 的 TLB 中有之前访问过的映射
CPU 1 的 TLB 中没有这些映射
```

程序迁移后可能出现额外 TLB miss，导致测量结果抖动。

因此测试时最好将程序固定到一个 CPU 上运行。

```text
只允许程序在 CPU 0 上运行
```

也可以固定到其他 CPU：

```bash
taskset -c 2 ./tlb 4096 1000000
```

## 50. 在 C 程序中设置 CPU affinity

可以在程序中加入：

```c
#define _GNU_SOURCE
#include <sched.h>
#include <stdio.h>
#include <stdlib.h>
```

在 `main()` 开头加入：

```c
cpu_set_t set;

CPU_ZERO(&set);
CPU_SET(0, &set);

if (sched_setaffinity(0, sizeof(set), &set) != 0) {
    perror("sched_setaffinity");
    exit(1);
}
```

含义：

```text
将当前进程绑定到 CPU 0
```

## 51. 为什么实测到 4096 页仍然可能只有 5ns

如果实验结果显示：

```text
访问 4096 页仍然约 5ns
```

不一定是程序错了，可能原因包括：

### 51.1 透明大页

现代系统可能使用 transparent huge page。

如果 4KB 页被合并成 2MB 大页：

```text
4096 个 4KB 页 = 16MB
16MB / 2MB = 8 个大页
```

TLB 看到的可能只有 8 个大页映射，因此不会出现明显 TLB miss。

---

### 51.2 现代 CPU TLB 更复杂

现代 CPU 通常有：

```text
一级 data TLB
二级 unified TLB
page walk cache
乱序执行
硬件预取
多级 cache
```

这些机制会隐藏部分 TLB miss 开销。

---

### 51.3 顺序访问太规律

如果访问模式是：

```text
第 0 页，第 1 页，第 2 页，第 3 页，...
```

硬件可能进行预取或优化。

更强的测试方式是随机页访问。

---

### 51.4 编译器优化

如果编译器认为访问结果没有被使用，可能优化循环。

可以考虑使用：

```c
volatile int *a;
```

或者最终输出访问结果，避免优化。

---

## 52. 可改进实验方法

为了更容易测出 TLB 影响，可以：

```text
禁用 transparent huge page
随机访问页顺序
使用 volatile 防止优化
固定在一个 CPU 上运行
增加访问页数
减少系统干扰
```

禁用 huge page 可尝试：

```c
#include <sys/mman.h>

madvise(a, total_ints * sizeof(int), MADV_NOHUGEPAGE);
```

# 线性页表空间开销

## 53. 分页引入的新问题

分页解决了内存分配中的外部碎片问题，也带来了新的问题：

```text
页表太大
页表本身会占用大量内存
```

---

## 54. 什么是线性页表

线性页表可以理解为一个数组：

```text
页表[VPN] = PTE
```

其中 PTE 是 Page Table Entry，页表项。

每个虚拟页都有一个对应页表项。

页表项中通常保存：

```text
PFN
valid bit
protection bits
dirty bit
accessed bit
```

---

## 55. 32 位地址空间中有多少虚拟页

假设：

```text
虚拟地址空间 = 32 位 = 2^32 字节
页大小 = 4KB = 2^12 字节
```

则虚拟页数量：

```text
2^32 / 2^12 = 2^20
```

而：

```text
2^20 = 1,048,576
```

所以一个 32 位地址空间大约有 100 万个虚拟页。

---

## 56. 线性页表大小

如果每个页表项大小为 4 字节，则一个进程的页表大小为：

```text
2^20 × 4B = 4MB
```

也就是说：

```text
一个进程的线性页表就需要 4MB 内存
```

注意这只是页表本身，不包括进程代码、堆、栈等数据。

---

## 57. 多进程下页表开销

每个进程都有自己的地址空间，所以通常每个进程都有自己的页表。

如果有 100 个活动进程：

```text
100 × 4MB = 400MB
```

系统仅页表就可能消耗数百 MB 内存。

---

## 58. 线性页表浪费的根源

线性页表为整个虚拟地址空间中的每个虚拟页都准备一个页表项。

但是进程通常不会使用整个地址空间。

例如实际只使用：

```text
代码段：几 MB
堆：几十 MB
栈：几 MB
共享库：几十 MB
```

中间大量虚拟地址区域根本没用。

但线性页表仍然要为这些未使用区域保留页表项。

---

## 59. 后续优化方向

为减少页表内存开销，后面会引出：

```text
更大的页
分段 + 分页
多级页表
倒排页表
```

核心目标是：

```text
不要为整个虚拟地址空间的所有虚拟页都无条件分配页表项
只为真正使用的虚拟地址区域付出页表开销
```

---

# 易错点总结

## 1. TLB 不保存 offset

错误理解：

```text
TLB 需要保存完整虚拟地址到完整物理地址的映射
```

正确理解：

```text
TLB 保存 VPN -> PFN
offset 直接从虚拟地址中保留
```

---

## 2. 全相联不是一对多映射

错误理解：

```text
全相联表示一个虚拟页可以映射到多个物理页
```

正确理解：

```text
全相联表示一条 TLB 映射可以放在 TLB 的任意位置
同一进程中一个虚拟页通常只能映射到一个物理页框
```

---

## 3. PFN 位数大不代表虚拟地址一对多

错误理解：

```text
物理地址空间更大，所以一个虚拟地址可以对应多个物理地址
```

正确理解：

```text
一个虚拟页仍然映射到一个物理页框
PFN 位数大只是说明这个物理页框可以从更大的物理内存范围中选择
```

---

## 4. Cache 不是按页存储

错误理解：

```text
一个 cache 项存一页
一个 way 存一页
```

正确理解：

```text
cache 存的是 cache line
一页通常包含很多条 cache line
```

---

## 5. index 不能唯一确定数据

错误理解：

```text
index 找到 cache 组后就能确定数据
```

正确理解：

```text
index 只选择 set
最终还需要比较 tag
```

---

## 6. VIPT 不是只靠 offset 找数据

错误理解：

```text
VIPT 用页内 offset 就能直接找到最终数据
```

正确理解：

```text
offset 中的一部分用于提前选择 cache set
TLB 翻译得到 PFN 后，还要用物理 tag 验证是否命中
```

---

## 7. TLB 测量结果不跳变不一定是程序错

可能原因：

```text
透明大页
二级 TLB
page walk cache
硬件预取
顺序访问过于规律
编译器优化
进程被调度到其他 CPU
```

---

## 8. malloc 后不等于物理页已经分配

错误理解：

```text
malloc 后数组已经完全在物理内存中
```

正确理解：

```text
malloc 可能只分配虚拟地址空间
首次访问页面时才真正分配物理页并清零
```

---

## 9. 线性页表的问题不是进程用了 4GB 内存

错误理解：

```text
页表 4MB 是因为进程真的用了 4GB 内存
```

正确理解：

```text
线性页表为整个 4GB 虚拟地址空间的每个虚拟页都准备 PTE
即使很多虚拟页没有被使用，也会占页表空间
```

---

# 简短记忆版

```text
TLB：缓存 VPN -> PFN，offset 不变。
全相联：映射可放任意 TLB 项，硬件并行查找。
TLB miss handler：必须避免自己再次 TLB miss。
LRU：一般好，但 n+1 循环访问会被克制。
Random：简单，但极端情况下可能比 LRU 好。
ASID：区分不同进程的相同 VPN。
G 位：全局映射，忽略 ASID。
Cache：存 cache line，不是页。
VIPT：虚拟 index，物理 tag。
TLB 实验：跨页访问，测平均访问时间。
实验注意：初始化、预热、绑核、禁用大页、避免优化。
线性页表：32 位地址空间 + 4KB 页 + 4B PTE = 每进程 4MB 页表。
```
