# 第 40 章：文件系统实现（VSFS）学习笔记

> 核心主线：**文件名 → 目录项 → inode number → inode → 数据块 → 文件内容**

---

## 1. 文件系统整体组织

文件系统不会直接按字节管理磁盘，而是把磁盘划分成固定大小的块。

在 VSFS 例子中：

```text
块大小 = 4KB
磁盘共有 64 个块
块编号 = 0 ~ 63
```

整体布局大致是：

```text
block 0       : superblock
block 1       : inode bitmap
block 2       : data bitmap
block 3 ~ 7   : inode table
block 8 ~ 63  : data region
```

含义如下：

| 区域 | 作用 |
|---|---|
| superblock | 记录文件系统整体信息 |
| inode bitmap | 记录哪些 inode 空闲，哪些已分配 |
| data bitmap | 记录哪些数据块空闲，哪些已分配 |
| inode table | 存放所有 inode |
| data region | 存放文件和目录的实际数据 |

---

## 2. 位图 bitmap

### 2.1 位图的作用

文件系统需要知道：

```text
哪些 inode 还空闲？
哪些数据块还空闲？
```

VSFS 使用位图记录这些信息。

每个 bit 表示一个对象的状态：

```text
0：空闲，未分配
1：已使用，已分配
```

例如：

```text
数据块编号：0 1 2 3 4 5 6 7
位图：      1 0 1 1 0 0 1 0
```

表示：

```text
数据块 0 已使用
数据块 1 空闲
数据块 2 已使用
数据块 3 已使用
数据块 4 空闲
数据块 5 空闲
数据块 6 已使用
数据块 7 空闲
```

---

### 2.2 为什么一个 4KB 位图可以记录 32K 个对象

关键点：

> 一个 bit 记录一个对象是否被分配，而不是记录一个字节是否被分配。

因为：

```text
1 byte = 8 bit
4KB = 4096 byte
```

所以：

```text
4KB = 4096 × 8 bit = 32768 bit
```

也就是：

```text
32768 bit = 32768 个对象的分配状态
32768 = 32K
```

所以：

```text
一个 4KB 位图块可以记录 32K 个对象是否被分配
```

注意：

```text
这里说的是 32K 个对象，不是 32KB 大小的对象。
```

---

### 2.3 为什么小文件系统中位图看起来很浪费

在 VSFS 例子中：

```text
inode 数量 = 80
数据块数量 = 56
```

理论上只需要：

```text
inode bitmap 需要 80 bit
data bitmap 需要 56 bit
```

但文件系统仍然给每个位图分配一个完整 4KB 块：

```text
inode bitmap 占 4KB
data bitmap 占 4KB
```

这是为了实现简单，因为文件系统通常以块为单位管理磁盘。

---

## 3. inode

### 3.1 inode 是什么

inode 是文件系统中非常重要的数据结构。

可以理解为：

```text
inode = 文件的身份证 + 文件内容索引表
```

inode 不存文件名，也不直接存文件内容，而是存文件的元数据和数据块指针。

inode 中通常包含：

```text
文件类型：普通文件、目录、符号链接等
文件大小：多少字节
权限信息：谁能读、写、执行
所有者信息：用户 ID、组 ID
时间信息：创建时间、修改时间、访问时间等
数据块位置：文件内容存放在哪些磁盘块中
```

---

### 3.2 文件名和 inode 的关系

用户平时看到的是文件名：

```text
foo.txt
bar
hello.c
```

但文件系统内部主要通过 inode number 识别文件。

整体关系是：

```text
文件名 → inode number → inode → 数据块 → 文件内容
```

例如：

```text
foo.txt → inode 12 → inode 表中的第 12 个 inode → 数据块 100、101、102
```

---

### 3.3 inode number 是什么

每个 inode 都有一个编号，叫：

```text
inumber
inode number
```

它可以理解为 inode 表中的下标。

例如：

```text
inode 0
inode 1
inode 2
...
inode 32
```

如果知道某文件的 inode number 是 32，就可以根据 inode 表起始地址和 inode 大小计算出它在磁盘上的位置。

---

### 3.4 如何根据 inumber 找到 inode

假设：

```text
inode 大小 = 256B
inode 表起始地址 = 12KB
要查找 inumber = 32 的 inode
```

第 32 个 inode 在 inode 表内部的偏移量为：

```text
32 × 256B = 8192B = 8KB
```

所以它的磁盘地址是：

```text
inode 地址 = inode 表起始地址 + 偏移量
          = 12KB + 8KB
          = 20KB
```

也就是：

```text
inode 32 位于磁盘 20KB 处
```

---

### 3.5 为什么还要算 sector

磁盘通常不是按字节直接读，而是按扇区或块读。

假设：

```text
sector size = 512B
inode 32 的地址 = 20KB = 20480B
```

那么它所在的扇区号为：

```text
20480 / 512 = 40
```

所以系统会向磁盘请求：

```text
读取第 40 个扇区
```

书中的公式为：

```c
blk = (inumber * sizeof(inode_t)) / blockSize;
sector = ((blk * blockSize) + inodeStartAddr) / sectorSize;
```

含义：

```text
blk：目标 inode 位于 inode 表内部的第几个块
sector：目标 inode 实际对应的磁盘扇区
```

---

## 4. inode 如何记录数据块位置

### 4.1 直接指针

最简单的方法是在 inode 中保存若干个直接指针。

例如 inode 有 12 个直接指针：

```text
inode
 ├── direct pointer 0 → 数据块 A
 ├── direct pointer 1 → 数据块 B
 ├── direct pointer 2 → 数据块 C
 ...
 └── direct pointer 11 → 数据块 L
```

如果：

```text
每个数据块大小 = 4KB
直接指针数量 = 12
```

那么最多能表示：

```text
12 × 4KB = 48KB
```

所以直接指针适合小文件，但无法支持大文件。

---

### 4.2 间接指针

间接指针不是直接指向数据块，而是指向一个“指针块”。

```text
inode
 └── indirect pointer → 指针块
                         ├── 数据块地址 1 → 数据块
                         ├── 数据块地址 2 → 数据块
                         ├── 数据块地址 3 → 数据块
                         ...
```

假设：

```text
块大小 = 4KB = 4096B
一个磁盘地址 = 4B
```

那么一个指针块可以存：

```text
4096 / 4 = 1024 个地址
```

所以一个间接指针可以额外指向：

```text
1024 个数据块
```

对应大小为：

```text
1024 × 4KB = 4096KB = 4MB
```

加上 12 个直接指针：

```text
(12 + 1024) × 4KB = 4144KB
```

---

### 4.3 双重间接指针

双重间接指针多了一层索引。

```text
inode
 └── double indirect pointer
        └── 一级指针块
              ├── 二级指针块 1
              │     ├── 数据块地址 1 → 数据块
              │     ├── 数据块地址 2 → 数据块
              │     ...
              ├── 二级指针块 2
              │     ├── 数据块地址 ...
              └── 二级指针块 1024
                    ├── 数据块地址 ...
```

一个指针块可以存 1024 个地址，所以双重间接指针可以表示：

```text
1024 × 1024 个数据块
```

对应大小：

```text
1024 × 1024 × 4KB = 4GB
```

---

### 4.4 三重间接指针

三重间接指针再多一层：

```text
inode
 └── triple indirect pointer
        └── 一级指针块
              └── 二级指针块
                    └── 三级指针块
                          └── 数据块
```

可以表示：

```text
1024 × 1024 × 1024 个数据块
```

支持非常大的文件。

---

### 4.5 多级索引的整体结构

传统 UNIX 文件系统常见 inode 结构：

```text
inode
 ├── 12 个直接指针
 ├── 1 个间接指针
 ├── 1 个双重间接指针
 └── 1 个三重间接指针
```

设计思想：

```text
小文件：用直接指针，访问快
中等文件：用间接指针
大文件：用双重间接指针
超大文件：用三重间接指针
```

---

### 4.6 为什么这种结构看起来“不平衡”

inode 中通常有很多直接指针，但只有少量间接指针。

原因是：

```text
大多数文件很小
少数大文件占用了大量磁盘空间
```

所以文件系统优先优化小文件访问。

小文件使用直接指针时：

```text
读 inode → 读数据块
```

如果小文件也用间接指针，就会变成：

```text
读 inode → 读指针块 → 读数据块
```

多一次磁盘访问，性能更差。

---

## 5. extent 和链表式分配

### 5.1 extent 是什么

extent 不记录每一个数据块地址，而是记录连续范围。

例如文件占用：

```text
100, 101, 102, 103, 104
```

如果用普通指针，要记录：

```text
100
101
102
103
104
```

如果用 extent，只需要记录：

```text
起始块号 = 100
长度 = 5
```

即：

```text
从 100 开始，连续 5 个块
```

优点：

```text
记录紧凑
顺序读取效率高
适合连续存放的大文件
```

缺点：

```text
依赖连续空闲空间
磁盘碎片多时，分配困难
文件增长时可能找不到紧邻空间
```

---

### 5.2 链表式分配

链表式分配的思想是：每个数据块记录下一个数据块的位置。

```text
数据块 A → 数据块 B → 数据块 C → 数据块 D
```

inode 只需要记录第一个块：

```text
inode
 └── first block pointer → 数据块 A
```

优点：

```text
inode 简单
文件容易增长
```

缺点：

```text
随机访问很差
```

例如要访问第 1000 个块，必须从第 1 个块一路顺着链表找：

```text
第 1 块 → 第 2 块 → ... → 第 1000 块
```

FAT 文件系统就是类似思想，只是它把“下一个块在哪里”的信息放在 FAT 表中。

## 6. 目录组织

### 6.1 目录本质上也是文件

目录不是特殊的“容器对象”，它本质上也是一种文件。

普通文件的数据块中存用户数据：

```text
hello world
```

目录文件的数据块中存目录项：

```text
文件名 → inode number
```

因此目录也是 inode 管理的对象。

目录 inode 中会记录：

```text
目录类型
目录大小
目录权限
目录数据块位置
时间信息
```

---

### 6.2 目录项是什么

目录项 directory entry 至少包含：

```text
文件名
inode number
```

例如目录 `dir` 中有三个文件：

```text
foo    → inode 12
bar    → inode 13
foobar → inode 24
```

目录数据块中可能存：

```text
inum | reclen | strlen | name
  5  |   4    |   2    | .
  2  |   4    |   3    | ..
 12  |   4    |   4    | foo
 13  |   4    |   4    | bar
 24  |   8    |   7    | foobar
```

---

### 6.3 目录项字段含义

#### inum

```text
inum = inode number
```

例如：

```text
foo → inode 12
bar → inode 13
```

#### name

```text
name = 文件名或目录名
```

#### strlen

```text
strlen = 文件名实际长度
```

因为文件名可以变长，所以需要记录长度。

#### reclen

```text
reclen = 当前目录项占用的总长度
```

因为目录项不是固定长度的，文件系统需要通过 `reclen` 知道下一个目录项从哪里开始。

可以理解为：

```text
当前目录项起点 + reclen = 下一个目录项起点
```

---

### 6.4 `.` 和 `..`

每个目录通常包含两个特殊目录项：

```text
.   当前目录
..  父目录
```

例如目录 `dir` 的 inode 是 5，它的父目录 inode 是 2：

```text
.  → inode 5
.. → inode 2
```

含义：

```text
dir/.  仍然是 dir 自己
dir/.. 是 dir 的父目录
```

---

### 6.5 访问 `dir/foo` 的过程

假设访问：

```text
dir/foo
```

文件系统大致执行：

```text
1. 找到目录 dir 的 inode
2. 根据 dir inode 找到 dir 的数据块
3. 扫描 dir 数据块中的目录项
4. 找到 name = foo 的目录项
5. 得到 foo 的 inode number
6. 去 inode 表中读取 foo 的 inode
7. 根据 foo inode 找到 foo 的数据块
8. 读取 foo 的文件内容
```

核心链路：

```text
目录名 dir
  ↓
dir inode
  ↓
dir 数据块
  ↓
foo → inode number
  ↓
foo inode
  ↓
foo 数据块
```

---

### 6.6 删除文件时目录中发生什么

删除一个文件，例如：

```c
unlink("bar");
```

目录中对应的目录项可能不会立刻被整体挪动删除，而是标记为空闲。

例如将：

```text
bar → inode 13
```

标记为无效：

```text
inum = 0
```

这样以后创建新文件时，可以复用这段目录空间。

---

### 6.7 线性目录和 B 树目录

VSFS 中目录是简单线性列表。

查找文件名时：

```text
从头扫描目录项
直到找到目标名字
```

优点：

```text
实现简单
```

缺点：

```text
目录很大时查找慢
```

例如一个目录有 10000 个文件，就可能扫描很多项。

因此一些文件系统使用 B 树组织目录，例如 XFS。

对比：

```text
线性列表：一个一个找
B 树：通过索引快速定位
```

---

## 7. 空闲空间管理

### 7.1 为什么需要空闲空间管理

文件系统经常需要分配和释放资源：

```text
创建文件：需要空闲 inode
写入文件：需要空闲数据块
删除文件：释放 inode 和数据块
创建目录：需要 inode，也可能需要数据块
```

所以必须记录：

```text
哪些 inode 空闲？
哪些数据块空闲？
```

---

### 7.2 VSFS 的方法：两个位图

VSFS 使用：

```text
inode bitmap：记录 inode 是否使用
data bitmap：记录数据块是否使用
```

状态：

```text
0：空闲
1：已使用
```

---

### 7.3 创建文件时发生什么

例如：

```bash
touch a.txt
```

文件系统需要：

```text
1. 扫描 inode bitmap，找空闲 inode
2. 将对应 bit 从 0 改成 1
3. 初始化 inode
4. 在父目录中增加目录项：a.txt → inode number
```

如果是空文件，暂时不需要数据块。

---

### 7.4 写入文件内容时发生什么

例如：

```bash
echo hello > a.txt
```

如果需要分配新数据块，文件系统需要：

```text
1. 扫描 data bitmap，找空闲数据块
2. 将对应 bit 从 0 改成 1
3. 把数据写入数据块
4. 把数据块地址写入 a.txt 的 inode
5. 更新文件大小和修改时间
```

---

### 7.5 删除文件时发生什么

例如：

```bash
rm a.txt
```

文件系统需要：

```text
1. 从父目录中删除 a.txt 的目录项
2. 释放 a.txt 使用的数据块
3. 将 data bitmap 中对应 bit 改回 0
4. 释放 inode
5. 将 inode bitmap 中对应 bit 改回 0
```

实际系统还会处理硬链接、引用计数等问题。

---

### 7.6 空闲列表 free list

除了位图，也可以使用空闲列表。

例如空闲块有：

```text
10, 18, 25, 31
```

可以组织成：

```text
10 → 18 → 25 → 31 → NULL
```

分配块时：

```text
取链表头部的空闲块
```

释放块时：

```text
把释放的块插回链表
```

缺点：

```text
不方便寻找连续空闲空间
```

---

### 7.7 为什么要寻找连续空闲块

连续数据块读写更快。

例如连续存放：

```text
100, 101, 102, 103, 104
```

比离散存放：

```text
100, 5000, 23, 881, 30000
```

更适合顺序读取。

机械磁盘上，连续块可以减少磁头移动；SSD 上，也有助于减少管理开销。

---

### 7.8 预分配 pre-allocation

预分配是指：

```text
文件系统不只分配当前马上需要的一个块，
而是提前分配多个连续块。
```

例如程序只写入 1 个块，但文件系统可能分配：

```text
100, 101, 102, 103, 104, 105, 106, 107
```

好处：

```text
减少碎片
提高顺序读写性能
减少频繁分配开销
```

---

## 8. 访问路径：读取文件

### 8.1 访问路径是什么

例如打开：

```text
/foo/bar
```

这是一条路径：

```text
/    → 根目录
foo  → 根目录下的目录
bar  → foo 目录下的文件
```

文件系统不能直接根据字符串 `bar` 找到文件，而是要逐级解析路径。

核心过程：

```text
路径名 → 目录遍历 → inode → 数据块
```

---

### 8.2 open("/foo/bar") 的过程

调用：

```c
open("/foo/bar", O_RDONLY);
```

文件系统需要找到 `bar` 的 inode。

过程如下：

```text
1. 读取 root inode
2. 读取 root data，在根目录中找到 foo 的 inode number
3. 读取 foo inode
4. 读取 foo data，在 foo 目录中找到 bar 的 inode number
5. 读取 bar inode
6. 建立打开文件表项
7. 返回文件描述符 fd
```

注意：

```text
open() 通常只是找到文件 inode，并建立内核中的打开文件结构；
它不一定读取文件内容。
```

---

### 8.3 文件描述符 fd 是什么

例如：

```c
int fd = open("/foo/bar", O_RDONLY);
```

返回的 `fd` 是进程使用的文件句柄。

它不是 inode number。

内核中会记录：

```text
fd → bar 的 inode
fd → 当前文件偏移量
fd → 打开模式
```

例如：

```text
fd = 3
文件 = /foo/bar
当前偏移量 = 0
模式 = 只读
```

后续调用：

```c
read(fd, buf, size);
```

就不需要重新解析路径 `/foo/bar`。

---

### 8.4 read() 的过程

假设文件 `bar` 大小是 12KB，块大小是 4KB，因此有三个数据块：

```text
bar data[0]
bar data[1]
bar data[2]
```

第一次调用：

```c
read(fd, buf, 4096);
```

过程：

```text
1. 根据 fd 找到 bar inode
2. 根据 offset = 0 知道要读 data[0]
3. 从 inode 中找到 data[0] 的磁盘地址
4. 读取 data[0]
5. 更新文件偏移量：0 → 4096
6. 可能更新 inode 的访问时间 atime
```

第二次：

```text
offset = 4096
读取 bar data[1]
offset = 8192
```

第三次：

```text
offset = 8192
读取 bar data[2]
offset = 12288
```

---

### 8.5 为什么 read() 可能会写 inode

因为很多文件系统会更新访问时间：

```text
atime = access time
```

所以即使用户只是读文件，文件系统也可能更新 inode：

```text
读 bar data[i]
写 bar inode，更新 atime
```

现代系统可能通过 `noatime`、`relatime` 等方式减少这种开销。

---

### 8.6 为什么读文件不会访问分配结构

读文件不需要找空闲 inode，也不需要找空闲数据块。

所以读文件通常不会访问：

```text
inode bitmap
data bitmap
```

读文件只需要：

```text
根据 inode 中已有的数据块指针读取数据
```

---

## 9. 访问路径：创建和写入文件

### 9.1 create("/foo/bar") 的表中顺序

表 40.4 表示“文件创建时间线”，时间向下递增。

创建：

```text
/foo/bar
```

表中顺序大致是：

```text
1. read root inode
2. read root data
3. read foo inode
4. read foo data
5. read inode bitmap
6. write foo data
7. write inode bitmap
8. read bar inode
9. write bar inode
10. write foo inode
```

注意：

> 由于表是向下时间递增，所以 `read inode bitmap` 之后，表中先出现的是 `write foo data`，然后才是 `write inode bitmap`。

---

### 9.2 create 阶段每一步解释

#### 1. read root inode

读取根目录 inode，获得根目录数据块位置。

#### 2. read root data

读取根目录内容，在根目录中找到 `foo` 的 inode number。

#### 3. read foo inode

读取 `/foo` 的 inode，确认它是目录，并获得它的数据块位置。

#### 4. read foo data

读取 `/foo` 目录数据，确认其中没有名为 `bar` 的目录项。

#### 5. read inode bitmap

寻找一个空闲 inode，准备分配给新文件 `bar`。

#### 6. write foo data

在 `/foo` 目录的数据块中加入目录项：

```text
bar → 新 inode number
```

#### 7. write inode bitmap

将刚才选中的 inode 标记为已使用：

```text
inode bitmap[新 inode number] = 1
```

#### 8. read bar inode

读取新 inode 所在的 inode 块。

因为磁盘通常按块读写，不是只读取单个 inode 字节。

#### 9. write bar inode

初始化新文件 inode：

```text
类型 = 普通文件
大小 = 0
权限 = 默认权限
所有者 = 当前用户
数据块指针 = 空
```

#### 10. write foo inode

因为 `/foo` 目录内容发生变化，所以需要更新 `/foo` 的 inode：

```text
mtime 改变
ctime 改变
目录大小可能改变
```

---

### 9.3 为什么 create 阶段不写 data bitmap

因为：

```text
create("/foo/bar")
```

只是创建一个空文件。

空文件需要 inode，但暂时不需要数据块。

因此 create 阶段需要操作：

```text
inode bitmap
bar inode
foo data
foo inode
```

但不需要操作：

```text
data bitmap
```

真正写入文件内容时才需要分配数据块。

---

### 9.4 write() 写入新块时的顺序

每次写入一个新的 4KB 数据块时，表中顺序大致是：

```text
1. read data bitmap
2. write data bitmap
3. read bar inode
4. write bar data[i]
5. write bar inode
```

含义：

```text
读 data bitmap：寻找空闲数据块
写 data bitmap：标记该数据块已使用
读 bar inode：查看文件当前状态和指针信息
写 bar data[i]：写入真正的用户数据
写 bar inode：更新文件大小、数据块指针、修改时间等
```

---

### 9.5 三次 write() 的例子

假设每次写入 4KB，连续写三次：

```c
write(fd, buf, 4096);
write(fd, buf, 4096);
write(fd, buf, 4096);
```

对应：

```text
第一次写：bar data[0]
第二次写：bar data[1]
第三次写：bar data[2]
```

每次都需要：

```text
read data bitmap
write data bitmap
read bar inode
write bar data[i]
write bar inode
```

因为每次都让文件增长一个新块，所以都要分配新数据块。

如果只是覆盖已经存在的数据块，则不一定需要访问 data bitmap。

---

### 9.6 create 和 write 的核心区别

create 主要修改元数据和目录：

```text
路径遍历
分配 inode
修改 inode bitmap
初始化 bar inode
修改 foo data
修改 foo inode
```

write 如果导致文件增长，则还要分配数据块：

```text
分配 data block
修改 data bitmap
写 bar data
更新 bar inode
```

---

## 10. read/write 中的 4096 参数

### 10.1 read/write 的参数含义

系统调用形式：

```c
ssize_t read(int fd, void *buf, size_t count);
ssize_t write(int fd, const void *buf, size_t count);
```

其中 `count` 表示：

```text
用户希望读/写多少字节
```

例如：

```c
read(fd, buf, 1);       // 读 1 字节
read(fd, buf, 100);     // 读 100 字节
read(fd, buf, 4096);    // 读 4096 字节
read(fd, buf, 10000);   // 读 10000 字节
```

因此：

```text
read/write 的 count 可以是任意字节数，
不必须等于磁盘扇区大小，也不必须等于文件系统块大小。
```

---

### 10.2 为什么教材常用 4096

因为教材假设：

```text
文件系统块大小 = 4KB = 4096B
```

所以一次：

```c
write(fd, buf, 4096);
```

刚好写一个数据块。

这样画表时很清楚：

```text
第一次 write → bar data[0]
第二次 write → bar data[1]
第三次 write → bar data[2]
```

---

### 10.3 用户层是字节级，底层是块级

用户调用：

```c
read(fd, buf, 100);
```

表示读 100 字节。

但底层可能实际读入一个完整块：

```text
磁盘读取 4KB 到页缓存
再从页缓存复制 100B 给用户
```

写也是类似：

```c
write(fd, buf, 100);
```

可能内部过程：

```text
把对应 4KB 页读入缓存
修改其中 100B
将该页标记为脏页
以后再写回磁盘
```

所以：

```text
用户接口是字节级的
文件系统和磁盘底层通常是块级的
```

---

### 10.4 direct I/O 的特殊情况

普通 read/write 可以用任意字节数。

但如果使用 direct I/O，通常有对齐要求：

```text
缓冲区地址要按 512B 或 4KB 对齐
读写大小要按 512B 或 4KB 对齐
文件偏移量也要对齐
```

这是 direct I/O 的特殊限制，不是普通 read/write 的限制。

---

## 11. 缓存和缓冲

### 11.1 为什么需要缓存

磁盘很慢，内存很快。

如果每次打开文件都真的访问磁盘，会非常慢。

例如打开：

```text
/foo/bar
```

可能需要：

```text
读 root inode
读 root data
读 foo inode
读 foo data
读 bar inode
```

如果路径很长：

```text
/1/2/3/.../100/file.txt
```

会产生大量 I/O。

所以文件系统会把常用的 inode、目录块、文件数据块缓存在内存中。

---

### 11.2 cache 的基本思想

第一次访问：

```text
磁盘 → 内存缓存 → 用户程序
```

第二次访问同一内容：

```text
内存缓存 → 用户程序
```

如果缓存命中，就不用访问磁盘。

---

### 11.3 固定大小缓存的问题

早期文件系统可能固定分配一部分内存做文件系统缓存，例如总内存 10%。

这种方式叫：

```text
static partitioning
静态划分
```

问题：

```text
如果文件系统不需要这么多缓存，内存浪费
如果文件系统需要更多缓存，又不够用
```

---

### 11.4 统一页缓存 unified page cache

现代操作系统通常使用统一页缓存：

```text
unified page cache
```

意思是：

```text
文件系统缓存页和普通虚拟内存页统一管理
```

也就是说，内存不再固定分成：

```text
程序内存区
文件缓存区
```

而是统一成：

```text
总内存
├── 程序正在使用的页
├── 文件缓存页
├── 空闲页
└── 内核其他页
```

谁需要更多，就动态分配更多。

这叫：

```text
dynamic partitioning
动态划分
```

---

### 11.5 统一页缓存的例子

第一次读文件：

```c
read(fd, buf, 4096);
```

如果缓存没有该文件页：

```text
磁盘 → 页缓存 → 用户 buf
```

第二次读同一位置：

```text
页缓存命中 → 用户 buf
```

如果系统内存紧张，操作系统可以回收文件缓存页：

```text
这个文件页可以从磁盘重新读回来，所以可以先丢弃
```

---

### 11.6 写缓冲 write buffering

写入和读取不同。

读取可以直接从缓存拿；但写入最终必须落盘，否则断电后数据会丢。

普通写入可能先写入内存缓存：

```text
write() → 内存页缓存
```

该页被标记为：

```text
dirty page
脏页
```

意思是：

```text
内存中的数据已经修改，但还没有同步到磁盘
```

之后系统再把脏页写回磁盘。

---

### 11.7 写缓冲的好处

#### 1. 合并多次写入

如果同一个 inode 被修改多次：

```text
修改 1
修改 2
修改 3
```

不必每次都写磁盘，可以最终只写一次。

#### 2. 批量写入

系统可以把多个写操作攒起来：

```text
写 inode bitmap
写 data bitmap
写 inode
写 data block
```

统一安排，提高效率。

#### 3. 调度写入顺序

系统可以调整磁盘写入顺序，减少寻道或提高设备吞吐。

#### 4. 避免无意义写入

如果应用程序创建一个文件后马上删除：

```text
create file
delete file
```

如果创建操作还没写到磁盘，系统可以发现最终不需要它，于是完全避免写入。

---

### 11.8 写缓冲的风险

风险是：

```text
数据还在内存中，尚未写到磁盘。
如果系统崩溃或断电，数据可能丢失。
```

所以：

```text
write() 返回成功，不一定代表数据已经真正落盘。
```

它可能只是写到了内存缓存。

---

### 11.9 fsync()

如果应用程序希望数据一定落盘，可以调用：

```c
fsync(fd);
```

作用：

```text
强制将该文件相关的脏数据写入磁盘
```

区别：

```text
write()：可能只是写入内存
fsync()：要求真正同步到磁盘
```

数据库等关键应用经常使用 `fsync()`。

---

### 11.10 direct I/O

普通 I/O 路径：

```text
应用程序 → 文件系统页缓存 → 磁盘
```

direct I/O 尽量绕过页缓存：

```text
应用程序 → 磁盘
```

数据库等系统可能自己实现缓存，不希望操作系统再缓存一份，否则会浪费内存。

---

### 11.11 raw disk

raw disk 是指绕过普通文件系统，直接访问裸磁盘设备。

优点：

```text
应用程序可以完全控制磁盘布局和写入策略
```

缺点：

```text
使用复杂
失去普通文件系统的便利
```

---

## 12. 性能和可靠性的权衡

文件系统设计中经常存在权衡：

```text
越追求性能 → 越可能延迟写入 → 崩溃时风险更高
越追求可靠性 → 越需要立即落盘 → 性能更差
```

例如：

```text
银行转账：必须安全，应尽快落盘
浏览器缓存图片：丢了问题不大，可以延迟写
```

所以不同应用有不同策略。

## 13. 核心流程总结

### 13.1 文件系统整体查找链路

```text
路径名
  ↓
目录遍历
  ↓
目录项：文件名 → inode number
  ↓
inode
  ↓
数据块指针
  ↓
文件内容
```

---

### 13.2 创建文件

```text
create("/foo/bar"):

1. 解析路径，找到父目录 /foo
2. 确认 /foo/bar 不存在
3. 读取 inode bitmap，找到空闲 inode
4. 修改父目录数据块，加入 bar → inode number
5. 写 inode bitmap，标记 inode 已用
6. 初始化 bar inode
7. 更新 foo inode
```

---

### 13.3 写入新数据块

```text
write():

1. 读取 data bitmap，找空闲数据块
2. 写 data bitmap，标记数据块已用
3. 读取 bar inode
4. 写入 bar data[i]
5. 写 bar inode，更新文件大小、指针和时间
```

---

### 13.4 读取文件

```text
read():

1. 根据 fd 找到打开文件表项
2. 找到 inode
3. 根据 offset 找到对应数据块
4. 读取数据块
5. 更新 offset
6. 可能更新 atime
```

---

## 14. 易错点

### 易错点 1：inode 不存文件名

文件名存在目录项中。

```text
目录项：文件名 → inode number
inode：文件元数据 + 数据块指针
```

---

### 易错点 2：目录也是文件

目录也有 inode。

区别是：

```text
普通文件的数据块存用户数据
目录的数据块存目录项
```

---

### 易错点 3：位图的 1 bit 表示 1 个对象

不是 1 bit 表示 1 字节，而是：

```text
1 bit → 1 个 inode 或 1 个数据块的分配状态
```

---

### 易错点 4：create 空文件不一定分配数据块

创建空文件时：

```text
需要 inode
需要目录项
不一定需要数据块
```

写入内容时才分配数据块。

---

### 易错点 5：read/write 的 4096 不是强制要求

`4096` 只是教材中为了对应 4KB 文件系统块。

普通 `read/write` 的 `count` 可以是任意字节数。

---

### 易错点 6：read 也可能导致 write

因为读文件可能更新访问时间 atime，所以可能写 inode。

---

### 易错点 7：write() 返回不一定代表真正落盘

普通写入可能只是进入页缓存。

若要强制落盘，需要：

```c
fsync(fd);
```

---

### 易错点 8：表 40.4 要按向下时间顺序读

例如 create 部分：

```text
read inode bitmap 之后，
表中先出现 write foo data，
再出现 write inode bitmap。
```

不要只按“逻辑模块”重排顺序。

---

## 15. 总结

文件系统的核心任务是把磁盘组织成可管理的数据结构。  
VSFS 通过 superblock、bitmap、inode table 和 data region 来完成文件管理。

最重要的链路是：

```text
文件名 → 目录项 → inode number → inode → 数据块 → 文件内容
```

其中：

```text
目录负责名字到 inode number 的映射
inode 负责记录文件元数据和数据块位置
bitmap 负责记录 inode 和数据块是否空闲
data region 负责存放真正的数据
```

---

## 16. 简短记忆版

```text
文件系统 = 管理 inode + 数据块 + 目录项 + 空闲空间。

文件名不直接指向数据块，而是：
文件名 → inode number → inode → 数据块。

目录也是文件，只是它的数据块存：
文件名 → inode number。

inode 存：
文件类型、大小、权限、时间、数据块指针。

小文件用直接指针，大文件用间接、双重间接、三重间接指针。

bitmap 用 0/1 记录 inode 或数据块是否被分配。

create 空文件：
找父目录 → 分配 inode → 写目录项 → 初始化 inode。

write 新块：
找空闲 data block → 更新 data bitmap → 写数据块 → 更新 inode。

read 文件：
根据 fd 找 inode → 根据 offset 找数据块 → 读取数据。

缓存减少读 I/O，写缓冲合并写 I/O。
write 不一定立刻落盘，fsync 才强制落盘。
```
