# 第 31 章：信号量相关经典问题整理

> 主题：生产者/消费者问题、读者-写者锁、哲学家就餐问题、用锁和条件变量实现信号量  
> 核心关键词：信号量、互斥、同步、死锁、饥饿、条件变量

---

## 1. 生产者/消费者问题：有界缓冲区

### 1.1 问题背景

生产者/消费者问题中，有一个固定大小的缓冲区。

- 生产者向缓冲区中放入数据；
- 消费者从缓冲区中取出数据；
- 缓冲区满时，生产者不能继续放；
- 缓冲区空时，消费者不能继续取；
- 多个生产者和多个消费者并发执行时，还要避免竞态条件。

---

## 2. `put()` 和 `get()` 函数

```c
int buffer[MAX];
int fill = 0;
int use = 0;

void put(int value) {
    buffer[fill] = value;          // line f1
    fill = (fill + 1) % MAX;       // line f2
}

int get() {
    int tmp = buffer[use];         // line g1
    use = (use + 1) % MAX;         // line g2
    return tmp;
}
```

### 2.1 变量含义

```c
int buffer[MAX];
```

表示大小为 `MAX` 的共享缓冲区。

```c
int fill = 0;
```

表示生产者下一次要写入的位置。

```c
int use = 0;
```

表示消费者下一次要读取的位置。

---

### 2.2 `put()` 函数解释

```c
buffer[fill] = value;
```

把生产出来的数据写入 `buffer[fill]` 位置。

```c
fill = (fill + 1) % MAX;
```

更新下一次写入的位置。

这里使用模运算 `% MAX`，是为了把数组当成循环队列。

例如 `MAX = 10` 时：

```text
fill: 0 → 1 → 2 → ... → 9 → 0 → 1 ...
```

---

### 2.3 `get()` 函数解释

```c
int tmp = buffer[use];
```

从 `buffer[use]` 位置取出数据。

```c
use = (use + 1) % MAX;
```

更新下一次读取的位置。

```c
return tmp;
```

返回取出的数据。

---

## 3. 第一次尝试：只使用 `empty` 和 `full`

```c
sem_t empty;
sem_t full;

void *producer(void *arg) {
    int i;
    for (i = 0; i < loops; i++) {
        sem_wait(&empty);      // line P1
        put(i);                // line P2
        sem_post(&full);       // line P3
    }
}

void *consumer(void *arg) {
    int i, tmp = 0;
    while (tmp != -1) {
        sem_wait(&full);       // line C1
        tmp = get();           // line C2
        sem_post(&empty);      // line C3
        printf("%d\n", tmp);
    }
}

int main(int argc, char *argv[]) {
    sem_init(&empty, 0, MAX);
    sem_init(&full, 0, 0);
}
```

---

### 3.1 `empty` 和 `full` 的含义

```c
sem_init(&empty, 0, MAX);
```

表示一开始有 `MAX` 个空槽位。

```c
sem_init(&full, 0, 0);
```

表示一开始有 `0` 个已经填满的槽位。

因此：

```text
empty：当前还有多少空位可以放数据
full ：当前有多少数据可以取
```

---

### 3.2 生产者逻辑

```c
sem_wait(&empty);
```

申请一个空槽位。如果没有空位，则阻塞。

```c
put(i);
```

向缓冲区放入数据。

```c
sem_post(&full);
```

通知消费者：现在多了一个可消费的数据。

---

### 3.3 消费者逻辑

```c
sem_wait(&full);
```

申请一个已有数据的槽位。如果没有数据，则阻塞。

```c
tmp = get();
```

从缓冲区取出一个数据。

```c
sem_post(&empty);
```

通知生产者：现在多了一个空槽位。

---

## 4. 为什么 `MAX = 1` 时看起来没问题？

当 `MAX = 1` 时，缓冲区只有一个位置。

一开始：

```text
empty = 1
full = 0
```

生产者通过 `sem_wait(&empty)` 后：

```text
empty = 0
```

这时其他生产者不能再进入 `put()`。

消费者通过 `sem_wait(&full)` 后，其他消费者也不能再进入 `get()`。

所以在 `MAX = 1` 时，虽然没有显式互斥，但问题不容易暴露。

---

## 5. 为什么 `MAX > 1` 时会出问题？

当 `MAX = 10` 时，一开始：

```text
empty = 10
```

多个生产者都可能成功执行：

```c
sem_wait(&empty);
```

然后同时进入：

```c
put(i);
```

这就会产生竞态条件。

---

## 6. 生产者之间的竞态条件

假设一开始：

```text
fill = 0
```

两个生产者 `Pa` 和 `Pb` 同时调用 `put()`。

### 可能的执行顺序

`Pa` 执行：

```c
buffer[fill] = value;
```

此时相当于：

```c
buffer[0] = Pa_value;
```

但是 `Pa` 还没执行：

```c
fill = (fill + 1) % MAX;
```

也就是说：

```text
fill 仍然等于 0
```

这时 `Pa` 被中断，`Pb` 开始运行。

`Pb` 也看到：

```text
fill = 0
```

于是执行：

```c
buffer[0] = Pb_value;
```

结果：

```text
Pb_value 覆盖了 Pa_value
```

这就是生产者之间的竞态条件。

---

## 7. 消费者之间也有类似问题

`get()` 函数如下：

```c
int get() {
    int tmp = buffer[use];
    use = (use + 1) % MAX;
    return tmp;
}
```

假设：

```text
use = 0
```

两个消费者 `Ca` 和 `Cb` 同时调用 `get()`。

可能发生：

```text
Ca 读取 buffer[0]
Ca 还没更新 use
Cb 也读取 buffer[0]
```

结果是：

```text
同一个数据被消费了两次
```

---

## 8. `int i, tmp = 0;` 的含义

```c
int i, tmp = 0;
```

等价于：

```c
int i;
int tmp = 0;
```

其中：

- `i` 通常作为循环变量；
- `tmp` 用来保存消费者从缓冲区取出的值。

在消费者代码中：

```c
tmp = get();
```

表示从缓冲区中取出一个值，并保存到 `tmp`。

```c
while (tmp != -1)
```

通常表示如果取到 `-1`，就认为结束。

这里的 `-1` 可以理解为特殊结束标志，也叫 sentinel value。

# 9. 增加互斥量的错误版本

为了保护 `put()` 和 `get()`，一种自然想法是增加一个互斥信号量 `mutex`。

```c
sem_t empty;
sem_t full;
sem_t mutex;

void *producer(void *arg) {
    int i;
    for (i = 0; i < loops; i++) {
        sem_wait(&mutex);      // line p0
        sem_wait(&empty);      // line p1
        put(i);                // line p2
        sem_post(&full);       // line p3
        sem_post(&mutex);      // line p4
    }
}

void *consumer(void *arg) {
    int i;
    for (i = 0; i < loops; i++) {
        sem_wait(&mutex);      // line c0
        sem_wait(&full);       // line c1
        int tmp = get();       // line c2
        sem_post(&empty);      // line c3
        sem_post(&mutex);      // line c4
        printf("%d\n", tmp);
    }
}

int main(int argc, char *argv[]) {
    sem_init(&empty, 0, MAX);
    sem_init(&full, 0, 0);
    sem_init(&mutex, 0, 1);
}
```

---

## 10. 为什么这个版本会死锁？

问题出在：

```c
sem_wait(&mutex);
sem_wait(&full);
```

或者：

```c
sem_wait(&mutex);
sem_wait(&empty);
```

也就是：

```text
先拿互斥锁，再等待条件。
```

---

### 10.1 死锁场景

假设缓冲区为空：

```text
empty = 1
full = 0
mutex = 1
```

消费者先运行：

```c
sem_wait(&mutex);
```

消费者成功拿到 `mutex`。

然后执行：

```c
sem_wait(&full);
```

因为 `full = 0`，说明没有数据，所以消费者阻塞。

关键是：

```text
消费者阻塞时仍然持有 mutex
```

接下来生产者运行：

```c
sem_wait(&mutex);
```

但是 `mutex` 已经被消费者持有，所以生产者也阻塞。

此时状态是：

```text
消费者：持有 mutex，等待 full
生产者：等待 mutex，无法生产数据，也无法 sem_post(&full)
```

于是形成死锁。

---

## 11. 正确版本：先等条件，再拿互斥锁

正确顺序是：

### 生产者

```c
sem_wait(&empty);
sem_wait(&mutex);

put(i);

sem_post(&mutex);
sem_post(&full);
```

### 消费者

```c
sem_wait(&full);
sem_wait(&mutex);

tmp = get();

sem_post(&mutex);
sem_post(&empty);
```

完整结构如下：

```c
void *producer(void *arg) {
    int i;
    for (i = 0; i < loops; i++) {
        sem_wait(&empty);
        sem_wait(&mutex);

        put(i);

        sem_post(&mutex);
        sem_post(&full);
    }
}

void *consumer(void *arg) {
    int i;
    for (i = 0; i < loops; i++) {
        sem_wait(&full);
        sem_wait(&mutex);

        int tmp = get();

        sem_post(&mutex);
        sem_post(&empty);

        printf("%d\n", tmp);
    }
}
```

---

## 12. 生产者/消费者问题的核心记忆

```text
empty/full：负责同步条件
mutex     ：负责互斥访问
```

更具体地说：

```text
empty：有没有空位可以放？
full ：有没有数据可以取？
mutex：同一时刻只能一个线程修改 buffer、fill、use。
```

# 13. 读者-写者锁

## 13.1 为什么需要读者-写者锁？

普通互斥锁不区分读和写。

但是很多数据结构中：

- 查找操作只读，不修改结构；
- 插入、删除操作会修改结构。

多个只读操作可以并发执行，而写操作必须独占。

读者-写者锁的目标是：

```text
读-读：可以并发
读-写：必须互斥
写-写：必须互斥
```

---

## 13.2 读者-写者锁代码

```c
typedef struct _rwlock_t {
    sem_t lock;        // binary semaphore
    sem_t writelock;   // allow ONE writer or MANY readers
    int readers;       // number of readers
} rwlock_t;

void rwlock_init(rwlock_t *rw) {
    rw->readers = 0;
    sem_init(&rw->lock, 0, 1);
    sem_init(&rw->writelock, 0, 1);
}

void rwlock_acquire_readlock(rwlock_t *rw) {
    sem_wait(&rw->lock);
    rw->readers++;
    if (rw->readers == 1)
        sem_wait(&rw->writelock);
    sem_post(&rw->lock);
}

void rwlock_release_readlock(rwlock_t *rw) {
    sem_wait(&rw->lock);
    rw->readers--;
    if (rw->readers == 0)
        sem_post(&rw->writelock);
    sem_post(&rw->lock);
}

void rwlock_acquire_writelock(rwlock_t *rw) {
    sem_wait(&rw->writelock);
}

void rwlock_release_writelock(rwlock_t *rw) {
    sem_post(&rw->writelock);
}
```

---

## 13.3 结构体成员解释

```c
sem_t lock;
```

保护 `readers` 变量。

因为多个读者可能同时修改 `readers`，所以 `readers++` 和 `readers--` 必须被互斥保护。

```c
sem_t writelock;
```

控制真正的数据结构访问。

它保证：

```text
要么一个写者进入；
要么多个读者进入；
不能读写同时进入。
```

```c
int readers;
```

记录当前有多少个读者正在读。

---

## 13.4 为什么需要 `readers`？

`readers` 的作用是判断：

```text
谁是第一个读者？
谁是最后一个读者？
```

第一个读者负责拿走 `writelock`，阻止写者进入。

最后一个读者负责释放 `writelock`，允许写者进入。

---

## 13.5 获取读锁过程

```c
void rwlock_acquire_readlock(rwlock_t *rw) {
    sem_wait(&rw->lock);
    rw->readers++;
    if (rw->readers == 1)
        sem_wait(&rw->writelock);
    sem_post(&rw->lock);
}
```

逐行解释：

```c
sem_wait(&rw->lock);
```

先锁住 `readers` 计数器。

```c
rw->readers++;
```

当前读者数量加一。

```c
if (rw->readers == 1)
```

如果当前读者数量变成 1，说明我是第一个读者。

```c
sem_wait(&rw->writelock);
```

第一个读者拿走 `writelock`，阻止写者进入。

```c
sem_post(&rw->lock);
```

释放对 `readers` 计数器的保护。

---

## 13.6 释放读锁过程

```c
void rwlock_release_readlock(rwlock_t *rw) {
    sem_wait(&rw->lock);
    rw->readers--;
    if (rw->readers == 0)
        sem_post(&rw->writelock);
    sem_post(&rw->lock);
}
```

逐行解释：

```c
sem_wait(&rw->lock);
```

锁住 `readers` 计数器。

```c
rw->readers--;
```

当前读者数量减一。

```c
if (rw->readers == 0)
```

如果读者数量变成 0，说明我是最后一个离开的读者。

```c
sem_post(&rw->writelock);
```

最后一个读者释放 `writelock`，允许写者进入。

```c
sem_post(&rw->lock);
```

释放 `readers` 计数器的保护。

---

## 13.7 写锁过程

```c
void rwlock_acquire_writelock(rwlock_t *rw) {
    sem_wait(&rw->writelock);
}

void rwlock_release_writelock(rwlock_t *rw) {
    sem_post(&rw->writelock);
}
```

写者直接申请 `writelock`。

如果当前没有读者，也没有写者，则写者成功进入。

如果当前有读者或者其他写者，则写者阻塞。

---

# 14. 读写锁的实际使用例子

假设有一个共享链表：

```c
typedef struct node {
    int key;
    struct node *next;
} node_t;

node_t *head;
rwlock_t rwlock;
```

---

## 14.1 查找操作使用读锁

```c
int list_search(int key) {
    rwlock_acquire_readlock(&rwlock);

    node_t *curr = head;
    while (curr != NULL) {
        if (curr->key == key) {
            rwlock_release_readlock(&rwlock);
            return 1;
        }
        curr = curr->next;
    }

    rwlock_release_readlock(&rwlock);
    return 0;
}
```

查找只读链表，不修改结构，所以可以和其他查找操作并发执行。

---

## 14.2 插入操作使用写锁

```c
void list_insert(int key) {
    rwlock_acquire_writelock(&rwlock);

    node_t *new_node = malloc(sizeof(node_t));
    new_node->key = key;
    new_node->next = head;
    head = new_node;

    rwlock_release_writelock(&rwlock);
}
```

插入会修改链表指针，所以必须独占访问。

---

## 14.3 读写锁的核心类比

可以把读写锁想象成图书馆：

```text
读者 = 看书的人
写者 = 整理书架的人
```

规则是：

```text
很多人可以同时看书；
整理书架时不能有人看书；
有人看书时不能整理书架。
```

`readers` 就像门口的计数器，用来记录当前有多少人在看书。

# 15. 读者-写者锁的公平性问题

上面的实现是读者优先的。

如果不断有新的读者进入，写者可能一直等不到机会。

---

## 15.1 写者饥饿示例

假设当前有读者正在读：

```text
readers = 3
writelock = 0
```

写者 `W1` 想写：

```c
sem_wait(&rw->writelock);
```

因为 `writelock = 0`，所以 `W1` 阻塞。

但是新的读者 `R4`、`R5`、`R6` 仍然可以进入。

只要不断有新读者加入，`readers` 就可能一直大于 0，写者就一直等不到 `writelock`。

这就是写者饥饿。

---

## 15.2 饥饿和死锁的区别

```text
死锁：所有相关线程互相等待，系统不再前进。
饥饿：系统仍然运行，但某些线程长期得不到机会。
```

读者-写者锁中的问题是饥饿，不是死锁。

---

## 15.3 避免写者饥饿的思路

核心思想：

```text
一旦有写者等待，就阻止新的读者进入。
让已有读者读完后，优先让写者进入。
```

可以引入一个入口闸门 `turnstile`。

---

### 简化伪代码

```c
sem_t turnstile;
sem_t writelock;
sem_t lock;
int readers;
```

读者：

```c
sem_wait(&turnstile);
sem_post(&turnstile);

sem_wait(&lock);
readers++;
if (readers == 1)
    sem_wait(&writelock);
sem_post(&lock);

// read data

sem_wait(&lock);
readers--;
if (readers == 0)
    sem_post(&writelock);
sem_post(&lock);
```

写者：

```c
sem_wait(&turnstile);
sem_wait(&writelock);

// write data

sem_post(&writelock);
sem_post(&turnstile);
```

`turnstile` 的作用是：

```text
写者来了之后，先关住入口；
后来的读者不能继续插队；
已有读者读完后，写者进入。
```

---

## 15.4 读写锁不一定更快

读写锁虽然允许多个读者并发，但它有额外开销：

```text
维护 readers 计数；
保护 readers 的锁；
判断第一个读者和最后一个读者；
复杂实现还要维护等待写者和公平性。
```

因此读写锁适合：

```text
读操作很多；
写操作较少；
读操作比较耗时；
读者并发能带来明显收益。
```

不适合：

```text
临界区很短；
写操作很多；
竞争不激烈；
普通互斥锁已经足够快。
```

# 16. 哲学家就餐问题

## 16.1 问题模型

哲学家就餐问题中：

```text
5 个哲学家围坐在圆桌旁；
相邻哲学家之间有一把餐叉；
一共有 5 把餐叉；
哲学家需要同时拿到左右两把餐叉才能吃饭。
```

配图占位：

```markdown
![哲学家就餐问题](images/dining-philosophers.png)
```

---

## 16.2 并发模型对应关系

```text
哲学家 = 线程
餐叉   = 共享资源
forks  = 全局共享信号量数组
```

通常可以定义：

```c
sem_t forks[5];
```

每一把叉子是一个二值信号量。

初始化：

```c
for (int i = 0; i < 5; i++) {
    sem_init(&forks[i], 0, 1);
}
```

每个信号量初始值为 1，表示这把叉子一开始可用。

---

## 16.3 每个哲学家的基本循环

```c
while (1) {
    think();
    getforks();
    eat();
    putforks();
}
```

含义是：

```text
思考 → 拿叉子 → 吃饭 → 放叉子
```

---

## 16.4 左右叉子的编号

```c
int left(int p) {
    return p;
}

int right(int p) {
    return (p + 1) % 5;
}
```

对于哲学家 `p`：

```text
左边叉子编号 = p
右边叉子编号 = (p + 1) % 5
```

对应关系：

| 哲学家 | 左手叉子 | 右手叉子 |
|---|---:|---:|
| 0 | forks[0] | forks[1] |
| 1 | forks[1] | forks[2] |
| 2 | forks[2] | forks[3] |
| 3 | forks[3] | forks[4] |
| 4 | forks[4] | forks[0] |

其中：

```c
right(4) = (4 + 1) % 5 = 0
```

所以哲学家 4 的右手边叉子是 `forks[0]`。

---

# 17. 有问题的哲学家就餐方案

```c
void getforks() {
    sem_wait(&forks[left(p)]);
    sem_wait(&forks[right(p)]);
}

void putforks() {
    sem_post(&forks[left(p)]);
    sem_post(&forks[right(p)]);
}
```

这个方案中，所有哲学家都：

```text
先拿左边叉子，再拿右边叉子。
```

---

## 17.1 为什么会死锁？

假设 5 个哲学家几乎同时运行。

第一步，每个人都拿到了左边叉子：

```text
P0 拿到 F0
P1 拿到 F1
P2 拿到 F2
P3 拿到 F3
P4 拿到 F4
```

此时所有叉子都被占用。

第二步，每个人都想拿右边叉子：

```text
P0 等 F1，但 F1 在 P1 手里
P1 等 F2，但 F2 在 P2 手里
P2 等 F3，但 F3 在 P3 手里
P3 等 F4，但 F4 在 P4 手里
P4 等 F0，但 F0 在 P0 手里
```

于是形成等待环：

```text
P0 → P1 → P2 → P3 → P4 → P0
```

这就是死锁。

---

# 18. 破除依赖：改变一个哲学家的拿叉顺序

一种简单解决方法是让哲学家 4 反过来拿叉子。

```c
void getforks() {
    if (p == 4) {
        sem_wait(&forks[right(p)]);
        sem_wait(&forks[left(p)]);
    } else {
        sem_wait(&forks[left(p)]);
        sem_wait(&forks[right(p)]);
    }
}
```

含义是：

```text
哲学家 0、1、2、3：先拿左边，再拿右边；
哲学家 4：先拿右边，再拿左边。
```

---

## 18.1 为什么这样能避免死锁？

原始方案中，每个人都按同一方向拿叉子，容易形成循环等待。

改变哲学家 4 的顺序后，等待环被破坏。

例如：

```text
P0 拿 F0
P1 拿 F1
P2 拿 F2
P3 拿 F3
```

此时 `P4` 想先拿右边叉子，也就是：

```text
right(4) = F0
```

但 `F0` 已经被 `P0` 拿走，所以 `P4` 不能拿到任何叉子。

这时 `F4` 仍然是空的，所以 `P3` 可以拿到 `F4` 并吃饭。

因此系统不会出现所有哲学家都拿着一把叉子并等待另一把叉子的局面。

---

## 18.2 核心思想

这个方案的本质是：

```text
破坏循环等待条件。
```

只要不能形成完整等待环，就不会发生这种死锁。

---

# 19. 用锁和条件变量实现信号量：Zemaphore

## 19.1 问题背景

前面一直使用系统提供的信号量：

```c
sem_wait(&s);
sem_post(&s);
```

现在反过来问：

```text
如果系统只提供互斥锁和条件变量，能不能自己实现一个信号量？
```

答案是可以。

这个自定义信号量叫 `Zemaphore`。

---

## 19.2 Zemaphore 结构体

```c
typedef struct _Zem_t {
    int value;
    pthread_cond_t cond;
    pthread_mutex_t lock;
} Zem_t;
```

---

### 成员解释

```c
int value;
```

表示当前可用资源数量。

```c
pthread_cond_t cond;
```

条件变量。用于在资源不可用时让线程睡眠，在资源变可用时唤醒线程。

```c
pthread_mutex_t lock;
```

互斥锁。用于保护 `value`。

---

## 19.3 初始化函数

```c
void Zem_init(Zem_t *s, int value) {
    s->value = value;
    Cond_init(&s->cond);
    Mutex_init(&s->lock);
}
```

逐行解释：

```c
s->value = value;
```

设置信号量初始值。

```c
Cond_init(&s->cond);
```

初始化条件变量。

```c
Mutex_init(&s->lock);
```

初始化互斥锁。

---

## 19.4 `Zem_wait()` 函数

```c
void Zem_wait(Zem_t *s) {
    Mutex_lock(&s->lock);
    while (s->value <= 0)
        Cond_wait(&s->cond, &s->lock);
    s->value--;
    Mutex_unlock(&s->lock);
}
```

它对应普通信号量的：

```c
sem_wait(&s);
```

作用是：

```text
如果资源可用，就拿走一个资源；
如果资源不可用，就睡眠等待。
```

---

### 逐行解释

```c
Mutex_lock(&s->lock);
```

先加锁，因为接下来要检查和修改 `s->value`。

```c
while (s->value <= 0)
```

如果当前资源数量小于等于 0，说明没有资源可用。

```c
Cond_wait(&s->cond, &s->lock);
```

让当前线程在条件变量上睡眠。

注意：`Cond_wait()` 会原子地完成三件事：

```text
释放 lock；
让当前线程睡眠；
被唤醒后重新获得 lock。
```

```c
s->value--;
```

当资源可用时，拿走一个资源。

```c
Mutex_unlock(&s->lock);
```

释放锁。

---

## 19.5 为什么这里要用 `while` 而不是 `if`？

正确写法：

```c
while (s->value <= 0)
    Cond_wait(&s->cond, &s->lock);
```

不推荐：

```c
if (s->value <= 0)
    Cond_wait(&s->cond, &s->lock);
```

原因是：

```text
线程被唤醒不代表条件一定满足。
```

可能出现：

```text
虚假唤醒；
多个线程被唤醒，但资源只有一个；
被唤醒后还没执行，资源被其他线程抢走。
```

所以被唤醒后必须重新检查条件。

---

## 19.6 `Zem_post()` 函数

```c
void Zem_post(Zem_t *s) {
    Mutex_lock(&s->lock);
    s->value++;
    Cond_signal(&s->cond);
    Mutex_unlock(&s->lock);
}
```

它对应普通信号量的：

```c
sem_post(&s);
```

作用是：

```text
增加一个可用资源；
唤醒一个等待线程。
```

---

### 逐行解释

```c
Mutex_lock(&s->lock);
```

加锁，保护 `s->value`。

```c
s->value++;
```

增加一个资源。

```c
Cond_signal(&s->cond);
```

唤醒一个正在等待的线程。

```c
Mutex_unlock(&s->lock);
```

释放锁。

---

## 19.7 Zemaphore 的运行例子

假设：

```c
Zem_t z;
Zem_init(&z, 1);
```

初始状态：

```text
z.value = 1
```

线程 A 执行：

```c
Zem_wait(&z);
```

因为 `value = 1`，所以 A 不睡眠，直接执行：

```text
value: 1 → 0
```

线程 B 再执行：

```c
Zem_wait(&z);
```

此时：

```text
value = 0
```

所以 B 睡眠。

之后 A 执行：

```c
Zem_post(&z);
```

使得：

```text
value: 0 → 1
```

并唤醒 B。

B 被唤醒后重新检查 `value`，发现资源可用，于是：

```text
value: 1 → 0
```

B 成功获得资源。

---

# 20. Zemaphore 和 Dijkstra 信号量的区别

Dijkstra 定义的信号量中，值可以为负数。

例如：

```text
value = -3
```

可能表示有 3 个线程正在等待。

但是 Zemaphore 的实现中：

```c
while (s->value <= 0)
    Cond_wait(&s->cond, &s->lock);

s->value--;
```

只有当 `value > 0` 时，才会执行 `value--`。

因此：

```text
Zemaphore 的 value 永远不会小于 0。
```

等待线程数量由条件变量内部的等待队列管理，而不是通过负数表示。

---

# 21. 为什么用锁和条件变量实现信号量比较自然？

因为信号量本质上需要三个东西：

```text
计数器：记录资源数量；
互斥锁：保护计数器；
等待队列：资源不足时让线程睡眠。
```

而这正好对应：

```text
value：计数器；
lock ：保护计数器；
cond ：等待队列和唤醒机制。
```

所以：

```text
Zemaphore = mutex + condition variable + counter
```

---

# 22. 为什么用信号量实现条件变量更难？

条件变量有一个特殊语义：

```c
pthread_cond_wait(&cond, &mutex);
```

它必须做到：

```text
原子地释放 mutex，并让线程睡眠。
```

这两个动作中间不能被打断。

---

## 22.1 丢失唤醒问题

假设条件变量的等待不是原子的，而是拆成：

```text
释放 mutex；
进入睡眠。
```

可能发生：

```text
消费者发现 count == 0；
消费者准备睡眠；
消费者刚释放 mutex，但还没有真正睡着；
生产者加入数据并 signal；
消费者错过 signal；
消费者随后睡眠；
之后没人再 signal；
消费者永远睡眠。
```

这就是丢失唤醒。

信号量和条件变量的语义不同：

```text
信号量：post 可以保存一次资源计数，有记忆。
条件变量：signal 如果没人等待，就不会保存，没有记忆。
```

因此用信号量模拟条件变量时，很容易出错。

---

# 23. “小心泛化”的含义

信号量看起来很通用：

```text
可以当锁；
可以做条件同步；
可以做资源计数。
```

但更通用不一定更好。

锁和条件变量的语义更清晰：

```text
mutex：专门负责互斥；
condition variable：专门负责等待某个条件变化；
semaphore：同时混合了计数、等待、通知和互斥等语义。
```

因此并发程序中不应盲目认为信号量可以替代一切同步原语。

# 24. 总结

## 24.1 生产者/消费者问题

```text
empty/full 解决缓冲区空满条件；
mutex 解决并发访问 buffer、fill、use 的互斥问题。
```

正确顺序：

```text
生产者：wait(empty) → wait(mutex) → put → post(mutex) → post(full)
消费者：wait(full)  → wait(mutex) → get → post(mutex) → post(empty)
```

---

## 24.2 读者-写者锁

```text
多个读者可以并发；
写者必须独占；
readers 用来判断第一个读者和最后一个读者；
lock 保护 readers；
writelock 负责阻止读写冲突和写写冲突。
```

核心规则：

```text
第一个读者拿 writelock；
最后一个读者释放 writelock。
```

---

## 24.3 哲学家就餐问题

```text
哲学家 = 线程；
餐叉 = 共享资源；
forks[5] = 全局共享信号量数组。
```

原始方案会死锁，因为所有哲学家都先拿左叉再等右叉，形成循环等待。

改进方案是改变一个哲学家的拿叉顺序，从而破坏循环等待。

---

## 24.4 Zemaphore

```text
Zemaphore = mutex + condition variable + counter
```

其中：

```text
value：记录可用资源数量；
lock ：保护 value；
cond ：在资源不足时睡眠，在资源增加时唤醒。
```

# 25. 易错点

## 25.1 `empty/full` 不等于互斥锁

`empty` 和 `full` 只表示缓冲区状态：

```text
empty：空槽位数量；
full ：已有数据数量。
```

它们不能保护：

```text
buffer[fill]
fill
buffer[use]
use
```

这些共享变量仍然需要 `mutex` 保护。

---

## 25.2 不要拿着互斥锁去等待可能阻塞的信号量

错误写法：

```c
sem_wait(&mutex);
sem_wait(&full);
```

如果 `full = 0`，线程会持有 `mutex` 睡眠，可能导致死锁。

正确写法：

```c
sem_wait(&full);
sem_wait(&mutex);
```

---

## 25.3 条件变量的 `wait` 和信号量的 `wait` 不一样

条件变量：

```c
pthread_cond_wait(&cond, &mutex);
```

会在睡眠时自动释放 `mutex`，被唤醒后再重新获得 `mutex`。

普通信号量：

```c
sem_wait(&s);
```

阻塞时不会自动释放你之前拿到的其他锁。

---

## 25.4 读写锁可能导致写者饥饿

简单读写锁通常偏向读者。

如果不断有新读者进入，写者可能一直得不到机会。

解决思路：

```text
有写者等待时，阻止新的读者进入。
```

---

## 25.5 哲学家就餐问题的关键是循环等待

死锁场景：

```text
每个线程都拿着一个资源，同时等待下一个资源。
```

避免方法之一：

```text
让至少一个线程改变资源申请顺序，破坏循环等待。
```

---

## 25.6 条件变量必须用 `while` 检查条件

推荐写法：

```c
while (condition_not_true)
    pthread_cond_wait(&cond, &mutex);
```

不要简单写成：

```c
if (condition_not_true)
    pthread_cond_wait(&cond, &mutex);
```

因为线程被唤醒后，条件不一定真的满足。

```text
生产者消费者：

```
