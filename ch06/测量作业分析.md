# 题目：测量系统调用和上下文切换的成本

## 实验代码（`hw.c`）

```c
#define _GNU_SOURCE // 必须定义，为了使用 sched_setaffinity
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/time.h>
#include <sched.h>
#include <sys/wait.h>

// 获取当前时间的微秒级时间戳
long long get_time_us() {
    struct timeval tv;
    gettimeofday(&tv, NULL);
    return (tv.tv_sec * 1000000LL) + tv.tv_usec;
}

// 将当前进程绑定到 CPU 核心 0
void bind_to_cpu0() {
    cpu_set_t set;
    CPU_ZERO(&set);        // 清空集合
    CPU_SET(0, &set);      // 将 CPU 0 加入集合
    // 设置当前进程(0代表当前进程)的 CPU 亲和性
    if (sched_setaffinity(0, sizeof(cpu_set_t), &set) == -1) {
        perror("sched_setaffinity failed");
        exit(1);
    }
}

int main() {
    long long start, end;
    int iterations = 1000000; // 迭代次数（100万次，为了让 gettimeofday 足够精确）

    printf("========== OS 性能测量实验 ==========\n");

    // =================================================================
    // 任务 1：测量单次系统调用的成本
    // 策略：调用 read(0, NULL, 0) 100万次，计算平均时间。
    // =================================================================
    start = get_time_us();
    for (int i = 0; i < iterations; i++) {
        // 读取 0 字节，这是一个合法的空操作系统调用，能纯粹触发 trap 陷入内核
        read(0, NULL, 0); 
    }
    end = get_time_us();

    double total_syscall_time = (double)(end - start);
    double avg_syscall_time = total_syscall_time / iterations;
    printf("[1] 平均系统调用时间: %f 微秒 (共执行 %d 次)\n", avg_syscall_time, iterations);


    // =================================================================
    // 任务 2：测量上下文切换的成本
    // 策略：创建两个管道，父子进程互相打乒乓球。绑定到同一个 CPU。
    // =================================================================
    int pipe1[2]; // 父写 -> 子读
    int pipe2[2]; // 子写 -> 父读

    if (pipe(pipe1) == -1 || pipe(pipe2) == -1) {
        perror("pipe creation failed");
        exit(1);
    }

    pid_t pid = fork();

    if (pid < 0) {
        perror("fork failed");
        exit(1);
    } 
    else if (pid == 0) {
        // ---------------- 子进程 ----------------
        bind_to_cpu0(); // 作业要求：绑定到同一个 CPU，防止多核并行干扰
        char msg_child;
        for (int i = 0; i < iterations; i++) {
            read(pipe1[0], &msg_child, 1); // 阻塞，等待父进程写入 -> 触发上下文切换
            write(pipe2[1], "a", 1);       // 写入数据，唤醒父进程
        }
        exit(0); // 子进程完成任务后退出
    } 
    else {
        // ---------------- 父进程 ----------------
        bind_to_cpu0(); // 父进程也绑定到 CPU 0
        char msg_parent;

        start = get_time_us();
        for (int i = 0; i < iterations; i++) {
            write(pipe1[1], "a", 1);       // 写入数据，唤醒子进程
            read(pipe2[0], &msg_parent, 1); // 阻塞，等待子进程写回 -> 触发上下文切换
        }
        end = get_time_us();
        
        wait(NULL); // 等待子进程彻底死亡，清理 PCB

        double total_switch_time = (double)(end - start);
        
        // 核心数学计算：
        // 1 次循环 = 父写 -> 父阻塞切换到子(1次切换) -> 子读 -> 子写 -> 子阻塞切换到父(1次切换) -> 父读
        // 所以 1 次循环经历了 2 次上下文切换。
        // 总切换次数 = iterations * 2。
        double avg_switch_cost = total_switch_time / (iterations * 2);

        printf("[2] 平均上下文切换时间: %f 微秒 (基于 %d 次往返)\n", avg_switch_cost, iterations);
        
        // 进阶推导：严格来说，这里的往返时间还包含了 pipe 的 read 和 write 系统调用的成本。
        // 如果追求极度精确，应该减去系统调用的时间。
        double pure_switch_cost = avg_switch_cost - avg_syscall_time;
        printf("    -> 扣除管道系统调用开销后的纯切换时间 (约): %f 微秒\n", pure_switch_cost);
    }

    return 0;
}
```
## 运行结果
```
========== OS 性能测量实验 ==========
[1] 平均系统调用时间: 0.855376 微秒 (共执行 1000000 次)
[2] 平均上下文切换时间: 4.980923 微秒 (基于 1000000 次往返)
    -> 扣除管道系统调用开销后的纯切换时间 (约): 4.125547 微秒
```

## 实验分析与思考

### 1. 系统调用的成本

系统调用就是 **trap 触发 → 保存少量寄存器 → 特权级提升到 0 → 内核立刻执行 sysret → 特权级降回 3** 的全过程物理耗时。你看，仅仅只是跨越一次"安检门"，即使不干任何事（读 0 字节），也要消耗零点几微秒。

### 2. 上下文切换的成本为什么更贵？

上下文切换的成本比单次系统调用贵了好几倍！为什么呢？

- 它不仅包含了系统调用（read/write 会触发 trap）。
- 它还包含了操作系统**调度器（Scheduler）**运行算法挑选下一个进程的时间。
- 更关键的是，它包含了**"换脑手术"**：把 CPU 的 SP 指针从父进程的 PCB 切换到子进程的 PCB，以及 CPU 内部缓存（如 TLB 页表缓存）失效带来的巨大隐形性能惩罚。

## 深入问答

### 问1：为什么是"父写 → 父阻塞切换到子 → 子读 → 子写 → 子阻塞切换到父 → 父读"这个顺序？

这是一个极其通透的联想！它不仅看懂这段测量的代码，还把它和操作系统最核心的**进程状态模型（运行、就绪、阻塞）**完美地闭环了。

管道通信的互相切换和等待磁盘 I/O 的切换，在操作系统的底层逻辑里，根本就是**同一件事**！

#### 为什么一定是这个"打乒乓球"的顺序？

代码里创建了两个进程（父和子），给了它们两根水管（pipe1 和 pipe2）：

- 父进程的代码顺序：写 pipe1 → 读 pipe2
- 子进程的代码顺序：读 pipe1 → 写 pipe2

当 fork() 执行完后，父子进程都处于**就绪（Ready）**状态，理论上谁先跑都可以。但操作系统的机制会强制它们陷入这个死循环顺序：

假设 OS 先让子进程跑：

1. 子进程冲上去执行 `read(pipe1, ...)`。
2. **发生阻塞！** 因为此时 pipe1 里根本没数据，水管是空的。操作系统一看："你没水喝，别占着茅坑"，直接强制剥夺子进程的 CPU 控制权，把它扔进**阻塞队列（Blocked Queue）**。
3. **发生第 1 次切换**：OS 从就绪队列里把父进程提上来。
4. 父进程执行 `write(pipe1, ...)`。水管 1 里有水了！操作系统立刻把刚才阻塞的子进程踢醒，放回就绪队列。
5. 父进程继续往下跑，执行 `read(pipe2, ...)`。
6. **再次阻塞！** 因为此时 pipe2 是空的。操作系统又把父进程扔进阻塞队列。
7. **发生第 2 次切换**：OS 把刚才踢醒的子进程提上来。
8. 子进程刚才卡在读 pipe1，现在读到了数据，继续往下跑，执行 `write(pipe2, ...)`，这又唤醒了父进程……

> 不管谁先跑，只要碰到空水管，就必须被强行切走。这就像是一个咬合得无比严密的齿轮，通过 **read 一个空管道必定阻塞**的特性，人为地逼迫操作系统在这里极其频繁地、100% 确定地发生上下文切换。这就是大名鼎鼎的 **lmbench** 测量算法的精妙之处。

### 问2：这是否类似 `-I IO_RUN_LATER` 和 `-I IO_RUN_IMMEDIATE` 的机制？

在真实的操作系统（如 Linux）中，底层的核心机制是 **`-I IO_RUN_LATER`（乖乖排队）**，但通过动态优先级调整（抢占机制），最终表现出的行为往往是 **`-I IO_RUN_IMMEDIATE`（立刻插队）**。

#### `-I IO_RUN_LATER`（稍后运行模式：老实排队）

在这种模式下，操作系统的规则是**绝对不抢占**。

- **规则**：当进程 A 的 I/O 完成（或收到管道数据）被唤醒时，OS 只是把它从"阻塞队列"捞出来，放到"就绪队列"的**队尾**。此时 CPU 依然由当前正在跑的进程 B 继续霸占，直到 B 的时间片用完，或者 B 自己阻塞。

- **在代码里的表现**：
  - 父进程执行 write(pipe1)。水管有水了。
  - 子进程被唤醒，放入就绪队列。（此时**没有**发生切换！因为是 RUN_LATER，父进程继续霸占 CPU）
  - 父进程接着往下执行下一行代码：read(pipe2)。
  - 父进程发现 pipe2 是空的，被迫阻塞，交出 CPU。
  - OS 调度子进程上台执行。

#### `-I IO_RUN_IMMEDIATE`（立即运行模式：霸道插队）

在这种模式下，操作系统的规则是**允许绝对抢占**。

- **规则**：当进程 A 的 I/O 完成被唤醒的那一微秒，OS 会立刻按下暂停键，无情地剥夺当前进程 B 的 CPU，强行进行上下文切换，让进程 A 立刻上台跑！

- **在代码里的表现**：
  - 父进程执行 write(pipe1)。水管有水了。
  - 子进程被唤醒。**触发 IMMEDIATE 规则！**
  - OS 立刻剥夺父进程的 CPU（父进程的 read(pipe2) 还没来得及执行），强制发生上下文切换。
  - 子进程抢到 CPU，执行 read(pipe1)，拿走数据，然后执行 write(pipe2)。
  - 子进程接着执行下一轮的 read(pipe1)，发现没数据，被迫阻塞。
  - OS 发生第二次切换，把 CPU 还给父进程，父进程这才如梦初醒般地执行刚才没来得及执行的 read(pipe2)。

#### 💡 核心真相：为什么我们的代码感觉不出区别？

在 Pipe 测速代码里，不管 OS 底层是用 LATER 还是 IMMEDIATE，**最终都会发生两次极其紧凑的上下文切换**！

- 如果是 **LATER**，父进程是因为执行了下一行 read 而**主动**让出 CPU。
- 如果是 **IMMEDIATE**，父进程在执行 write 的瞬间就被**被动**抢占了 CPU。

因为父进程的 write 和 read 之间没有任何其他的耗时计算代码，这两行代码的执行间隔是纳秒级的。所以从宏观的测量时间来看，这两种模式在实验里**殊途同归**，都完美地逼迫 OS 进行了切换。

#### 为什么现实中的 OS 更偏爱 `IO_RUN_IMMEDIATE`？

答案是两个字：**系统资源利用率（重叠执行）**。

想象一下真实场景：进程 A 一直在疯狂读写磁盘，进程 B 一直在疯狂算圆周率（纯占用 CPU）。

| 模式 | 行为 | 结果 |
|------|------|------|
| **IO_RUN_LATER** | 进程 A 的磁盘数据读取完毕，但进程 B 还在霸占 CPU 算圆周率，进程 A 只能在就绪队列里干等。 | CPU 在忙，但昂贵的磁盘**闲置**了。系统吞吐量极速下降。 |
| **IO_RUN_IMMEDIATE** | 磁盘数据一读完，OS 立刻打断算圆周率的 B，让 A 上台。A 拿到数据后稍微处理一下，马上又发起下一次磁盘读取请求。A 发出请求后再次阻塞，OS 再把 CPU 还给 B 算圆周率。 | CPU 在算圆周率的**同时**，磁盘也在马不停蹄地转动读数据。实现了 **CPU 和 I/O 设备的完美并行**。 |

日常用的 Windows 和 Linux，其底层调度算法虽然极其复杂（比如 Linux 的 CFS 完全公平调度器），但在处理策略上，它们都会极其偏心地给 **I/O 密集型进程**（比如响应你鼠标键盘的进程、读写文件的进程）临时提升极高的优先级。这相当于在底层变相实现了 `IO_RUN_IMMEDIATE`，保证了电脑的绝对流畅和硬件资源的最大化压榨。
