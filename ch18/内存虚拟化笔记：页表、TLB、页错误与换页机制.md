# 内存虚拟化笔记：页表、TLB、页错误与换页机制

## 1. 页表存在哪里？

页表本质上存放在**物理内存 RAM** 中。

每个进程通常都有自己的页表，页表由操作系统创建和维护。CPU/MMU 在进行地址转换时，会根据当前进程的页表把虚拟地址转换成物理地址。

```text
虚拟地址
   ↓
MMU 查询 TLB
   ↓
TLB miss 时查询页表
   ↓
得到物理页框号 PFN
   ↓
访问物理内存
```

页表不在 CPU 中完整保存。CPU 中通常只有：

```text
TLB：页表项的高速缓存
页表基址寄存器：指向当前进程页表的根位置
```

完整页表仍然在物理内存中。

---

## 2. 页表基址保存在哪里？

进程创建时，操作系统会为进程创建地址空间和页表，并记录页表的物理基址。

逻辑上可以理解为：

```text
PCB
 ├── 进程状态
 ├── 寄存器上下文
 ├── 调度信息
 └── 内存管理信息
       └── 页表基址
```

更真实的系统中，PCB 可能不直接存页表基址，而是指向一个内存管理结构，例如：

```text
PCB / task_struct
   ↓
mm_struct
   ↓
页表根基址
```

当进程被调度运行时，操作系统会把该进程的页表基址加载到 CPU 的特殊寄存器中。

例如抽象上叫：

```text
PTBR：Page Table Base Register
```

在 x86 中类似：

```text
CR3
```

进程切换时，操作系统也会切换这个寄存器。

```text
进程 A 运行：
PTBR → 进程 A 的页表

切换到进程 B：
PTBR → 进程 B 的页表
```

---

## 3. 线性页表的基本结构

线性页表可以理解为一个数组：

```text
page_table[VPN] = PTE
```

其中：

```text
VPN：虚拟页号
PTE：页表项
PFN：物理页框号
```

一个 PTE 通常记录：

```text
valid / present 位
权限位
dirty 位
accessed / reference 位
PFN
OS 保留字段
```

如果一个进程的虚拟地址空间能划分为 n 个虚拟页，每个 PTE 大小为 4B，那么线性页表光存 PTE 就需要：

```text
4n B
```

例如：

```text
32 位虚拟地址空间
页大小 = 4KB = 2^12 B
PTE 大小 = 4B
```

虚拟页数量：

```text
2^32 / 2^12 = 2^20
```

页表大小：

```text
2^20 × 4B = 4MB
```

所以一个进程的线性页表就可能达到 4MB。如果有 100 个进程，仅页表就可能占用数百 MB 物理内存。

---

## 4. 为什么线性页表中大量 PTE 是 invalid？

程序的虚拟地址空间通常是稀疏的。

典型布局：

```text
低地址
┌──────────────┐
│ 代码段        │
├──────────────┤
│ 数据段        │
├──────────────┤
│ 堆，向上增长   │
│              │
│ 大片未使用空间 │
│              │
│ 栈，向下增长   │
└──────────────┘
高地址
```

如果使用线性页表，必须为整个虚拟地址空间准备 PTE。

即使中间大片地址没有使用，也仍然需要对应的 PTE，只是这些 PTE 被标记为 invalid。

```text
VPN 0   → valid
VPN 1   → invalid
VPN 2   → invalid
...
VPN 14  → valid
VPN 15  → valid
```

这里浪费的不是物理数据页，而是**页表项本身占用的内存**。

invalid 的意思是：

```text
这个虚拟页当前没有映射到任何物理页框
```

并不是说：

```text
它已经占用了某个物理页框但没用
```

---

## 5. 为什么 invalid PTE 不能分给其他进程？

因为线性页表依赖数组下标：

```text
PTE = page_table[VPN]
```

例如访问 VPN 6，硬件就会去查：

```text
page_table[6]
```

所以即使 `page_table[6]` 是 invalid，它的位置也不能拿走给别的进程使用。

否则当前进程访问 VPN 6 时，就会查到别的进程的映射信息，地址转换会混乱。

类比：

```text
一个班有 100 个学号
即使只有 20 个学生
如果用学号作为数组下标
空学号的位置也不能放另一个班学生的信息
```

---

## 6. 页表本身也要占用物理页框

页表是数据结构，也要放在物理内存中。

所以图中常见两类 PFN：

```text
第一类 PFN：页表页自己所在的物理页框
第二类 PFN：PTE 指向的目标数据物理页框
```

例如：

```text
PFN 201：存放页表项 PTE
```

这表示物理页框 201 被用来存页表。

而页表项里面可能记录：

```text
VPN 0 → PFN 12
VPN 1 → PFN 13
VPN 3 → PFN 100
```

这里的 PFN 12、13、100 才是用户数据或代码真正所在的物理页框。

可以画成：

```text
物理页框 PFN 201
┌────────────────────┐
│ PTE for VPN 0       │ → PFN 12
│ PTE for VPN 1       │ → PFN 13
│ PTE for VPN 2       │ → invalid
│ PTE for VPN 3       │ → PFN 100
└────────────────────┘
```

---

## 7. 一个页表页为什么能存多个 PTE？

一个页表页能存多少 PTE，取决于：

```text
一个页表页能存的 PTE 数量 = 页大小 / PTE 大小
```

例如真实系统常见：

```text
页大小 = 4KB = 4096B
PTE 大小 = 4B
```

则一个页表页可以存：

```text
4096 / 4 = 1024
```

个 PTE。

如果某个教材例子里一个页表页能存 4 个 PTE，那只是为了教学简化，表示：

```text
页大小 / PTE 大小 = 4
```

## 8. 线性页表的问题

线性页表有两个主要问题。

### 8.1 空间浪费

线性页表必须为整个虚拟地址空间准备 PTE。

即使大量地址没有使用，也要有 invalid PTE。

### 8.2 要求连续物理内存

线性页表本质是一个连续数组：

```text
page_table[0]
page_table[1]
page_table[2]
...
page_table[n-1]
```

硬件通过公式定位 PTE：

```text
PTE 地址 = 页表基址 + VPN × PTE 大小
```

所以整个线性页表通常要求连续存放。

如果页表大小是 4MB，就需要找到一整块连续的 4MB 物理内存。

但物理内存可能碎片化：

```text
空闲 4KB
已用 4KB
空闲 8KB
已用 4KB
空闲 16KB
...
```

总空闲内存可能够，但没有连续 4MB，就放不下线性页表。

## 9. 多级页表的思想

多级页表的核心思想是：

> 不为未使用的大段虚拟地址空间分配页表页。

线性页表：

```text
PTBR → 一个大页表
```

多级页表：

```text
PDBR → 页目录 → 页表页 → 数据页
```

例如两级页表：

```text
虚拟地址
┌──────────────┬──────────────┬──────────────┐
│ 页目录索引    │ 页表索引      │ 页内偏移      │
└──────────────┴──────────────┴──────────────┘
```

页目录项 PDE 指向某个二级页表页。

如果某一大段虚拟地址完全没用，对应的页目录项可以直接 invalid，二级页表页根本不分配。

```text
页目录项 0 → valid，指向页表页
页目录项 1 → invalid，不分配页表页
页目录项 2 → invalid，不分配页表页
页目录项 3 → valid，指向页表页
```

---

## 10. 多级页表为什么不要求连续物理内存？

线性页表要求：

```text
PFN 201
PFN 202
PFN 203
PFN 204
```

连续存放。

多级页表通过页目录增加了一层间接：

```text
页目录项 0 → PFN 201
页目录项 1 → PFN 900
页目录项 2 → PFN 17
页目录项 3 → PFN 204
```

各个页表页可以散落在物理内存的任意位置。

这样操作系统只需要每次找一个空闲物理页框来放页表页，不需要找一大块连续物理内存。

---

## 11. 为什么有时需要三级或更多级页表？

多级页表的设计目标之一是：

> 每一级页表结构最好都能放入一个页中。

假设：

```text
虚拟地址空间 = 30 位
页大小 = 512B = 2^9 B
PTE 大小 = 4B
```

页内偏移需要 9 位，所以 VPN 位数是：

```text
30 - 9 = 21
```

一个页表页能放：

```text
512 / 4 = 128 = 2^7
```

个 PTE。

因此每一级索引最多用 7 位。

如果使用两级页表：

```text
| 页目录索引 14 位 | 页表索引 7 位 | 页内偏移 9 位 |
```

页目录有：

```text
2^14 项
```

每项 4B，页目录大小：

```text
2^14 × 4B = 64KB
```

但页大小只有 512B，所以页目录放不进一个页。

于是继续拆页目录，变成三级：

```text
| 一级索引 7 位 | 二级索引 7 位 | 三级索引 7 位 | 页内偏移 9 位 |
```

这样每一级都是：

```text
2^7 × 4B = 512B
```

刚好一个页。

## 12. 反向页表 Inverted Page Table

普通页表方向是：

```text
VPN → PFN
```

反向页表方向是：

```text
PFN → PID + VPN
```

也就是说，整个系统维护一张表，每个表项对应一个物理页框，记录：

```text
这个物理页框被哪个进程的哪个虚拟页使用
```

例如：

```text
PFN 0：进程 A，VPN 3
PFN 1：空闲
PFN 2：进程 B，VPN 10
PFN 3：进程 A，VPN 0
```

反向页表的优点：

```text
表大小取决于物理页框数量
而不是所有进程的虚拟页数量
```

所以很省空间。

缺点是查找不方便。

进程访问某个虚拟页时，输入是：

```text
PID + VPN
```

但反向页表按 PFN 排列。要从 `PID + VPN` 找 PFN，不能直接下标访问。

最朴素方式是线性扫描整个表，太慢。

实际系统通常配合哈希表：

```text
hash(PID, VPN) → 反向页表候选项 → PFN
```

## 13. TLB 的作用

TLB 是 Translation Lookaside Buffer，本质是页表项的硬件缓存。

```text
TLB 缓存最近用过的 VPN → PFN 映射
```

内存访问过程：

```text
CPU 产生虚拟地址
   ↓
取出 VPN
   ↓
查 TLB
   ↓
TLB hit：直接得到 PFN
   ↓
TLB miss：查页表
```

没有 TLB 时，每次访存都要查页表，尤其多级页表下会非常慢。

例如四级页表：

```text
查一级页表
查二级页表
查三级页表
查四级页表
访问真实数据
```

TLB 使大多数地址转换可以快速完成。

## 14. TLB miss 和 page fault 的区别

### 14.1 TLB miss

TLB miss 表示：

```text
TLB 中没有这个 VPN 的映射
```

但页可能仍然在内存中。

处理方式：

```text
查页表
如果 PTE valid
把 VPN → PFN 填入 TLB
重新执行访问
```

TLB miss 不一定是错误。

### 14.2 page fault

page fault 表示：

```text
查页表后发现该页当前不能直接访问
```

可能原因：

```text
虚拟地址非法
页被换出到硬盘
按需置零页尚未分配物理页
写时复制需要复制新页
权限错误
```

page fault 必须由操作系统处理。

---

## 15. 硬件管理 TLB 和软件管理 TLB

### 15.1 硬件管理 TLB

TLB miss 后，由硬件 MMU 自动查页表：

```text
TLB miss
   ↓
硬件查页表
   ↓
找到 PTE
   ↓
填入 TLB
```

这种方式要求硬件必须理解页表格式。

操作系统必须按照硬件规定的格式建立页表。

### 15.2 软件管理 TLB

TLB miss 后，硬件触发异常，进入操作系统：

```text
TLB miss
   ↓
陷入内核
   ↓
OS 的 TLB miss handler 查页表
   ↓
OS 填写 TLB
   ↓
返回用户程序
```

这种方式下，硬件不需要知道页表结构，页表可以更灵活。

操作系统可以使用：

```text
线性页表
多级页表
哈希页表
反向页表
其他自定义结构
```

区别总结：

| 类型 | TLB miss 后谁查页表 | 页表格式 |
|---|---|---|
| 硬件管理 TLB | 硬件 MMU | 受硬件限制 |
| 软件管理 TLB | 操作系统 | 更灵活 |

## 16. 页错误 page fault 的处理

当页不在内存中时，PTE 可能是：

```text
present = 0
swap location = 硬盘位置
```

当进程访问该页：

```text
访问虚拟地址
   ↓
TLB miss
   ↓
查页表
   ↓
PTE present = 0
   ↓
触发 page fault
```

操作系统的 page-fault handler 会判断原因。

如果该页被换出到硬盘：

```text
1. 从 PTE 或辅助结构中找到 swap 位置
2. 找一个空闲物理页框
3. 从硬盘把页读入该物理页框
4. 更新 PTE：
   present = 1
   PFN = 新物理页框号
5. 重新执行刚才失败的指令
```

---

## 17. 硬盘地址是谁写入 PTE 的？

是操作系统在页面被换出时写入的。

例如原来：

```text
VPN 5:
present = 1
PFN = 120
```

内存不足时，操作系统选择换出 VPN 5。

如果页面是脏页，先写入 swap 区：

```text
PFN 120 → swap slot 3000
```

然后修改 PTE：

```text
VPN 5:
present = 0
swap location = 3000
```

之后如果进程再次访问 VPN 5，page fault handler 从 PTE 中读出 swap location，把页面换回内存。

---

## 18. 换出时脏页和干净页的区别

### 18.1 脏页 dirty page

脏页表示：

```text
内存中的内容被修改过
硬盘上的旧内容已经不是最新的
```

换出前必须写回硬盘或 swap 区，否则数据会丢失。

```text
脏页 → 写回硬盘 → 释放物理页框
```

### 18.2 干净页 clean page

干净页表示：

```text
内存中的内容和硬盘/文件中的内容一致
```

可以直接丢弃，不需要写回。

以后再次访问时，从原文件、swap 或重新构造读回即可。

```text
干净页 → 直接释放物理页框
```

### 18.3 已写回的脏页

脏页被后台线程提前写回后，如果之后没有再次修改，就变成干净页。

这样以后回收时可以直接释放。

## 19. High Watermark 和 Low Watermark

操作系统通常维持一定数量的空闲物理页。

两个阈值：

```text
LW：Low Watermark，低水位线
HW：High Watermark，高水位线
```

规则：

```text
如果空闲页数量 < LW：
    唤醒后台 page daemon / swap daemon
    开始回收页面

直到空闲页数量恢复到 HW：
    page daemon 睡眠
```

例如：

```text
LW = 100
HW = 500
```

如果空闲页只剩 80 个：

```text
80 < LW
```

后台页守护进程开始回收页面，直到空闲页恢复到 500 个。

---

## 20. 为什么后台 page daemon 可以提高性能？

如果等到 page fault 时才临时找 victim page，可能需要：

```text
发生 page fault
   ↓
没有空闲页
   ↓
找 victim page
   ↓
如果 victim 是脏页，写回硬盘
   ↓
再读入需要的页
```

这会使当前进程等待很久。

后台 page daemon 会提前准备空闲页：

```text
后台扫描页面
   ↓
选择不常用页面
   ↓
脏页提前写回
   ↓
形成一批 clean page / free page
```

这样之后 page fault 发生时，更可能直接拿到空闲页。

## 21. 页聚集 clustering

硬盘 I/O 的固定开销很大，尤其机械硬盘有：

```text
寻道
旋转等待
I/O 请求开销
中断处理开销
```

如果每次只写一个小页，比如 512B 或 4KB，效率很低。

页聚集就是把多个要写回的脏页合并成一次较大的写操作：

```text
page A
page B
page C
page D
   ↓
一次性批量写入 swap 区
```

优点：

```text
减少 I/O 请求次数
减少寻道和旋转开销
提高顺序写吞吐量
```

现代系统也普遍使用类似思想。

## 22. 物理内存是否划分为用户和内核两部分？

物理内存通常不是固定划分为用户区和内核区。

物理内存更像一个页框池：

```text
PFN 0：可能给内核
PFN 1：可能给用户进程 A
PFN 2：可能作为页表页
PFN 3：可能作为文件缓存
PFN 4：可能给用户进程 B
```

某个物理页框当前属于用户还是内核，取决于：

```text
它被映射到哪里
PTE 权限如何设置
操作系统如何管理它
```

虚拟地址空间通常会划分用户区和内核区：

```text
进程虚拟地址空间
低地址：用户空间
高地址：内核空间
```

但物理内存本身通常由操作系统统一动态管理。

---

## 23. 物理页框管理器不是系统页表

两者是不同的数据结构。

### 23.1 页表 / 系统页表

页表负责：

```text
虚拟地址 → 物理地址
```

例如：

```text
进程 A 页表：
VPN 20 → PFN 100

系统页表：
内核虚拟页 KVPN 5 → PFN 200
```

### 23.2 物理页框管理器 frame table

物理页框管理器负责：

```text
PFN → 这个物理页框当前的状态
```

例如：

```text
frame_table[100]:
    owner = process A
    type = user page
    dirty = yes
    reference_count = 1

frame_table[200]:
    owner = kernel
    type = page table page

frame_table[300]:
    state = free
```

所以方向不同：

```text
页表：VPN → PFN
frame table：PFN → 状态信息
```

PTE 可以提供部分信息，例如权限、dirty、present，但不能完整管理所有物理页框状态。

## 24. 内核为什么映射到每个进程的地址空间？

现代系统通常把同一份内核映射到每个进程的虚拟地址空间高地址部分。

例如：

```text
进程 A 地址空间：
低地址：A 的用户空间
高地址：内核空间 → 同一份物理内核

进程 B 地址空间：
低地址：B 的用户空间
高地址：内核空间 → 同一份物理内核
```

注意：不是每个进程有一份内核拷贝，而是都映射到同一份内核物理页。

---

## 25. 内核映射到每个地址空间的好处

### 25.1 系统调用时方便访问用户指针

例如用户调用：

```c
write(fd, buf, len);
```

其中 `buf` 是用户虚拟地址。

进入内核态后，CPU 权限改变，但通常仍然使用当前进程的页表。

当前进程页表里同时包含：

```text
用户空间映射
内核空间映射
```

所以内核可以理解用户传入的指针。

```text
buf = 0x40010000
```

在进程 A 的页表中仍然表示进程 A 的用户缓冲区。

内核可以执行：

```text
copy_from_user(kernel_buffer, buf, len)
```

当然，内核会检查地址是否合法，不能盲目信任用户指针。

### 25.2 内核代码和数据地址固定

无论当前运行哪个进程，内核虚拟地址都相同。

```text
进程 A 页表：
0xC0000000 → 内核代码物理页

进程 B 页表：
0xC0000000 → 同一份内核代码物理页
```

这样内核代码可以使用固定虚拟地址运行。

### 25.3 如果内核有完全独立地址空间会变复杂

如果系统调用时切换到完全独立的内核地址空间，用户传入的虚拟地址在内核页表中可能无效。

内核要访问用户数据，就必须：

```text
查用户页表
找到物理页
临时映射到内核空间
再复制数据
```

很麻烦。

---

## 26. VMS 中的页表设计背景

VAX 的页大小很小：

```text
页大小 = 512B
```

页越小，虚拟页数量越多，页表越大。

VMS 设计者担心页表占满内存，因此采取了两个办法。

---

## 27. VMS 方法一：把用户地址空间分成 P0 和 P1

VAX-11 把用户地址空间分为：

```text
P0：代码、数据、堆
P1：栈
```

每个区域有自己的页表。

```text
P0 页表：管理代码、堆等
P1 页表：管理栈
```

这样堆和栈之间的大段未使用空间不需要页表项。

每个区域有：

```text
基址寄存器：保存该区域页表位置
界限寄存器：保存该区域页表大小
```

---

## 28. VMS 方法二：把用户页表放入内核虚拟内存

VMS 把用户页表放到内核虚拟地址空间中。

这并不是说页表不占物理内存，而是说：

> 用户页表不必永久固定占用某些物理页框。

用户页表本身也可以被当成虚拟内存对象管理。

例如：

```text
进程 A 的 P0 页表
位于内核虚拟地址 0x80000000 ~ 0x80003FFF
```

这些内核虚拟地址当前可能映射到物理页框：

```text
0x80000000 → PFN 201
0x80001000 → PFN 850
```

也可能某些页表页被换出到硬盘：

```text
0x80002000 → not present，内容在 swap slot 3000
```

好处：

```text
不活跃进程的部分用户页表页可以换出到磁盘
释放物理内存
```

代价：

```text
地址转换更复杂
```

---

## 29. 为什么用户页表放在内核虚拟内存会让转换更复杂？

因为要访问用户数据，需要先查用户页表；但用户页表本身又在内核虚拟地址空间中。

过程可能变成：

```text
用户虚拟地址
   ↓
需要查用户 P0/P1 页表
   ↓
用户页表页在内核虚拟地址空间
   ↓
先查系统页表，找到用户页表页的物理地址
   ↓
再查用户页表页，找到用户数据页的物理地址
   ↓
访问用户数据
```

这是一种递归式的间接。

幸好 TLB 可以缓存最终转换，大多数时候不用走完整流程。

---

## 30. VAX PTE 中没有 reference bit 的问题

VAX 的 PTE 包含：

```text
valid 位
保护字段
dirty 位
OS 保留字段
PFN
```

但没有：

```text
reference bit / accessed bit
```

引用位用于记录：

```text
这个页最近有没有被访问
```

没有引用位，操作系统很难实现 LRU 或 Clock 这类需要访问历史的替换算法。

---

## 31. Memory hog 问题

memory hog 指占用大量内存的进程。

如果使用全局 LRU，一个进程频繁访问大量页面，可能把其他进程的页面都挤出去。

```text
进程 C 大量访问页面
   ↓
全局 LRU 换出 A、B 进程页面
   ↓
A、B 频繁 page fault
```

为了防止这种情况，VMS 给每个进程设置：

```text
RSS：Resident Set Size
```

RSS 表示：

```text
一个进程最多允许保留在物理内存中的页数
```

超过 RSS 时，必须从该进程自己的页面中换出一个。

---

## 32. VMS 的 segmented FIFO

VMS 使用分段 FIFO 替换策略。

“分段”可以理解为：

```text
每个进程都有自己的 FIFO 队列
```

例如进程 P：

```text
最老                                 最新
page A → page B → page C → page D
```

如果进程 P 超过 RSS，就从队头移除最老的页面：

```text
移除 page A
```

优点：

```text
简单
不需要 reference bit
限制单个进程占用过多内存
```

缺点：

```text
纯 FIFO 可能换出仍然活跃的页面
```

---

## 33. VMS 的二次机会列表

VMS 没有把从进程 FIFO 移除的页立刻彻底释放，而是放入全局二次机会列表。

有两个列表：

```text
全局干净页列表 clean list
全局脏页列表 dirty list
```

当进程 P 超过 RSS，移除一个页 A：

如果 A 是干净页：

```text
从 P 的 FIFO 移除
放到全局 clean list 末尾
```

如果 A 是脏页：

```text
从 P 的 FIFO 移除
放到全局 dirty list 末尾
```

此时 A 的内容仍然在物理内存中，只是已经不算进程 P 的驻留集。

如果 P 在 A 被真正回收前再次访问 A：

```text
P 访问 A
   ↓
page fault
   ↓
OS 发现 A 还在 clean/dirty list 中
   ↓
直接从列表取回 A
   ↓
重新加入 P 的 FIFO
   ↓
避免磁盘 I/O
```

这就是二次机会。

---

## 34. 是否需要遍历全局列表找页面？

逻辑上是在判断：

```text
A 是否还在二次机会列表中？
```

但实际系统一般不会遍历整个链表。

操作系统会在 PTE 或辅助数据结构中记录状态，例如：

```text
VPN A:
    state = reclaimable
    PFN = 120
    list = clean list
```

物理页框表中也可能记录：

```text
frame_table[120]:
    owner = P
    vpn = A
    state = clean-list
    prev = ...
    next = ...
```

因此 page fault handler 可以直接定位到对应物理页框，并从双向链表中摘除。

```text
已知节点 → 修改 prev/next 指针 → O(1) 删除
```

---

## 35. VMS 页聚集 clustering

由于 VAX 页大小只有 512B，如果每次只写一个页面到硬盘，I/O 效率很低。

VMS 会从全局 dirty list 中选择一批脏页，聚集后一次写入磁盘：

```text
dirty page A
dirty page B
dirty page C
dirty page D
   ↓
一次大写入
```

写完后这些页变干净，可以转入 clean list。

这能减少小 I/O 次数，提高磁盘效率。

---

## 36. 按需置零 demand zeroing

按需置零是一种懒惰策略。

普通做法：

```text
进程申请新页
   ↓
OS 立刻分配物理页
   ↓
清零
   ↓
建立映射
```

问题是，如果进程从不访问该页，这些工作就浪费了。

按需置零做法：

```text
进程申请新页
   ↓
OS 只在页表中标记：
    valid = 0
    type = demand-zero
   ↓
不分配物理页
   ↓
不清零
```

如果进程后来访问该页：

```text
访问 demand-zero 页
   ↓
page fault
   ↓
OS 发现这是按需置零页
   ↓
分配物理页
   ↓
清零
   ↓
更新 PTE 为 valid
   ↓
重新执行指令
```

如果进程从不访问该页，物理页分配和清零都可以避免。

---

## 37. 为什么新页必须清零？

因为物理页框之前可能被其他进程使用过。

如果不清零，新进程可能读到旧进程的数据，例如：

```text
密码
文件内容
私有数据
```

所以分配给用户进程的新物理页必须保证初始内容为 0。

按需置零只是把“清零”推迟到第一次真正访问时。

---

## 38. 内存访问完整心智模型

一次内存访问可以这样理解：

```text
程序发出虚拟地址
        ↓
CPU 查 TLB
        ↓
如果 TLB 命中：
        得到 PFN
        访问 cache / 物理内存

如果 TLB 未命中：
        查页表
        ↓
        如果 PTE valid：
                填入 TLB
                重新访问

        如果 PTE invalid：
                触发 page fault
                ↓
                OS 判断原因：
                    非法访问 → 杀死进程
                    页在 swap → 从硬盘读回
                    demand-zero → 分配零页
                    COW → 复制新页
                    权限错误 → 报错或特殊处理
                ↓
                更新页表
                重新执行指令
```

---

## 39. 常见性能现象如何解释？

### 39.1 为什么访问大数组可能变慢？

可能超过了 TLB 覆盖范围，导致 TLB miss 增多。

即使数据在 cache 中，地址转换本身也可能拖慢访问。

### 39.2 为什么 malloc 可能很快，但第一次访问内存慢？

因为操作系统可能用了按需置零。

```text
malloc 只分配虚拟地址范围
第一次访问时才 page fault
再分配物理页并清零
```

### 39.3 为什么内存不足时系统会卡？

因为可能频繁发生 page fault 和 swap。

硬盘 I/O 比内存访问慢很多，频繁换入换出会导致系统抖动。

### 39.4 为什么打印指针不是物理地址？

程序看到的是虚拟地址。

真实物理地址被操作系统和硬件隐藏。

---

## 40. TLB miss、page fault、swap 的层次关系

```text
TLB miss：
    TLB 中没有映射
    页可能仍然在内存中

page fault：
    页表显示当前不能直接访问
    可能是非法，也可能是按需分配、换页等

swap：
    页面被换出到硬盘
    再访问时需要 page fault 读回
```

简单对比：

| 事件 | 含义 | 是否一定是错误 | 是否涉及磁盘 |
|---|---|---|---|
| TLB miss | TLB 没有缓存映射 | 否 | 通常否 |
| page fault | 页表显示不能直接访问 | 否 | 不一定 |
| swap in/out | 页面在内存和硬盘间移动 | 否 | 是 |

---

## 41. 内存虚拟化总结

内存虚拟化的核心目标是：

```text
让每个进程都以为自己拥有独立、连续、受保护的地址空间
```

实现这个目标依赖硬件和操作系统协作：

```text
页表：
    记录虚拟页到物理页的映射

TLB：
    缓存最近使用的地址转换，加速访问

PTE：
    保存 valid、dirty、权限、PFN 等信息

page fault：
    当映射不存在或不可访问时，让操作系统介入处理

swap：
    物理内存不足时，把不常用页换到硬盘

page daemon：
    后台维护空闲页数量，提高换页效率

多级页表：
    节省页表空间，避免连续物理内存需求

反向页表：
    从物理页角度组织映射，进一步节省空间

按需置零：
    延迟物理页分配和清零，避免无用工作
```

一句话概括：

> 虚拟内存通过页表制造地址空间假象，通过 TLB 让假象运行得足够快，通过 page fault 让操作系统在需要时介入完成分配、换页、保护和恢复。

---

## 易错点

### 1. 页表不是存在 CPU 中

完整页表在物理内存中，CPU 中主要是 TLB 和页表基址寄存器。

---

### 2. invalid PTE 不代表占用了物理页

invalid 表示该虚拟页没有有效物理映射。

浪费的是 PTE 本身的空间，不是数据物理页框。

---

### 3. 页表页的 PFN 和数据页的 PFN 不要混淆

```text
PFN 201：可能是存页表的物理页框
PFN 12：可能是 PTE 指向的数据页框
```

---

### 4. 多级页表不是每一级都需要一个寄存器

只需要一个寄存器指向最高一级页表。

后续每一级的位置由上一级页表项指出。

---

### 5. TLB miss 不等于 page fault

TLB miss 只是 TLB 没有缓存映射。

page fault 是页表显示当前访问不能直接完成。

---

### 6. page fault 不一定是程序错误

它可能是：

```text
按需置零
页面换入
写时复制
栈增长
```

只有非法地址或权限错误才通常导致进程异常终止。

---

### 7. 干净页不需要写回硬盘

干净页可以直接丢弃。

脏页必须先写回硬盘或 swap 区。

---

### 8. 物理页框管理器不是系统页表

页表负责地址转换：

```text
VPN → PFN
```

物理页框管理器负责物理页状态：

```text
PFN → 状态信息
```

---

### 9. 内核映射到每个进程地址空间，不代表每个进程有一份内核

它们映射到同一份物理内核，只是每个进程的页表中都有内核虚拟地址区域。

---

### 10. 按需置零不是不清零，而是延迟清零

新物理页最终仍然必须清零，防止泄露其他进程数据。

只是清零发生在第一次真正访问该页时。

---

## 简短记忆版

```text
程序看到的是虚拟地址，不是物理地址。

虚拟地址先查 TLB。
TLB 命中，直接快走。
TLB 未命中，查页表。

页表项 valid：
    填 TLB，继续访问。

页表项 invalid：
    触发 page fault，让 OS 判断原因。

OS 可能：
    杀死非法访问进程；
    从 swap 读回页面；
    分配 demand-zero 零页；
    处理写时复制；
    更新页表后重试指令。

内存不够时：
    干净页直接丢；
    脏页先写回；
    page daemon 后台维持空闲页；
    clustering 批量写回提高磁盘效率。

线性页表简单但浪费；
多级页表按需分配页表页；
反向页表极省空间但查找复杂；
TLB 是让虚拟内存高效运行的关键。
```
