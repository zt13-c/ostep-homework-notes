# 1. 非死锁并发缺陷

并发程序的问题不只有死锁。

Lu 等人的研究表明，很多并发 bug 并不是程序互相等待导致的死锁，而是程序仍然运行，但结果错误、崩溃或者读到错误数据。

常见的非死锁并发缺陷主要有两类：

```text
1. 违反原子性 atomicity violation
2. 违反顺序 order violation
```

---

## 1.1 违反原子性

### 1.1.1 示例代码

```c
Thread 1:
if (thd->proc_info) {
    ...
    fputs(thd->proc_info, ...);
    ...
}

Thread 2:
thd->proc_info = NULL;
```

### 1.1.2 问题在哪里

Thread 1 先检查：

```c
if (thd->proc_info)
```

然后使用：

```c
fputs(thd->proc_info, ...);
```

表面上看没有问题，因为已经判断了 `proc_info` 不是空指针。

但在多线程中，可能发生如下执行顺序：

```text
Thread 1: 判断 thd->proc_info != NULL，成立
Thread 1: 被中断
Thread 2: 执行 thd->proc_info = NULL
Thread 1: 恢复执行
Thread 1: fputs(thd->proc_info, ...)
```

此时 `thd->proc_info` 已经变成 `NULL`，Thread 1 再使用它就可能发生空指针引用，导致程序崩溃。

### 1.1.3 本质

这段代码隐含了一个假设：

```text
检查 proc_info 和使用 proc_info 之间不会被其他线程修改。
```

但是实际并没有任何机制保证这一点。

所以问题不是没有检查空指针，而是：

```text
检查和使用这两个操作本来应该作为一个整体执行，
但实际执行中间被其他线程插入了。
```

这就是违反原子性。

### 1.1.4 什么是原子性

原子性表示：

```text
一组操作要么完整执行完，中间不能被别人插入；
要么还没有开始执行。
```

在这个例子中：

```text
检查 proc_info
使用 proc_info
```

这两个操作应该像一个不可分割的整体一样执行。

---

### 1.1.5 修复方法：加锁

```c
pthread_mutex_t proc_info_lock = PTHREAD_MUTEX_INITIALIZER;

Thread 1:
pthread_mutex_lock(&proc_info_lock);
if (thd->proc_info) {
    ...
    fputs(thd->proc_info, ...);
    ...
}
pthread_mutex_unlock(&proc_info_lock);

Thread 2:
pthread_mutex_lock(&proc_info_lock);
thd->proc_info = NULL;
pthread_mutex_unlock(&proc_info_lock);
```

### 1.1.6 逐行解释

```c
pthread_mutex_t proc_info_lock = PTHREAD_MUTEX_INITIALIZER;
```

定义并初始化一把互斥锁，用来保护 `proc_info`。

```c
pthread_mutex_lock(&proc_info_lock);
```

访问共享变量前先加锁。

```c
if (thd->proc_info) {
```

在持有锁的情况下检查 `proc_info` 是否为空。

```c
fputs(thd->proc_info, ...);
```

在持有锁的情况下使用 `proc_info`。

```c
pthread_mutex_unlock(&proc_info_lock);
```

访问结束后释放锁。

Thread 2 修改 `proc_info` 时也必须持有同一把锁：

```c
pthread_mutex_lock(&proc_info_lock);
thd->proc_info = NULL;
pthread_mutex_unlock(&proc_info_lock);
```

这样就保证：

```text
Thread 1 检查和使用 proc_info 的过程中，
Thread 2 不能把 proc_info 改成 NULL。
```

### 1.1.7 重要原则

不是只给某一段代码加锁就够了。

必须保证：

```text
所有访问同一个共享变量的代码，都使用同一把锁。
```

否则仍然可能出现竞争。

---

## 1.2 违反顺序

### 1.2.1 示例代码

```c
Thread 1:
void init() {
    ...
    mThread = PR_CreateThread(mMain, ...);
    ...
}

Thread 2:
void mMain(...) {
    ...
    mState = mThread->State;
    ...
}
```

### 1.2.2 `PR_CreateThread()` 是什么

```c
mThread = PR_CreateThread(mMain, ...);
```

意思是：

```text
创建一个新线程，让新线程从 mMain 函数开始执行，
并把新线程对象保存到 mThread 中。
```

可以把 `mThread` 理解成指向线程信息的指针。

后面：

```c
mState = mThread->State;
```

表示读取线程对象中的 `State` 成员。

### 1.2.3 问题在哪里

程序员可能以为：

```text
先创建线程；
再把线程对象赋值给 mThread；
然后新线程才开始执行 mMain。
```

但实际可能是：

```text
Thread 1: 调用 PR_CreateThread(mMain, ...)
系统: 新线程 Thread 2 被创建
Thread 2: 立刻开始执行 mMain()
Thread 2: 执行 mState = mThread->State
Thread 1: 还没来得及把返回值赋给 mThread
```

这时 `mThread` 可能还是 `NULL`，于是访问：

```c
mThread->State
```

就可能导致崩溃。

### 1.2.4 本质

Thread 2 假设：

```text
mThread 已经初始化完成。
```

但是程序没有保证：

```text
初始化 mThread 一定发生在使用 mThread 之前。
```

这就是违反顺序。

### 1.2.5 修复方法：条件变量

```c
pthread_mutex_t mtLock = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t mtCond = PTHREAD_COND_INITIALIZER;
int mtInit = 0;

Thread 1:
void init() {
    ...
    mThread = PR_CreateThread(mMain, ...);

    pthread_mutex_lock(&mtLock);
    mtInit = 1;
    pthread_cond_signal(&mtCond);
    pthread_mutex_unlock(&mtLock);
    ...
}

Thread 2:
void mMain(...) {
    ...
    pthread_mutex_lock(&mtLock);
    while (mtInit == 0)
        pthread_cond_wait(&mtCond, &mtLock);
    pthread_mutex_unlock(&mtLock);

    mState = mThread->State;
    ...
}
```

### 1.2.6 逐行解释

```c
pthread_mutex_t mtLock = PTHREAD_MUTEX_INITIALIZER;
```

定义锁，用来保护状态变量 `mtInit`。

```c
pthread_cond_t mtCond = PTHREAD_COND_INITIALIZER;
```

定义条件变量，用来让线程等待初始化完成。

```c
int mtInit = 0;
```

状态变量，表示 `mThread` 是否已经初始化完成。

Thread 1 中：

```c
pthread_mutex_lock(&mtLock);
mtInit = 1;
pthread_cond_signal(&mtCond);
pthread_mutex_unlock(&mtLock);
```

含义是：

```text
设置 mtInit = 1，表示初始化完成；
然后唤醒等待这个条件的线程。
```

Thread 2 中：

```c
pthread_mutex_lock(&mtLock);
while (mtInit == 0)
    pthread_cond_wait(&mtCond, &mtLock);
pthread_mutex_unlock(&mtLock);
```

含义是：

```text
如果初始化还没有完成，就等待；
被唤醒后重新检查条件；
条件满足后才继续执行。
```

注意这里必须使用 `while`，不是 `if`。

原因是：

```text
条件变量可能出现虚假唤醒；
或者被唤醒后条件又不满足。
```

所以醒来后必须重新检查条件。

# 2. 死锁

## 2.1 死锁的基本例子

```c
Thread 1:
lock(L1);
lock(L2);

Thread 2:
lock(L2);
lock(L1);
```

可能发生如下执行顺序：

```text
Thread 1: lock(L1) 成功
Thread 2: lock(L2) 成功
Thread 1: 想 lock(L2)，但 L2 被 Thread 2 持有
Thread 2: 想 lock(L1)，但 L1 被 Thread 1 持有
```

于是：

```text
Thread 1 持有 L1，等待 L2
Thread 2 持有 L2，等待 L1
```

两个线程都在等对方释放锁，程序无法继续执行。

这就是死锁。

---

## 2.2 死锁的 4 个必要条件

死锁必须同时满足 4 个条件。

### 2.2.1 互斥

资源一次只能被一个线程持有。

例如锁就是互斥资源：

```text
一个线程拿到锁以后，其他线程不能同时拿到这把锁。
```

### 2.2.2 持有并等待

线程已经持有一个资源，同时还在等待另一个资源。

例如：

```text
Thread 1 持有 L1，等待 L2
Thread 2 持有 L2，等待 L1
```

### 2.2.3 非抢占

资源不能被强行夺走。

锁通常只能由持有者主动释放：

```text
系统不能强制把锁从某个线程手里抢回来。
```

### 2.2.4 循环等待

线程之间形成环形等待关系。

例如：

```text
Thread 1 -> L2 -> Thread 2 -> L1 -> Thread 1
```

这表示：

```text
Thread 1 等 Thread 2 手里的 L2；
Thread 2 等 Thread 1 手里的 L1。
```

---

## 2.3 死锁预防

死锁需要同时满足 4 个条件。

因此，只要破坏其中任意一个条件，就可以避免死锁。

---

### 2.3.1 破坏循环等待：统一加锁顺序

错误示例：

```c
Thread 1:
lock(L1);
lock(L2);

Thread 2:
lock(L2);
lock(L1);
```

两个线程加锁顺序相反，可能形成循环等待。

解决方法：

```text
规定所有线程必须按照同一个顺序加锁。
```

例如规定：

```text
必须先拿 L1，再拿 L2。
```

所有线程都写成：

```c
lock(L1);
lock(L2);
```

这样 Thread 2 如果拿不到 L1，会在第一步等待，不会先拿着 L2 去等 L1，因此不会形成环。

---

### 2.3.2 全序和偏序

#### 全序 total ordering

给系统中所有锁排一个完整顺序：

```text
L1 < L2 < L3 < L4
```

任何线程拿多个锁时，都必须按照这个顺序拿。

优点：

```text
简单、安全。
```

缺点：

```text
大型系统中锁很多，维护全局顺序很困难。
```

#### 偏序 partial ordering

只规定某些相关锁之间的顺序。

例如：

```text
i_mutex 必须早于 i_mmap_mutex
i_mmap_mutex 必须早于 private_lock
swap_lock 必须早于 i_mmap_mutex
```

偏序比全序灵活，但更难维护。

---

### 2.3.3 按锁地址决定加锁顺序

如果函数需要同时锁两个对象：

```c
void do_something(mutex_t *m1, mutex_t *m2) {
    pthread_mutex_lock(m1);
    pthread_mutex_lock(m2);

    ...

    pthread_mutex_unlock(m2);
    pthread_mutex_unlock(m1);
}
```

如果调用方式不同：

```c
do_something(L1, L2);
do_something(L2, L1);
```

就可能死锁。

解决方法是按地址排序：

```c
void do_something(mutex_t *m1, mutex_t *m2) {
    if (m1 > m2) {
        pthread_mutex_lock(m1);
        pthread_mutex_lock(m2);
    } else {
        pthread_mutex_lock(m2);
        pthread_mutex_lock(m1);
    }

    ...

    if (m1 > m2) {
        pthread_mutex_unlock(m2);
        pthread_mutex_unlock(m1);
    } else {
        pthread_mutex_unlock(m1);
        pthread_mutex_unlock(m2);
    }
}
```

核心思想：

```text
不管调用者传入顺序如何，函数内部都按照固定规则加锁。
```

---

### 2.3.4 破坏持有并等待：一次性拿到所有锁

一种做法是加一把全局锁：

```c
lock(prevention);
lock(L1);
lock(L2);
...
unlock(prevention);
```

含义是：

```text
所有线程在获取多个锁之前，先获取 prevention。
```

这样同一时刻只有一个线程能进入获取多个锁的过程。

优点：

```text
避免死锁。
```

缺点：

```text
并发性下降；
所有锁获取过程被串行化；
不适合封装复杂的大型系统。
```

---

### 2.3.5 破坏非抢占：trylock 失败就释放已有锁

普通锁：

```c
lock(L1);
lock(L2);
```

如果拿不到 `L2`，线程会阻塞，同时继续持有 `L1`。

可以改成：

```c
top:
    lock(L1);
    if (trylock(L2) == -1) {
        unlock(L1);
        goto top;
    }
```

含义：

```text
先拿 L1；
尝试拿 L2；
如果 L2 拿不到，就释放 L1，之后重试。
```

这样不会一直持有 `L1` 等待 `L2`。

缺点是可能出现活锁：

```text
线程一直在运行、释放、重试，
但没有真正取得进展。
```

死锁和活锁区别：

```text
死锁：大家都不动了。
活锁：大家都在动，但事情没有推进。
```

# 3. CAS 与无锁思想

## 3.1 CAS 是什么

CAS 是 Compare-And-Swap，即比较并交换。

简化版本：

```c
int CompareAndSwap(int *address, int expected, int new) {
    if (*address == expected) {
        *address = new;
        return 1;
    }
    return 0;
}
```

含义是：

```text
如果 address 指向的值等于 expected，
就把它改成 new，并返回成功；
否则不修改，并返回失败。
```

关键点：

```text
比较和修改由硬件保证原子完成。
```

---

## 3.2 CAS 实现自旋锁

CAS 可以用来实现自旋锁：

```c
typedef struct {
    int flag;
} lock_t;

void lock(lock_t *lock) {
    while (CompareAndSwap(&lock->flag, 0, 1) == 0)
        ;
}

void unlock(lock_t *lock) {
    lock->flag = 0;
}
```

含义：

```text
如果 flag 是 0，说明锁空闲，把它改成 1，表示拿到锁；
如果 flag 是 1，说明锁已被占用，就一直循环等待。
```

这里 CAS 的作用是：

```text
安全地抢锁。
```

但这种方式本质上仍然是锁。

---

## 3.3 CAS 替代锁

CAS 也可以不用于实现锁，而是直接修改共享数据。

例如原来用锁实现加法：

```c
lock(&mutex);
value = value + amount;
unlock(&mutex);
```

可以用 CAS 写成：

```c
void AtomicIncrement(int *value, int amount) {
    do {
        int old = *value;
    } while (CompareAndSwap(value, old, old + amount) == 0);
}
```

含义：

```text
读取旧值 old；
计算新值 old + amount；
尝试用 CAS 提交；
如果期间没人改过 value，就成功；
如果失败，说明别人改过，重新读取再试。
```

### CAS 实现自旋锁和 CAS 替代锁的区别

| 用法 | CAS 操作对象 | 有没有锁 | 本质 |
|---|---|---|---|
| CAS 实现自旋锁 | 锁变量 flag | 有锁 | 用 CAS 造一把锁 |
| CAS 实现无锁算法 | 共享数据本身 | 没有锁 | 用 CAS 直接提交修改 |

一句话：

```text
CAS 既可以用来实现锁，也可以用来绕开锁。
```

---

## 3.4 无锁链表插入

### 3.4.1 普通链表插入

```c
void insert(int value) {
    node_t *n = malloc(sizeof(node_t));
    assert(n != NULL);
    n->value = value;
    n->next = head;
    head = n;
}
```

单线程中没问题。

但多线程中可能丢节点。

假设初始链表：

```text
head -> A
```

线程 1 插入 X，线程 2 插入 Y：

```text
线程 1: X->next = A
线程 2: Y->next = A
线程 1: head = X
线程 2: head = Y
```

最终：

```text
head -> Y -> A
```

节点 X 丢失。

---

### 3.4.2 用锁修复

```c
void insert(int value) {
    node_t *n = malloc(sizeof(node_t));
    assert(n != NULL);
    n->value = value;

    lock(listlock);
    n->next = head;
    head = n;
    unlock(listlock);
}
```

真正需要保护的是：

```c
n->next = head;
head = n;
```

因为这两行访问共享链表头 `head`。

前面的：

```c
node_t *n = malloc(sizeof(node_t));
n->value = value;
```

只操作当前线程私有的新节点，不需要加锁。

原则：

```text
锁只保护共享数据，不保护线程私有数据。
```

---

### 3.4.3 为什么不一进入 insert 就加锁

因为新节点 `n` 在执行：

```c
head = n;
```

之前，还没有被发布到共享链表中，其他线程无法访问它。

所以这些操作是线程私有的：

```c
node_t *n = malloc(sizeof(node_t));
assert(n != NULL);
n->value = value;
```

可以放在锁外执行。

好处：

```text
缩短临界区；
减少锁竞争；
提高并发性能。
```

---

### 3.4.4 CAS 实现无锁插入

```c
void insert(int value) {
    node_t *n = malloc(sizeof(node_t));
    assert(n != NULL);
    n->value = value;
    do {
        n->next = head;
    } while (CompareAndSwap(&head, n->next, n) == 0);
}
```

含义：

```text
先让 n->next 指向当前 head；
尝试把 head 从旧值改成 n；
如果成功，插入完成；
如果失败，说明 head 已经被别人改过，重新读取 head 再试。
```

假设初始链表：

```text
head -> A
```

线程 1 插入 X：

```text
X->next = A
CAS(&head, A, X)
```

线程 2 插入 Y：

```text
Y->next = A
CAS(&head, A, Y)
```

如果线程 1 先成功：

```text
head -> X -> A
```

线程 2 的 CAS 失败，因为它期望 `head == A`，但实际 `head == X`。

线程 2 重新执行：

```text
Y->next = X
CAS(&head, X, Y)
```

最终：

```text
head -> Y -> X -> A
```

两个节点都插入成功。

---

### 3.4.5 CAS 不是万能的

CAS 可以避免传统锁死锁，但会带来新问题：

```text
ABA 问题
内存回收困难
活锁
算法复杂
正确性难证明
```

---

# 4. 基于事件的并发

## 4.1 为什么需要事件驱动并发

传统并发通常依赖多线程：

```text
一个任务一个线程。
```

这种方式有两个主要问题：

```text
1. 多线程正确性难保证，容易出现锁错误、死锁、原子性问题、顺序问题。
2. 线程调度由操作系统控制，程序员无法精确控制线程何时运行。
```

事件驱动并发提供另一种方式：

```text
不用一个连接一个线程；
而是用一个事件循环管理多个事件。
```

常见应用：

```text
GUI 应用
网络服务器
Node.js
Nginx
Redis
```

---

## 4.2 并发和并行的区别

### 并发

```text
一个系统同时管理多个任务。
```

### 并行

```text
多个任务真的在多个 CPU 核上同时执行。
```

### 单线程事件驱动

```text
有并发，但没有 CPU 并行。
```

它可以管理大量连接，但同一时刻只有一个事件处理函数在 CPU 上运行。

---

## 4.3 事件循环

事件循环基本形式：

```c
while (1) {
    events = get_events();

    for each event in events {
        handle_event(event);
    }
}
```

网络服务器中的事件可能是：

```text
有新连接到达
某个 socket 可读
某个 socket 可写
连接断开
定时器到期
```

事件驱动服务器不为每个连接创建线程，而是：

```text
哪个连接准备好了，就处理哪个；
没准备好的连接，不阻塞等待。
```

---

## 4.4 多 CPU 下事件驱动怎么扩展

纯单线程事件循环只能使用一个 CPU 核。

在多核系统中，常见做法是：

```text
多个进程或多个线程，每个都有一个事件循环。
```

例如 8 核机器：

```text
CPU0: event loop 1
CPU1: event loop 2
CPU2: event loop 3
...
CPU7: event loop 8
```

每个事件循环负责一部分连接。

这样：

```text
每个 event loop 内部仍然是单线程；
多个 event loop 之间可以并行。
```

事件驱动不是说整个系统永远只有一个线程，而是：

```text
不采用一个连接一个线程的模型。
```

---

# 5. 使用 select/poll 接收事件

## 5.1 文件描述符

在 UNIX/Linux 中，很多对象都可以用文件描述符表示：

```text
普通文件
网络 socket
管道
终端输入
```

例如 Web 服务器有多个客户端连接：

```text
客户端 A -> fd 4
客户端 B -> fd 5
客户端 C -> fd 6
客户端 D -> fd 7
```

服务器需要知道：

```text
哪个 fd 有数据可读？
哪个 fd 可以写数据？
哪个 fd 出错了？
```

---

## 5.2 select 的作用

`select()` 的作用是：

```text
给操作系统一组 fd，
让操作系统告诉你哪些 fd 已经准备好了。
```

函数形式：

```c
int select(int nfds,
           fd_set *restrict readfds,
           fd_set *restrict writefds,
           fd_set *restrict errorfds,
           struct timeval *restrict timeout);
```

---

## 5.3 select 参数解释

### nfds

```c
int nfds
```

表示检查的 fd 范围：

```text
检查 0 到 nfds - 1 的 fd。
```

如果最大 fd 是 7，则通常传：

```c
select(8, ...);
```

也就是最大 fd 加 1。

---

### readfds

```c
fd_set *readfds
```

表示关心哪些 fd 是否可以读。

对于网络服务器：

```text
socket 可读通常表示客户端发来了数据。
```

---

### writefds

```c
fd_set *writefds
```

表示关心哪些 fd 是否可以写。

```text
socket 可写表示发送缓冲区有空间，可以继续发送数据。
```

---

### errorfds

```c
fd_set *errorfds
```

表示关心哪些 fd 是否出现异常。

---

### timeout

```c
struct timeval *timeout
```

表示最多等待多久。

常见情况：

```c
timeout = NULL;
```

表示一直阻塞，直到有 fd 准备好。

```c
timeout = 0;
```

表示立即返回，不等待。

指定时间值：

```text
最多等待这么久，期间有 fd 准备好就提前返回。
```

---

## 5.4 使用 select 的事件循环

示例代码：

```c
#include <stdio.h>
#include <stdlib.h>
#include <sys/time.h>
#include <sys/types.h>
#include <unistd.h>

int main(void) {
    // open and set up a bunch of sockets (not shown)

    while (1) {
        fd_set readFDs;
        FD_ZERO(&readFDs);

        int fd;
        for (fd = minFD; fd < maxFD; fd++)
            FD_SET(fd, &readFDs);

        int rc = select(maxFD + 1, &readFDs, NULL, NULL, NULL);

        for (fd = minFD; fd < maxFD; fd++)
            if (FD_ISSET(fd, &readFDs))
                processFD(fd);
    }
}
```

### 逐行解释

```c
fd_set readFDs;
```

定义一个 fd 集合。

```c
FD_ZERO(&readFDs);
```

清空集合。

```c
for (fd = minFD; fd < maxFD; fd++)
    FD_SET(fd, &readFDs);
```

把服务器关心的 fd 加入集合。

```c
int rc = select(maxFD + 1, &readFDs, NULL, NULL, NULL);
```

调用 `select()`，等待这些 fd 中某些变为可读。

```c
if (FD_ISSET(fd, &readFDs))
    processFD(fd);
```

检查某个 fd 是否真的可读，如果可读就处理它。

---

### 为什么每轮都要重新 FD_ZERO 和 FD_SET

因为 `select()` 会修改传入的 fd 集合。

调用前：

```text
readFDs = {4, 5, 6, 7}
```

如果只有 fd 5 可读，调用后：

```text
readFDs = {5}
```

所以下一轮循环必须重新构造完整集合。

准确理解：

```text
每次调用 select() 之前，readFDs 是所有关心的 fd 集合；
每次 select() 返回之后，readFDs 变成就绪 fd 的子集。
```

---

# 6. 异步 I/O

## 6.1 为什么需要异步 I/O

`select()` / `poll()` 主要适合网络 socket。

但是事件驱动服务器还可能需要磁盘 I/O。

例如 Web 服务器收到请求：

```text
GET /index.html
```

需要从磁盘读取文件。

普通磁盘 `read()` 可能阻塞：

```c
read(fd, buffer, size);
```

如果文件不在内存缓存中，需要从磁盘读取，事件循环就会被卡住。

事件驱动最怕：

```text
单线程事件循环中出现阻塞调用。
```

因此需要异步 I/O。

---

## 6.2 aiocb 结构

异步 I/O 使用 AIO 控制块：

```c
struct aiocb {
    int aio_fildes;         /* File descriptor */
    off_t aio_offset;       /* File offset */
    volatile void *aio_buf; /* Location of buffer */
    size_t aio_nbytes;      /* Length of transfer */
};
```

可以理解为：

```text
一次异步 I/O 请求的说明书。
```

字段含义：

```c
aio_fildes
```

要读写的文件描述符。

```c
aio_offset
```

文件中的偏移量。

```c
aio_buf
```

读出来的数据放到哪里。

```c
aio_nbytes
```

要读写多少字节。

---

## 6.3 aio_read 和 aio_error

### aio_read

```c
int aio_read(struct aiocb *aiocbp);
```

作用：

```text
发起一次异步读取。
```

如果成功，它会立即返回，不等待 I/O 完成。

事件循环可以继续处理其他事件。

---

### aio_error

```c
int aio_error(const struct aiocb *aiocbp);
```

作用：

```text
检查异步 I/O 请求是否完成。
```

返回值：

```text
0: I/O 已完成
EINPROGRESS: I/O 仍在进行中
```

示例：

```c
if (aio_error(&cb) == 0) {
    // I/O 完成，可以使用 aio_buf 中的数据
} else {
    // I/O 还没完成，继续处理其他事件
}
```

---

## 6.4 信号通知

如果程序发出很多异步 I/O，请求可能有几十个或几百个。

如果每轮都检查：

```text
请求 1 完成了吗？
请求 2 完成了吗？
请求 3 完成了吗？
...
请求 100 完成了吗？
```

会很麻烦。

这叫轮询。

为了解决这个问题，有些系统可以在 I/O 完成时使用 UNIX 信号通知进程。

---

### UNIX 信号

信号是一种通知进程的机制。

常见信号：

```text
SIGSEGV: 段错误
SIGINT: 中断，通常是 Ctrl+C
SIGHUP: 挂断信号
```

示例代码：

```c
#include <stdio.h>
#include <signal.h>

void handle(int arg) {
    printf("stop wakin' me up...\n");
}

int main(int argc, char *argv[]) {
    signal(SIGHUP, handle);

    while (1)
        ;

    return 0;
}
```

含义：

```text
当程序收到 SIGHUP 信号时，执行 handle()。
```

命令：

```bash
kill -HUP 36705
```

表示给进程号为 `36705` 的进程发送 `SIGHUP` 信号。

注意：

```text
kill 命令本质是发送信号，不一定是杀死进程。
```

在真实异步 I/O 中，不需要用户手动输入命令。

真实流程是：

```text
程序提交 aio_read()
操作系统执行磁盘 I/O
磁盘 I/O 完成
内核自动给进程发送信号
进程执行信号处理函数
程序知道 I/O 完成
```

书中手动输入 `kill` 命令只是为了演示信号机制。

---

# 7. 事件驱动的困难

事件驱动解决了多线程的一些问题，但也带来了新的复杂性。

---

## 7.1 手工栈管理

线程模型中代码可以写得很自然：

```c
int rc = read(fd, buffer, size);
rc = write(sd, buffer, size);
```

含义：

```text
先从文件 fd 读取数据到 buffer；
读完后，把 buffer 写到 socket sd。
```

在线程模型中，如果 `read()` 阻塞，当前线程会停在那里。

等 `read()` 返回后，代码继续执行下一行：

```c
write(sd, buffer, size);
```

此时程序还知道 `sd` 是什么，因为它在线程栈中。

线程栈保存了：

```text
局部变量
函数调用关系
返回地址
下一步执行位置
```

---

### 事件驱动中的问题

事件驱动不能阻塞等待 `read()`。

所以它要这样做：

```text
发起异步 read
立即返回事件循环
之后 I/O 完成时再继续执行 write
```

问题是：

```text
异步 read 完成时，程序怎么知道应该写到哪个 socket sd？
```

原来的事件处理函数早就返回了，局部变量可能已经不存在。

所以必须手动保存状态。

---

### 延续 continuation

延续可以理解为：

```text
记录某个异步操作完成之后，下一步该做什么。
```

例如发起异步读时，保存：

```text
文件 fd
socket sd
buffer
size
下一步动作 write(sd, buffer, size)
```

可以用结构体表示：

```c
struct continuation {
    int fd;
    int sd;
    char *buffer;
    int size;
};
```

也可以用散列表保存：

```text
fd -> continuation
```

当磁盘 I/O 完成后：

```text
用 fd 找到 continuation；
取出 sd；
把 buffer 写入 sd。
```

这就是手工栈管理。

一句话：

```text
线程模型靠线程栈自动保存状态；
事件模型要程序员自己保存状态。
```

---

## 7.2 多核下同步问题重新出现

单线程事件驱动的好处是：

```text
同一时刻只有一个事件处理程序运行；
普通共享数据不需要加锁。
```

但是多核系统中，如果为了利用多个 CPU，让多个事件处理程序并行运行：

```text
CPU0: 处理事件 A
CPU1: 处理事件 B
CPU2: 处理事件 C
```

那么它们可能同时访问共享数据。

于是又会出现：

```text
数据竞争
临界区
锁
死锁
原子性问题
顺序问题
```

所以在多核系统上，事件驱动的无锁简单性会下降。

---

## 7.3 隐式阻塞

事件驱动可以避免显式阻塞调用，例如阻塞 `read()`。

但有些阻塞是隐式的，比如页错误。

### 页错误 page fault

如果程序访问某个虚拟地址，而对应物理页不在内存中，操作系统需要从磁盘把页面读回来。

这时当前线程会阻塞。

对于单线程事件循环来说：

```text
事件处理程序发生页错误；
整个事件循环被阻塞；
其他客户端无法被处理。
```

因此：

```text
显式阻塞容易避免；
隐式阻塞很难完全避免。
```

---

## 7.4 API 语义变化问题

事件驱动要求：

```text
事件处理函数不能调用阻塞 API。
```

如果某个函数原来是非阻塞的：

```c
foo();
```

事件处理函数可以直接调用它。

但如果以后 `foo()` 的实现变了，变成可能阻塞：

```text
handle_event() 调用 foo()
foo() 阻塞
整个事件循环卡住
```

这会让事件驱动代码很难维护。

程序员必须始终清楚：

```text
每个 API 到底会不会阻塞。
```

---

## 7.5 网络 I/O 和磁盘 I/O 难以统一

理想情况下，我们希望：

```text
select() 同时管理网络 I/O 和磁盘 I/O。
```

例如：

```text
socket 可读
socket 可写
磁盘读完成
磁盘写完成
```

都由一个统一接口返回。

但现实中通常是：

```text
网络 I/O 用 select/poll/epoll/kqueue；
磁盘 I/O 用 aio_read/aio_error 或线程池。
```

于是服务器可能需要同时管理多套机制：

```text
select() 管网络事件；
aio_error() 或信号管磁盘事件；
线程池处理阻塞磁盘 I/O。
```

这增加了复杂性。

---

# 8. 总结

## 8.1 非死锁缺陷

```text
违反原子性：本来应该一起执行的操作，中间被其他线程插入。
违反顺序：本来应该先后执行的操作，实际顺序被打乱。
```

修复方法：

```text
违反原子性：用锁保护临界区。
违反顺序：用条件变量或信号量强制顺序。
```

---

## 8.2 死锁

死锁的典型形式：

```text
Thread 1 持有 L1，等待 L2
Thread 2 持有 L2，等待 L1
```

死锁 4 个条件：

```text
互斥
持有并等待
非抢占
循环等待
```

预防方法：

```text
统一加锁顺序
一次性获取所有锁
trylock 失败就释放已有锁
避免互斥，使用无锁结构
```

---

## 8.3 CAS

CAS 是底层原子指令。

它可以：

```text
实现自旋锁
直接实现无锁数据结构
```

区别：

```text
CAS 操作锁变量：实现锁。
CAS 操作共享数据：绕开锁。
```

---

## 8.4 事件驱动

事件驱动的核心是：

```text
一个事件循环管理多个 I/O 事件。
```

适合：

```text
大量连接
I/O 密集
请求处理较短
等待时间多
```

不适合：

```text
单线程执行大量 CPU 计算
调用阻塞 API
复杂异步状态管理
```

# 9. 易错点

## 9.1 判断空指针后再使用，不一定安全

```c
if (p != NULL) {
    use(p);
}
```

如果其他线程能在中间修改 `p`，仍然不安全。

---

## 9.2 只给一部分代码加锁不够

访问同一共享变量的所有代码都必须遵守同一个加锁规则。

---

## 9.3 条件变量要配合 while

错误写法：

```c
if (mtInit == 0)
    pthread_cond_wait(&mtCond, &mtLock);
```

正确写法：

```c
while (mtInit == 0)
    pthread_cond_wait(&mtCond, &mtLock);
```

原因：

```text
防止虚假唤醒；
防止醒来后条件又不满足。
```

---

## 9.4 CAS 不等于锁

CAS 是原子指令。

它既可以用来实现锁，也可以用来实现无锁算法。

---

## 9.5 单线程事件驱动不是并行

单线程事件循环可以管理很多连接，但同一时刻只在一个 CPU 核上执行。

---

## 9.6 事件驱动不是永远不用线程

现实系统常常是混合模型：

```text
事件循环处理网络 I/O；
线程池处理阻塞 I/O；
多个 worker 进程利用多核。
```

---

## 9.7 select 会修改 fd_set

调用前：

```text
readFDs = 所有关心的 fd
```

调用后：

```text
readFDs = 已经就绪的 fd 子集
```

所以每轮循环都要重新 `FD_ZERO()` 和 `FD_SET()`。

---

## 9.8 异步 I/O 完成通知不需要人手动 kill

书中的 `kill -HUP` 只是演示信号机制。

真实异步 I/O 中，I/O 完成后由内核自动通知进程。

---

## 9.9 事件驱动代码复杂的根源是状态拆分

线程模型：

```text
线程栈自动保存状态。
```

事件模型：

```text
程序员手动保存 continuation。
```

---

# 10. 简短记忆版

```text
违反原子性：
    我刚检查完，别人就改了。

违反顺序：
    我还没准备好，别人就用了。

死锁：
    你拿着我想要的，我拿着你想要的，大家都不放。

死锁四条件：
    互斥、持有并等待、非抢占、循环等待。

避免死锁：
    统一加锁顺序，或者拿不到就释放，或者不用锁。

CAS：
    如果值还是我看到的旧值，就把它改成新值；
    如果别人改过了，就失败并重试。

事件驱动：
    不给每个连接创建线程；
    一个事件循环管理很多 I/O 事件。

select：
    给它一组 fd，它返回哪些 fd 已经可读、可写或出错。

异步 I/O：
    提交 I/O 后立即返回，完成后再通知或检查。

事件驱动的难点：
    线程栈没了，程序员要自己保存下一步该干什么。
```
