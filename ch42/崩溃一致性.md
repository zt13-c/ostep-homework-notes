# 文件系统崩溃一致性与日志机制笔记

## 1. 崩溃一致性问题

### 1.1 什么是崩溃一致性问题

文件系统更新一个文件时，通常不是只修改一个地方，而是要同时修改多个磁盘结构。

例如，向一个文件追加一个数据块，可能需要修改：

```text
1. 新数据块 Db
   写入真正的用户数据。

2. inode I[v2]
   更新文件大小，并让 inode 指向新的数据块。

3. 数据块位图 B[v2]
   标记这个数据块已经被占用。
```

理想情况下，这些更新应该具有原子性：

```text
要么全部完成；
要么全部不完成。
```

但是磁盘写入不是天然原子的。系统可能在多个写入之间崩溃，例如断电、内核崩溃等。

因此可能出现：

```text
inode 已经更新了，但 bitmap 没更新；
bitmap 已经更新了，但 inode 没更新；
inode 指向了数据块，但数据块还没正确写入。
```

这就是崩溃一致性问题。

也叫：

```text
crash-consistency problem
consistent-update problem
```

---

### 1.2 不一致的几种例子

#### 情况一：inode 更新了，bitmap 没更新

```text
inode 说：文件正在使用 block 1000
bitmap 说：block 1000 是空闲的
```

后果：

```text
文件系统以后可能把 block 1000 分配给另一个文件，
导致两个文件共享同一个数据块，数据会被破坏。
```

---

#### 情况二：bitmap 更新了，inode 没更新

```text
bitmap 说：block 1000 已经被占用
inode 说：没有文件指向 block 1000
```

后果：

```text
block 1000 明明没人用，
但文件系统认为它被占用了。
```

这叫：

```text
空间泄露
space leak
```

---

#### 情况三：inode 指向了数据块，但数据块没写好

```text
inode 说：文件包含 block 1000
block 1000 中却是旧数据、垃圾数据，甚至别的文件残留数据
```

后果：

```text
用户读取文件时可能读到错误数据或旧数据。
```

---

### 1.3 一句话总结

```text
崩溃一致性问题 = 文件系统更新做到一半时崩溃，导致 inode、bitmap、数据块、目录项等结构之间状态不一致。
```

一个一致状态应该满足：

```text
inode 说文件使用某个块；
bitmap 也说这个块已占用；
数据块中确实有正确内容。
```

---

## 2. 日志 journaling / 预写日志 write-ahead logging

### 2.1 日志的基本思想

日志的核心思想是：

```text
在真正修改文件系统结构之前，
先把“接下来要做哪些修改”写到日志区。
```

也就是说，文件系统不会一开始就直接修改 inode、bitmap、数据块的最终位置，而是先写日志。

例如追加文件时，原本要修改：

```text
I[v2]   新 inode
B[v2]   新 bitmap
Db      新数据块
```

日志机制会先把这些更新记录到日志区。

---

### 2.2 为什么日志能解决崩溃一致性

如果系统崩溃，重启后文件系统不用扫描整个磁盘来猜测哪里坏了，而是只需要检查日志区。

恢复时大致逻辑是：

```text
如果日志事务没有完整提交：
    忽略它，文件系统保持旧状态。

如果日志事务已经完整提交：
    重放 redo 这个事务，把日志中的更新写回最终位置。
```

因此，文件系统可以避免停留在“只更新了一半”的状态。

---

### 2.3 ext2 与 ext3 的布局区别

ext2 没有日志，简化布局类似：

```text
Super | Group 0 | Group 1 | ... | Group N
```

ext3 增加了日志区，简化布局类似：

```text
Super | Journal | Group 0 | Group 1 | ... | Group N
```

真正的区别不只是多了一个 Journal 区，而是更新文件系统时：

```text
先写日志；
再写最终位置。
```

---

## 3. 关于超级块 superblock

### 3.1 图里的 Super 是不是只有一个

书中的图通常是简化图，只画了一个 `Super`。

但真实 ext2/ext3/ext4 文件系统中，通常有：

```text
1. 主超级块 primary superblock
2. 一些超级块副本 backup superblocks
```

超级块副本用于主超级块损坏时恢复文件系统。

---

### 3.2 每个块组都有超级块副本吗

不一定。

早期设计可能很多块组都有超级块副本，但现代 ext 文件系统通常使用稀疏超级块设计：

```text
不是每个块组都有超级块副本；
只有部分块组保存副本。
```

所以可以这样理解：

```text
图中的 Super 是简化画法；
真实文件系统中有主超级块，也可能在部分块组中有超级块副本。
```

---

## 4. 数据日志 data journaling

### 4.1 数据日志记录什么

数据日志会把元数据和普通用户数据都写入日志。

一次追加文件可能涉及：

```text
I[v2]   inode 的新版本
B[v2]   bitmap 的新版本
Db      普通用户数据块
```

日志区中会写：

```text
TxB | I[v2] | B[v2] | Db | TxE
```

其中：

```text
TxB = Transaction Begin，事务开始块
TxE = Transaction End，事务结束块
```

---

### 4.2 TxB 和 TxE 的作用

`TxB` 表示事务开始，记录这次事务有哪些更新。

`TxE` 表示事务结束，也就是提交标志。

恢复时，文件系统会检查：

```text
有没有 TxB；
有没有 TxE；
事务编号是否匹配。
```

如果 TxB 和 TxE 都存在且匹配，说明事务已经提交。

如果只有 TxB，没有 TxE，说明事务没有完整提交，恢复时忽略。

---

### 4.3 物理日志 physical logging

书中例子属于物理日志。

物理日志记录的是要写入磁盘的真实块内容。

例如：

```text
把 I[v2] 写到 inode 表中的某个位置；
把 B[v2] 写到 bitmap 的某个位置；
把 Db 写到文件数据块的最终位置。
```

恢复时只需要把日志中的块复制到对应的最终位置。

---

### 4.4 逻辑日志 logical logging

逻辑日志不直接记录完整块内容，而是记录操作语义。

例如：

```text
在目录 / 中添加一个目录项 foo，指向 inode 12。
```

逻辑日志的优点：

```text
日志占用空间较小。
```

缺点：

```text
恢复逻辑更复杂。
```

---

## 5. checkpoint 检查点

### 5.1 什么是 checkpoint

日志区只是临时记录本，最终文件系统结构还是要被更新。

checkpoint 就是：

```text
把日志中的更新内容写回它们在文件系统中的最终位置。
```

例如：

```text
I[v2] -> inode 表中的最终位置
B[v2] -> bitmap 的最终位置
Db    -> 文件数据块的最终位置
```

---

### 5.2 数据日志的基本流程

数据日志的基本流程是：

```text
1. 日志写入
   把 TxB、I[v2]、B[v2]、Db 写入日志。

2. 日志提交
   把 TxE 写入日志。
   事务被认为 committed。

3. 检查点 checkpoint
   把 I[v2]、B[v2]、Db 写回最终磁盘位置。

4. 释放日志
   事务已经不再需要，日志空间可以复用。
```

可以记成：

```text
日志写入 -> 日志提交 -> 检查点 -> 释放
```

---

## 6. 为什么 TxE 必须小心写入

### 6.1 原始问题

如果文件系统想写入：

```text
TxB | I[v2] | B[v2] | Db | TxE
```

但磁盘可能因为缓存和调度重排写入顺序。

文件系统发出的顺序可能是：

```text
TxB -> I[v2] -> B[v2] -> Db -> TxE
```

但磁盘真实落盘顺序可能变成：

```text
TxB -> I[v2] -> B[v2] -> TxE
```

此时 `Db` 还没有写好。

崩溃后日志区可能是：

```text
TxB | I[v2] | B[v2] | ?? | TxE
```

这里的 `??` 可能是旧数据或无效数据。

---

### 6.2 为什么危险

恢复程序看到：

```text
TxB 存在；
TxE 存在；
事务编号匹配。
```

就可能认为事务已经完整提交，然后重放：

```text
I[v2] -> inode 表
B[v2] -> bitmap
??    -> Db 的最终位置
```

结果会把错误数据写回文件系统。

---

### 6.3 安全的三阶段协议

为了避免 TxE 过早落盘，安全协议是：

```text
1. 先写 TxB、I[v2]、B[v2]、Db
2. 确保这些内容都写完
3. 再写 TxE
```

关键原则：

```text
TxE 必须最后安全落盘。
```

因为 TxE 的含义是：

```text
前面的日志内容已经完整，可以恢复时使用。
```

---

## 7. 写入屏障 write barrier

### 7.1 写入屏障解决什么问题

写入屏障用于控制磁盘写入顺序。

可以理解成：

```text
屏障之前的写入必须先完成；
屏障之后的写入不能提前越过去。
```

日志协议中可以这样理解：

```text
TxB | I[v2] | B[v2] | Db
------------------------- barrier
TxE
```

这保证：

```text
日志内容先落盘；
TxE 后落盘。
```

---

### 7.2 写入屏障的缺点

写入屏障会引入等待。

在机械硬盘上，这可能导致额外旋转延迟：

```text
本来可以连续写一整段；
现在必须写一段，等待完成，再写 TxE。
```

在 SSD 上也可能带来 flush 开销。

---

## 8. checksum 校验和优化

### 8.1 原始日志写入为什么慢

没有 checksum 时，必须分两轮写：

```text
第一轮：
TxB | I[v2] | B[v2] | Db

等待完成

第二轮：
TxE
```

原因是必须保证 TxE 不会比日志内容更早有效落盘。

---

### 8.2 checksum 的思想

优化方法是在事务开始块和结束块中加入整个事务内容的 checksum。

事务变成：

```text
TxB(checksum) | I[v2] | B[v2] | Db | TxE(checksum)
```

checksum 根据事务内容计算，例如：

```text
checksum(I[v2], B[v2], Db)
```

---

### 8.3 有 checksum 后为什么可以一次性写出整个事务

有 checksum 后，文件系统可以把整个事务一次性发给磁盘：

```text
TxB(checksum) | I[v2] | B[v2] | Db | TxE(checksum)
```

即使崩溃导致日志变成：

```text
TxB(checksum) | I[v2] | B[v2] | ?? | TxE(checksum)
```

恢复时重新计算：

```text
checksum(I[v2], B[v2], ??)
```

如果和 TxB/TxE 中保存的 checksum 不匹配，就说明事务不完整，恢复程序会丢弃这个事务。

---

### 8.4 checksum 和写入屏障的关系

二者解决的是同一个安全问题：

```text
不能让 TxE 表示的事务在日志内容未完整写入时被误认为有效。
```

但方法不同：

```text
write barrier:
    写的时候强制顺序。
    不允许 TxE 提前落盘。

checksum:
    允许乱序落盘。
    但恢复时检测事务是否完整。
```

可以记成：

```text
write barrier = 预防错误
checksum      = 检测错误
```

或者：

```text
write barrier = 写入时保证顺序
checksum      = 恢复时验证完整性
```

---

### 8.5 checksum 是否完全替代 barrier

在教材语境中，checksum 减少了“日志内容和 TxE 之间”的强制等待。

但真实系统中，barrier/flush 仍可能在其他地方需要，例如：

```text
1. 保证提交记录最终到达稳定存储；
2. 保证 checkpoint 和日志释放之间的顺序；
3. 处理磁盘缓存和控制器缓存带来的问题。
```

所以更准确地说：

```text
checksum 减少了对某些写入屏障的依赖，
但不代表所有屏障都不需要了。
```

## 9. 崩溃恢复 recover

### 9.1 恢复的基本规则

恢复时，文件系统扫描日志。

如果事务没有完整提交：

```text
没有 TxE
```

则忽略。

如果事务已经提交：

```text
有 TxB 和 TxE，且编号匹配
```

则重放 redo。

---

### 9.2 redo logging 重做日志

这里的日志是 redo logging。

意思是：

```text
只要事务已经提交，崩溃后就重新做一遍。
```

例如日志中有：

```text
TxB | I[v2] | B[v2] | Db | TxE
```

恢复时重放：

```text
I[v2] -> inode 表
B[v2] -> bitmap
Db    -> 数据块最终位置
```

---

### 9.3 checkpoint 过程中崩溃为什么没问题

如果 checkpoint 做到一半崩溃，例如只写回了 inode，还没写回 bitmap 和数据块。

恢复时日志仍然存在，系统可以再次重放。

重放操作通常是幂等的：

```text
把 inode 改成 I[v2]
```

执行一次是 I[v2]，执行多次仍然是 I[v2]。

所以重复重放不会造成问题。

---

## 10. 批处理日志更新

### 10.1 为什么需要批处理

如果每个小操作都单独写一次日志，开销很大。

例如在同一目录下连续创建两个文件：

```text
file1
file2
```

如果分别提交两个事务：

```text
Tx1: 创建 file1
Tx2: 创建 file2
```

每个事务都要经历：

```text
日志写入 -> 日志提交 -> checkpoint
```

这会产生大量磁盘 I/O。

---

### 10.2 批处理的思想

文件系统可以把多个更新先缓存在内存中，过一小段时间一起组成一个全局事务提交。

例如创建两个文件涉及：

```text
inode bitmap
file1 inode
file2 inode
父目录数据块
父目录 inode
```

可以合并为一个事务。

优点：

```text
1. 减少日志提交次数；
2. 减少磁盘寻道和旋转；
3. 避免同一个块被反复写入日志。
```

---

## 11. 日志空间有限与循环日志

### 11.1 日志为什么会满

日志区是磁盘上的固定区域，不可能无限增长。

如果一直追加事务：

```text
Tx1 | Tx2 | Tx3 | Tx4 | Tx5 | ...
```

迟早会写满。

日志满了会带来两个问题：

```text
1. 恢复时间变长；
2. 新事务没有地方写。
```

---

### 11.2 循环日志 circular log

解决办法是把日志当作循环数组使用。

例如：

```text
Journal:
[空闲][Tx1][Tx2][Tx3][空闲][空闲]
       ↑             ↑
     oldest        newest
```

日志写到末尾后，可以绕回开头继续写。

前提是：

```text
被覆盖的旧事务已经 checkpoint 完成，不再需要恢复。
```

---

## 12. Journal Superblock 日志超级块

### 12.1 Journal Super 是什么

Journal Super 不是主文件系统的 superblock，而是日志区自己的管理信息。

它记录日志区的状态，例如：

```text
1. 最旧的有效事务在哪里；
2. 最新的有效事务在哪里；
3. 哪些日志区域还需要恢复；
4. 哪些日志区域可以复用。
```

---

### 12.2 “只需标记最旧和最新事务”是什么意思

日志超级块不需要记录每一个日志块是空闲还是占用。

它只需要记录：

```text
oldest = 最旧的有效事务位置
newest = 最新的有效事务位置
```

例如：

```text
Journal:
[空闲][Tx1][Tx2][Tx3][空闲][空闲]
       ↑             ↑
     oldest        newest
```

含义是：

```text
从 Tx1 到 Tx3 这一段是有效日志；
其他位置都是空闲，可以复用。
```

---

### 12.3 循环情况下的 oldest 和 newest

如果日志写到末尾又绕回开头，可能是：

```text
Journal:
[Tx5][空闲][空闲][Tx3][Tx4]
 ↑                 ↑
newest           oldest
```

有效日志范围是：

```text
Tx3 -> Tx4 -> Tx5
```

空闲空间是：

```text
Tx5 后面到 Tx3 前面的区域。
```

---

### 12.4 日志写入前是否要修改 Journal Super 标记非空闲

通常不需要在日志写入前就把 Journal Super 标记为非空闲。

原因是：

```text
未提交事务没有 TxE；
即使写到日志区，崩溃恢复时也会被忽略。
```

事务真正变得有效的关键是：

```text
TxE 成功落盘。
```

Journal Super 更重要的作用是：

```text
1. 记录有效日志范围；
2. checkpoint 完成后释放旧事务；
3. 支持日志空间循环复用。
```

---

## 13. 元数据日志 metadata journaling

### 13.1 为什么需要元数据日志

数据日志虽然安全，但效率低。

原因是普通用户数据也会写两次：

```text
第一次：Db -> Journal
第二次：Db -> 最终数据块位置
```

如果用户写入大量数据，日志 I/O 会大幅增加。

---

### 13.2 元数据日志的思想

元数据日志只记录文件系统元数据，不记录普通用户数据。

例如追加文件时，日志只写：

```text
TxB | I[v2] | B[v2] | TxE
```

不写：

```text
Db
```

其中：

```text
I[v2] = inode 新版本
B[v2] = bitmap 新版本
Db    = 普通用户数据块
```

---

### 13.3 什么是元数据

元数据是描述文件系统结构的数据，例如：

```text
inode
bitmap
目录项
超级块
块组描述符
inode 表
目录数据块
```

普通用户数据是文件真正内容，例如：

```text
文本内容
图片内容
视频内容
代码内容
```

注意：

```text
目录本身也是文件；
但目录文件的内容是目录项；
因此目录数据块属于元数据。
```

---

### 13.4 元数据日志的问题

如果只记录元数据，而不记录 Db，就可能出现：

```text
inode 已经指向 Db；
bitmap 也标记 Db 已占用；
但 Db 还没正确写入。
```

从元数据角度看：

```text
inode 和 bitmap 是一致的。
```

但从用户数据角度看：

```text
文件内容可能是旧数据或垃圾数据。
```

---

## 14. Ordered Journaling 有序元数据日志

### 14.1 为什么要先写 Db

为了避免 inode 指向未写好的数据块，元数据日志通常要求：

```text
先把用户数据 Db 写到最终位置；
再提交相关元数据日志。
```

关键顺序是：

```text
Db 落盘 < TxE 落盘
```

也就是：

```text
数据必须先于相关元数据事务提交。
```

---

### 14.2 有序元数据日志的流程

完整流程是：

```text
1. 数据写入
   Db -> 最终数据块位置

2. 日志元数据写入
   TxB | I[v2] | B[v2]

3. 日志提交
   TxE

4. checkpoint 元数据
   I[v2]、B[v2] -> 最终位置

5. 释放日志
```

---

### 14.3 是否必须等 Db 写完后才能写元数据日志内容

不一定。

真正要求是：

```text
Db 必须在 TxE 提交之前完成。
```

也就是说，可以并行发出：

```text
Db -> 最终位置
TxB、I[v2]、B[v2] -> 日志区
```

但必须保证：

```text
TxE 不能在 Db 落盘前完成。
```

---

### 14.4 “先写被指对象，再写指针对象”

这里的规则可以记成：

```text
先写被指向的数据；
再提交指向它的元数据。
```

例如：

```text
Db 是被指对象；
inode 中的数据块指针是指针对象。
```

如果 inode 先提交，而 Db 没写好，就会导致文件指向错误数据。

---

## 15. Db 最终位置是怎么知道的

### 15.1 先在内存中完成块分配

在写 Db 之前，文件系统已经在内存中完成了块分配。

例如：

```text
用户追加文件
    ↓
文件系统查内存中的 bitmap
    ↓
找到空闲块 10086
    ↓
决定 Db 最终写到 block 10086
```

所以 `Db -> 最终位置` 中的“最终位置”是文件系统提前分配好的物理块号。

---

### 15.2 磁盘上的 bitmap 还没改怎么办

磁盘上的 bitmap 可能还没更新。

但内存中的 bitmap 已经被修改：

```text
磁盘 bitmap:
    block 10086 仍显示空闲

内存 bitmap:
    block 10086 已被当前事务占用
```

随后新的 bitmap 版本 `B[v2]` 会被写入日志并 checkpoint。

---

### 15.3 inode 也是先在内存中改好

例如原 inode：

```text
inode v1:
    size = 4096
    blocks = [20]
```

追加一个块后，内存中生成：

```text
inode v2:
    size = 8192
    blocks = [20, 10086]
```

然后 `I[v2]` 被写入元数据日志。

---

### 15.4 崩溃时的结果

如果 Db 写完，但元数据事务还没提交就崩溃：

```text
block 10086 中可能有新数据；
但 inode 没指向它；
bitmap 也没标记它已占用。
```

恢复后文件系统仍然一致，只是这次写入丢失。

以后 block 10086 可以重新分配并覆盖。

---

## 16. 块复用 block reuse 问题

### 16.1 块复用问题的背景

元数据日志中，目录数据属于元数据，会写入日志。

普通文件数据不属于元数据，通常不会写入日志。

这可能导致块复用问题。

---

### 16.2 例子：目录 foo 占用 block 1000

假设目录 `foo` 的目录数据块位于 block 1000。

修改目录 foo 后，日志中记录：

```text
Tx1:
TxB id=1 | I[foo] ptr:1000 | D[foo] final addr:1000 | TxE id=1
```

其中：

```text
I[foo] = foo 目录的 inode
D[foo] = foo 的目录数据块
```

因为 `foo` 是目录，所以 `D[foo]` 属于元数据，会进入日志。

---

### 16.3 删除 foo 后 block 1000 被释放

用户删除 foo 后，block 1000 被释放。

注意：

```text
释放块通常只是修改 bitmap；
不会立即把 block 1000 清零。
```

---

### 16.4 创建普通文件 foobar，复用 block 1000

之后用户创建新文件 `foobar`。

文件系统可能把刚释放的 block 1000 分配给 `foobar` 作为普通数据块。

简化日志可能表示为：

```text
Tx2:
TxB id=2 | I[foobar] ptr:1000 | TxE id=2
```

严格来说，真实创建文件还要记录更多元数据，见后文。

关键点是：

```text
foobar 的 inode 指向 block 1000；
但 foobar 的普通文件数据 D[foobar] 不写入元数据日志。
```

---

### 16.5 崩溃恢复时的问题

如果这时崩溃，恢复程序按顺序重放日志。

它可能先重放 Tx1：

```text
D[foo] -> block 1000
```

但 block 1000 现在已经属于 foobar。

然后重放 Tx2：

```text
I[foobar] -> inode 表
```

恢复后可能变成：

```text
foobar 的 inode 指向 block 1000；
block 1000 中却是旧目录 foo 的目录数据 D[foo]。
```

用户打开 foobar 时，会看到旧目录数据或垃圾内容。

这就是块复用问题。

---

### 16.6 块复用问题的本质

本质是：

```text
日志中记录的是物理块号；
但物理块的归属可能在之后发生改变。
```

旧日志说：

```text
把 D[foo] 写回 block 1000。
```

但后来：

```text
block 1000 已经释放并重新分配给 foobar。
```

恢复时如果仍然重放旧日志，就会污染新文件。

---

## 17. 为什么删除 foo 后旧事务不能立即释放

### 17.1 committed 不等于 checkpoint 完成

日志事务状态大致是：

```text
已提交到日志 committed
        ↓
已经 checkpoint 到最终位置
        ↓
可以从日志中释放
```

有 TxE 只说明：

```text
事务已提交，恢复时可以重放。
```

不说明：

```text
事务内容已经写回最终位置。
```

只有 checkpoint 完成后，事务才能安全释放。

---

### 17.2 删除 foo 不能自动证明 Tx1 已经 checkpoint

时间线可能是：

```text
1. 修改 foo
   Tx1 写入日志并提交，但还没 checkpoint。

2. 删除 foo
   删除操作生成新的事务。

3. 创建 foobar
   创建操作生成新的事务。
```

删除 foo 只是一个后续逻辑操作，不能证明 Tx1 的内容已经写回最终位置。

---

### 17.3 如果强制释放 Tx1 会怎样

如果 Tx1 还没 checkpoint 完成，却在 journal superblock 中把它标记为空闲，可能导致：

```text
系统崩溃后找不到 Tx1；
无法重放 Tx1；
Tx1 的更新可能丢失。
```

所以：

```text
不能随便释放已经提交但尚未 checkpoint 的事务。
```

---

### 17.4 为什么不强制删除前先 checkpoint Tx1

理论上可以，但性能差。

如果每次释放或复用块前都强制等待旧事务 checkpoint，会产生大量同步磁盘 I/O。

日志机制本来就是为了允许：

```text
先快速提交到日志；
之后批量 checkpoint。
```

所以实际系统更倾向于异步 checkpoint，并用其他机制避免块复用错误。

---

## 18. revoke 撤销记录

### 18.1 revoke 的作用

revoke 记录用于告诉恢复程序：

```text
不要再重放旧日志中对某个块的写入。
```

例如删除 foo 并释放 block 1000 时，可以写入：

```text
revoke block 1000
```

恢复时如果看到旧事务中有：

```text
D[foo] -> block 1000
```

但后面有 revoke 记录，则跳过这条重放。

---

### 18.2 revoke 解决了什么矛盾

一方面：

```text
旧事务 Tx1 还没 checkpoint，不能整体释放。
```

另一方面：

```text
Tx1 中针对 block 1000 的旧写入不能再重放，
否则会污染新文件 foobar。
```

revoke 的作用是：

```text
事务可以继续留在日志中；
但其中某些块的旧写回请求被撤销。
```

---

### 18.3 revoke 的记忆方式

可以把 revoke 理解为日志里的“撤销通知”。

原日志说：

```text
恢复时请把 D[foo] 写回 block 1000。
```

revoke 说：

```text
block 1000 已经释放或复用；
旧日志中关于 block 1000 的写回不要再执行。
```

---

## 19. 创建 foobar 时是否需要记录 inode bitmap

需要。

前面简化写成：

```text
TxB id=2 | I[foobar] ptr:1000 | TxE id=2
```

只是为了突出：

```text
foobar 的 inode 指向 block 1000；
但 foobar 的普通文件数据不写入元数据日志。
```

真实创建一个普通文件通常至少涉及：

```text
1. inode bitmap
   标记某个 inode 已分配给 foobar。

2. block bitmap
   标记 block 1000 已分配。

3. I[foobar]
   foobar 的 inode，记录大小、权限、数据块指针等。

4. 父目录数据块
   添加目录项：foobar -> inode number。

5. 父目录 inode
   更新时间戳、目录大小等。
```

更完整的 Tx2 类似：

```text
Tx2:
TxB id=2
Ibitmap[v2]
Bbitmap[v2]
I[foobar] ptr:1000
D[parent_dir]
I[parent_dir]
TxE id=2
```

但通常不包括：

```text
D[foobar]
```

因为 `D[foobar]` 是普通用户数据，不是元数据。

## 20. 数据日志、元数据日志、有序日志对比

### 20.1 数据日志 data journaling

日志记录：

```text
TxB | I[v2] | B[v2] | Db | TxE
```

特点：

```text
数据和元数据都写日志；
安全性强；
写入开销大；
普通用户数据会写两次。
```

---

### 20.2 元数据日志 metadata journaling

日志记录：

```text
TxB | I[v2] | B[v2] | TxE
```

特点：

```text
只记录元数据；
普通数据不写日志；
性能更好；
如果没有顺序约束，可能出现 inode 指向未写好的数据块。
```

---

### 20.3 有序日志 ordered journaling

流程：

```text
1. Db -> 最终位置
2. TxB | I[v2] | B[v2] -> 日志
3. TxE -> 日志
4. checkpoint 元数据
```

关键保证：

```text
Db 必须先于 TxE 落盘。
```

特点：

```text
数据只写一次；
元数据写两次；
兼顾性能和一致性；
很多实际文件系统采用类似思想。
```

---

## 21. 常见易错点

### 易错点 1：以为有 TxE 就一定所有内容都正确

没有 checksum 或屏障时，可能出现：

```text
TxB | I[v2] | B[v2] | ?? | TxE
```

所以必须通过写入顺序或 checksum 保证 TxE 的可信性。

---

### 易错点 2：把 Journal Super 和主 Superblock 混为一谈

```text
主 superblock:
    管理整个文件系统。

journal superblock:
    管理日志区，例如 oldest 和 newest。
```

二者不是同一个东西。

---

### 易错点 3：以为日志写入前必须更新 Journal Super

通常不是。

未提交事务没有 TxE，恢复时会忽略。

事务是否有效主要看：

```text
TxE 是否成功落盘。
```

---

### 易错点 4：以为 committed 就能释放日志

不能。

```text
committed = 已经写入完整日志，可以恢复时重放；
checkpointed = 已写回最终位置，可以释放日志。
```

只有 checkpoint 完成后，日志事务才能释放。

---

### 易错点 5：以为元数据日志不需要关心用户数据顺序

元数据日志不记录 Db，但必须保证：

```text
Db 落盘 < TxE 落盘
```

否则 inode 可能指向未写好的数据块。

---

### 易错点 6：以为删除文件会清空数据块

删除文件通常只是修改元数据，例如 inode、bitmap、目录项。

数据块内容可能仍然留在磁盘上，直到被覆盖。

---

### 易错点 7：以为块号永远属于同一个文件

物理块可能被释放并重新分配。

所以旧日志中记录的物理块写回可能污染新文件，这就是块复用问题。

---

## 22. 总结

文件系统崩溃一致性问题来自：

```text
一次文件更新需要修改多个磁盘结构；
崩溃可能发生在这些写入之间；
导致 inode、bitmap、数据块、目录项之间不一致。
```

日志机制的核心是：

```text
先写日志；
再写最终位置；
崩溃后根据日志恢复。
```

数据日志：

```text
记录元数据和用户数据；
安全但写入开销大。
```

元数据日志：

```text
只记录元数据；
性能更好；
但必须保证数据先于相关元数据提交。
```

循环日志通过 Journal Super 管理有效事务范围：

```text
oldest = 最旧有效事务
newest = 最新有效事务
```

日志事务只有在 checkpoint 完成后才能释放。

块复用问题说明：

```text
旧日志记录的是物理块号；
如果该块后来被释放并复用，
恢复时旧日志可能污染新文件。
```

解决办法包括：

```text
1. 延迟复用块；
2. 使用 revoke 记录，恢复时跳过被撤销块的旧写入。
```

## 23. 简短记忆版

```text
崩溃一致性：
    多个结构更新到一半崩溃，状态不一致。

日志：
    先记账，再落账。

TxB:
    事务开始。

TxE:
    事务提交标志。

checkpoint:
    把日志内容写回最终位置。

release:
    checkpoint 后释放日志空间。

write barrier:
    写的时候强制顺序。

checksum:
    恢复时检查事务完整性。

data journaling:
    数据和元数据都写日志，安全但慢。

metadata journaling:
    只写元数据日志，快但要保证数据先落盘。

ordered journaling:
    Db 先于 TxE 落盘。

journal superblock:
    记录日志有效范围，例如 oldest 和 newest。

block reuse:
    旧日志写物理块，但物理块后来可能属于新文件。

revoke:
    告诉恢复程序不要重放某些旧块写入。
```
